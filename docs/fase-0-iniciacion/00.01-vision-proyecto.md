## Problema
Wakaya necesita operar reservas, solicitudes de reserva y contenido publico desde un mismo producto. Hoy el flujo mezcla atencion comercial, disponibilidad de bungalows, seguimiento operativo y actualizacion del sitio publico, lo que genera duplicidad manual, riesgo de sobreasignacion y poca trazabilidad entre la web publica y el backoffice.

## Objetivo
Implementar Wakaya ERP como una aplicacion web Next.js con App Router que unifique:

- sitio publico bilingue ES/EN para mostrar el hospedaje y captar solicitudes;
- backoffice para reservas, bungalows y workbench de booking requests;
- APIs seguras mediante route handlers;
- persistencia principal en PostgreSQL con fallback local controlado para bootstrap y contingencia de desarrollo.

## Alcance
- Sitio publico bilingue ES/EN bajo `src/app/[locale]/` con home, catalogo de bungalows, paginas informativas y formulario de solicitud de reserva.
- Gestion de reservas con consulta, detalle, asignacion de bungalow, cambios de estado, pagos y auditoria.
- Gestion de booking requests originados desde la web publica y otros canales operativos.
- Gestion administrativa de bungalows, ocupacion y contenido publico localizado de bungalows con publicacion controlada desde backoffice.
- Autenticacion compatible con sesion de backoffice y OIDC/JWT, con autorizacion RBAC por roles.
- Observabilidad basica y endpoints de salud para la aplicacion y sus route handlers.

## No alcance
- Motor PMS/hospitality completo para multiples sedes o multiples propiedades.
- Channel manager avanzado con sincronizacion bidireccional con OTAs externas.
- CMS desacoplado o headless independiente del producto principal.
- Analitica avanzada, revenue management o pricing dinamico.
- Integraciones no visibles en el repo actual como pasarelas de pago, facturacion o terceros adicionales.

## Stakeholders
- Sponsor: direccion/operaciones del hospedaje Wakaya.
- Product Owner: responsable funcional de Wakaya ERP.
- Recepcion y reservas: operacion diaria de reservas, ocupacion y atencion al huesped.
- Comercial/marketing: responsable del sitio publico y contenido visible.
- Business Analyst: responsable de reglas operativas, estados y trazabilidad.
- Tech Lead: responsable de implementacion.
- Security Lead: responsable de sesion, OIDC, RBAC y auditoria.
- Operaciones/SRE: responsable de despliegue, continuidad y monitoreo.

## Restricciones
- Stack real del repo: Next.js 16 App Router sobre Node, con UI publica, backoffice y route handlers en la misma aplicacion.
- Rutas canonicas del producto: `src/app/[locale]/` para sitio publico, `src/app/admin/` para backoffice y `src/app/api/` para endpoints.
- Persistencia principal en PostgreSQL; cuando `DATABASE_URL` no esta disponible, el proyecto usa fallback local controlado mediante snapshots en `.data/`.
- Seguridad compatible con sesion persistida de backoffice y validacion OIDC/JWT para APIs protegidas.
- La capa editorial persistida actual debe mantenerse bilingue ES/EN y cubre el contenido visible de bungalows y su presencia en home.

## Metricas de exito
- 95% de consultas criticas de reservas y booking requests responden en menos de 2 segundos en carga nominal.
- 100% de cambios de estado y asignaciones de reserva quedan auditados.
- 0 solapes de ocupacion permitidos al asignar bungalows.
- 0 accesos administrativos efectivos sin rol autorizado.
- El sitio publico ES/EN y el backoffice operan sobre la misma fuente de datos cuando PostgreSQL esta disponible.

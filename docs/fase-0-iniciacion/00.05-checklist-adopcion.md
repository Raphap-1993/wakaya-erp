<a id="objetivo"></a>
## Objetivo
Usar esta checklist para implementar la plantilla en un proyecto nuevo sin dejar huecos de alcance, arquitectura, gobierno o ejecucion.

<a id="1-preparacion-inicial"></a>
## 1. Preparacion inicial
- [ ] Confirmar el tipo de proyecto, contexto y objetivo de negocio.
- [ ] Definir quien sera sponsor, PO, BA y responsable tecnico inicial.
- [ ] Crear el repositorio del proyecto a partir de esta plantilla.
- [ ] Revisar `docs/fase-0-iniciacion/00.00-guia-de-uso.md`, `docs/transversal/90.10-entregables-minimos-por-fase.md`, `docs/transversal/90.13-modos-de-trabajo.md`, `docs/transversal/90.14-instanciacion-fases-proyectos-reales.md` y `AGENTS.md`.
- [ ] Recorrer `docs/fase-0-iniciacion/00.06-ruta-guiada-caso-canonico.md` si el equipo necesita una explicacion unica del template en accion.
- [ ] Copiar `template.config.example.json` y completarlo con organizacion, repositorio, dominios, imagenes, soporte y datos de catalogo reales.
- [ ] Ejecutar `node scripts/init-project.mjs --config <tu-config>.json --dry-run` para revisar placeholders y archivos afectados antes de aplicar cambios.

<a id="2-encaje-del-proyecto"></a>
## 2. Encaje del proyecto
- [ ] Seleccionar escenario base en `docs/transversal/90.02-escenarios-de-referencia.md`.
- [ ] Seleccionar stack de referencia en `docs/transversal/90.04-stacks-de-referencia.md`.
- [ ] Documentar supuestos o diferencias relevantes respecto al escenario y stack elegidos.
- [ ] Confirmar si `stacks/*/template/` sera la base ejecutable oficial y dejar `src/` / `tests/` raiz como estructura neutral de trazabilidad.

<a id="3-definicion-del-proyecto"></a>
## 3. Definicion del proyecto
- [ ] Completar `docs/fase-0-iniciacion/00.01-vision-proyecto.md`.
- [ ] Revisar `plantillas/fase-1-analisis-requerimientos/README.md` y usar `modulo.md`, `caso-uso.md` y `historia-usuario.md` como insumo de discovery antes de consolidar fase 1.
- [ ] Completar `docs/fase-1-analisis-requerimientos/01.00-analisis-requerimientos.md`.
- [ ] Confirmar que `docs/fase-1-analisis-requerimientos/01.00-analisis-requerimientos.md` quedo instanciado al dominio real del proyecto y que ya no conserva texto instructivo de la plantilla.
- [ ] Si fase 1 necesita mas detalle, crear artefactos reales como `modulos/`, `casos-de-uso/` o `historias-usuario/` dentro de `docs/fase-1-analisis-requerimientos/`.
- [ ] Crear o actualizar `docs/fase-0-iniciacion/00.02-roadmap.md`.
- [ ] Crear o actualizar `docs/fase-0-iniciacion/00.03-estimacion-tiempo-costo.md`.
- [ ] Definir `docs/fase-0-iniciacion/00.04-roles-y-responsabilidades.md`.

<a id="4-experiencia-y-arquitectura"></a>
## 4. Experiencia y arquitectura
- [ ] Completar `docs/fase-2-ux-ui/02.00-ux-ui.md`.
- [ ] Completar `docs/fase-3-arquitectura/03.00-arquitectura.md`.
- [ ] Completar `docs/fase-3-arquitectura/03.01-decisiones-tecnologia.md`.

<a id="5-adopcion-de-sdd"></a>
## 5. Adopcion de SDD
- [ ] Confirmar si la iniciativa todavia esta en `modo exploratorio` o si ya corresponde pasar a `modo estructurado` segun `docs/transversal/90.13-modos-de-trabajo.md`.
- [ ] Revisar `docs/fase-4-sdd/04.00-spec-driven-development.md`.
- [ ] Revisar `ai/ejemplos/README.md` si el equipo usara agentes, prompts o skills para acelerar refinement y construccion.
- [ ] Si el proyecto adopta capa de IA oficial, adaptar `ai/agents/`, `ai/prompts/`, `ai/skills/`, `ai/commands/`, `ai/quality-gates/`, `ai/references/` y `ai/external-agent-skills.md` al dominio real en lugar de copiar `ai/ejemplos/`.
- [ ] Validar que cada command lifecycle (`/plan`, `/spec`, `/build`, `/test`, `/review`, `/ship`) tenga entradas, salidas y evidencia compatibles con el stack y proceso real del equipo.
- [ ] Confirmar que cada gate AI-first enlaza el checklist documental equivalente y que no hay criterios duplicados sin dueno.
- [ ] Tomar una primera feature priorizada del backlog.
- [ ] Crear `specs/<nnn-feature>/spec-funcional.md`.
- [ ] Crear `specs/<nnn-feature>/spec-tecnica.md`.
- [ ] Crear `specs/<nnn-feature>/spec-tareas.md`.
- [ ] Validar la feature con `docs/fase-4-sdd/04.01-checklist-spec-driven-development.md`.

<a id="6-preparacion-operativa-minima"></a>
## 6. Preparacion operativa minima
- [ ] Definir plantillas o rutas reales para QA.
- [ ] Definir pipeline objetivo y estrategia de release.
- [ ] Preparar `ci/pipeline-baseline.md` o documentar el workflow real equivalente a partir de [../../ci/README.md](../../ci/README.md).
- [ ] Ejecutar `node scripts/init-project.mjs --config <tu-config>.json` y confirmar que no quedan placeholders sin instanciar en workflows, `ops/`, catalogos ni stacks.
- [ ] Ejecutar `node ci/scripts/check-template-instantiation.mjs --mode instantiated --root .` antes del primer merge del proyecto.
  - [ ] Si se usara preview automatico, definir `PREVIEW_ARTIFACTS_JSON` como fuente oficial cuando el stack publique una o mas imagenes; usar `PREVIEW_BUILD_CONTEXT`, `PREVIEW_DOCKERFILE` y `PREVIEW_IMAGE_NAME` solo como fallback de single-image.
  - [ ] Definir `PREVIEW_REGISTRY_PROVIDER` (`ghcr`, `ecr`, `gar`, `acr` o `custom`) y sus credenciales/variables asociadas antes del primer push de preview.
  - [ ] Configurar `PREVIEW_KUBECONFIG` y `PREVIEW_KUSTOMIZE_PATH` en GitHub Actions para habilitar el deploy de preview.
  - [ ] Definir `PREVIEW_PUBLIC_URL_TEMPLATE` solo si el overlay de preview tambien expone una ruta publica real; si no existe esa exposicion, el workflow debe reportar namespace e imagenes, no una URL.
- [ ] Preparar runbook, rollback y monitoreo minimo.
- [ ] Confirmar como se gestionara backlog evolutivo y metricas operativas.

<a id="7-cierre-de-adopcion"></a>
## 7. Cierre de adopcion
- [ ] Revisar `docs/transversal/90.11-checklist-entregables.md`.
- [ ] Confirmar que la nomenclatura usada en el proyecto coincide con el estandar.
- [ ] Confirmar que los ejemplos no sustituyen entregables reales.
- [ ] Confirmar que cada fase relevante enlaza sus artefactos reales desde su `README.md`.
- [ ] Confirmar que los placeholders tecnicos quedaron reemplazados y que las referencias del repositorio apuntan al proyecto real.
- [ ] Alinear al equipo sobre como usar IA, specs, ADR y criterios de salida.

<a id="resultado-esperado"></a>
## Resultado esperado
Al terminar esta checklist, el proyecto ya deberia poder usar la plantilla como sistema de trabajo y no solo como repositorio de referencia.

<a id="objetivo"></a>
## Objetivo
Definir el minimo comun de metricas y eventos de producto para que cada proyecto derivado pueda medir adopcion, activacion, retencion y satisfaccion desde el primer sprint.

<a id="metricas-base"></a>
## Metricas base
- Usuarios activos diarios/semanales/mensuales (DAU/WAU/MAU).
- Tasa de activacion (usuarios que completan el primer flujo clave).
- Tasa de retencion (D7, D30) por cohorte.
- Funnel del flujo critico (en el caso canonico: `bandeja -> apertura -> transicion`).
- Satisfaccion: NPS trimestral, CSAT post-interaccion critica.

<a id="eventos-canonicos"></a>
## Eventos canonicos
Nombrado en `snake_case`, verbo + objeto. Ejemplos para el caso canonico:

- `bandeja_consultada`
- `expediente_abierto`
- `estado_transicionado`
- `auditoria_consultada`

Cada evento debe declarar: `event_name`, `ts`, `user_id`, `session_id`, `props` (objeto con atributos adicionales). Las props sensibles se anonimizan.

<a id="arquitectura-de-eventos"></a>
## Arquitectura de eventos
- Captura: SDK en frontend + emisor en backend.
- Transporte: cola de eventos (SQS, Kafka o equivalente) -> warehouse (BigQuery, Snowflake, Redshift).
- Exploracion: herramienta de analytics (Mixpanel, Amplitude, Metabase) conectada al warehouse.
- Gobierno: catalogo de eventos y tracking plan versionado en `ops/analytics/tracking-plan.md`.

<a id="privacidad"></a>
## Privacidad
- Respetar las preferencias del usuario: si se opt-out, no se envian eventos identificables.
- Nunca enviar PII en props ni en `user_id` plano: usar hash estable cuando sea necesario.
- Las retenciones minimas coinciden con las politicas descritas en `docs/transversal/90.16-privacidad-compliance.md`.

<a id="checklist"></a>
## Checklist
- [ ] Hay un tracking plan vigente con version y duenos.
- [ ] Cada feature nueva declara sus eventos y metricas objetivo.
- [ ] Los dashboards de producto y los SLI operativos no se contradicen.
- [ ] Existe un proceso para revisar calidad del dato al menos mensualmente.

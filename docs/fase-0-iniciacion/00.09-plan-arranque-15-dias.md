<a id="objetivo"></a>
## Objetivo
Un plan ejecutable para llevar un proyecto real desde cero hasta una primera feature productiva y un despliegue observado en 15 dias habiles (3 semanas). Sirve como baseline para discovery pagado, arranque de squad nuevo o migracion legacy a la plantilla.

No es un cronograma rigido: es una lista ordenada de entregables minimos con fechas objetivo. Cada equipo ajusta la cadencia segun su contexto, pero los entregables deben existir al cerrar cada semana.

<a id="supuestos"></a>
## Supuestos
- Equipo nucleo disponible: product owner, tech lead, 1-2 desarrolladores, 1 QA, 1 SRE o referente de plataforma.
- Una decision ya tomada sobre el stack objetivo (`node-next`, `java-monolith`, `quarkus-angular` o `spring-react`). Si no existe, anadir 2 dias previos para `docs/fase-3-arquitectura/03.01-decisiones-tecnologia.md`.
- Acceso a cloud (AWS, GCP o Azure) con cuenta `dev` provisionable, o a Kubernetes on-prem con namespace dedicado.
- Repositorio GitHub o GitLab ya autorizado con permisos para crear workflows, ambientes y secrets.
- Presupuesto para observabilidad (Grafana Cloud, Datadog, New Relic u otro) o capacidad de operar el stack auto-hospedado en `ops/observability/`.

<a id="semana-1---discovery-y-vision"></a>
## Semana 1 - Discovery y vision
Objetivo: cerrar la fase 0 y la fase 1 con artefactos reales, no placeholders.

Dia 1. Kickoff y alineacion.
- Completar `docs/fase-0-iniciacion/00.01-vision-proyecto.md`.
- Levantar stakeholders y registrar en `docs/fase-0-iniciacion/stakeholders/` o equivalente.
- Ejecutar `scripts/bootstrap.sh` o `scripts/bootstrap.ps1` en la maquina del tech lead y el PO.

Dia 2. Roadmap y estimacion inicial.
- Completar `docs/fase-0-iniciacion/00.02-roadmap.md` con 3-5 hitos trimestrales.
- Poblar `docs/fase-0-iniciacion/00.03-estimacion-tiempo-costo.md` con bloques gruesos; no se espera precision de T-shirt fina.
- Registrar dimensiones del escenario con `escenarios/README.md` para dejar explicito el perfil (tamano equipo, criticidad, cumplimiento).

Dia 3. Roles y research plan.
- Completar `docs/fase-0-iniciacion/00.04-roles-y-responsabilidades.md` (RACI).
- Definir el plan de user research de las proximas dos semanas en `docs/fase-0-iniciacion/00.07-user-research.md`.
- Elegir metricas producto base en `docs/fase-0-iniciacion/00.08-product-analytics.md`.

Dia 4. Analisis de dominio.
- Trabajar `docs/fase-1-analisis-requerimientos/01.00-analisis-requerimientos.md` con al menos 2 modulos, 3 casos de uso y 10 requerimientos funcionales.
- Identificar requerimientos no funcionales iniciales (performance, disponibilidad, cumplimiento).

Dia 5. Backlog y priorizacion.
- Cerrar backlog inicial en `docs/fase-1-analisis-requerimientos/01.00-analisis-requerimientos.md` o en subcarpetas reales enlazadas desde el README de fase 1.
- Elegir la feature canonica que se construira en la semana 3 y registrar su `specs/NNN-slug/README.md` vacio.
- Entregable de fin de semana: walkthrough de 30 minutos con stakeholders mostrando vision, roadmap, modulos y feature elegida.

<a id="semana-2---ux-arquitectura-y-scaffolding"></a>
## Semana 2 - UX, arquitectura y scaffolding
Objetivo: convertir la vision en arquitectura ejecutable y preparar el entorno productivo.

Dia 6. UX baseline.
- Completar `docs/fase-2-ux-ui/02.00-ux-ui.md` y levantar los primeros flujos con Product Design, SPDD inicial y `docs/fase-2-ux-ui/02.14-html5-first-prototyping.md`.
- Si el stack elegido usa Angular, definir desde este dia que el frontend real vivira en el workspace del stack. No crear Angular Mock salvo excepcion tecnica documentada.
- Definir design tokens en `docs/fase-2-ux-ui/02.02-design-system-tokens.md` aunque sean basicos.

Dia 7. Arquitectura inicial.
- Completar `docs/fase-3-arquitectura/03.00-arquitectura.md` con diagrama C4 de nivel contexto y contenedor en `likec4/`.
- Registrar ADR-001 en `docs/fase-3-arquitectura/adr/` con el contexto de negocio y la primera decision de alto impacto (por ejemplo `event-driven vs request-response`, o `SPA vs SSR`).

Dia 8. Decisiones tecnologicas y plan de despliegue.
- Completar `docs/fase-3-arquitectura/03.01-decisiones-tecnologia.md` y `docs/fase-3-arquitectura/03.03-plan-despliegue.md` con ambientes y estrategia (canary, blue-green, feature flags).
- Confirmar estrategia de datos en `docs/fase-3-arquitectura/03.06-modelo-datos.md` si aplica.

Dia 9. Scaffolding real.
- Poblar `template.config.example.json` con los datos reales (nombres, owners, dominios, Terraform state bucket).
- Ejecutar `node scripts/init-project.mjs --config template.config.example.json --dry-run` y revisar el diff.
- Si el output es correcto, ejecutar `node scripts/new-service.mjs --stack <stack> --config template.config.example.json --dest ../<slug>`.
- Si el stack es Angular, promover el frontend instanciado a un workspace Nx antes de abrir el desarrollo sostenido.
- Validar con `node ci/scripts/check-template-instantiation.mjs --mode instantiated --root ../<slug>`.

Dia 10. Infra y pipelines.
- Adaptar `stacks/<stack>/template/infra/terraform/` para el ambiente `dev` y ejecutar `terraform init` + `terraform plan`.
- Habilitar los workflows de GitHub Actions en `.github/workflows/` y cargar secrets (cosign keyless por OIDC no requiere secret).
- Ejecutar el primer pipeline y verificar que SARIF, SBOM y firmas (cosign) se publican correctamente.
- Entregable de fin de semana: scaffolding desplegado en `dev`, pipeline verde, ADR-001 aprobado, arquitectura base revisada.

<a id="semana-3---primer-deploy-y-feature-canonica"></a>
## Semana 3 - Primer deploy y feature canonica
Objetivo: una feature real en produccion con observabilidad, SLO declarado y runbook valido.

Dia 11. Spec driven development.
- Completar `specs/NNN-slug/spec-funcional.md` y `spec-tecnica.md` para la feature canonica elegida el dia 5.
- Generar `plan-tareas.md` y abrir los issues correspondientes en el tracker.

Dia 12. Implementacion primer vertical.
- Desarrollar el camino feliz (un endpoint API + una vista UI o un consumer + producer) con tests unitarios y al menos 1 test de contrato (`stacks/node-next/template/tests/contract/` como referencia).
- Desplegar a `staging` via pipeline.

Dia 13. Testing y accesibilidad.
- Correr suite de Playwright + Axe definida en `stacks/node-next/template/tests/a11y/` o equivalente del stack.
- Completar `qa/fase-6-qa/plan-pruebas.md` y publicar primeros resultados en `qa/fase-6-qa/evidencias/`.
- Definir carga base con k6 (`stacks/node-next/template/tests/load/`) y registrar numeros en `docs/fase-6-qa/06.00-plan-pruebas.md` y `docs/fase-8-operacion/08.01-observabilidad.md`.

Dia 14. Observabilidad y SLO.
- Declarar SLOs en `ops/observability/slos.yaml` (ver `docs/fase-3-arquitectura/03.07-slos-como-codigo.md`).
- Habilitar el dashboard `ops/observability/grafana-dashboard-api.json` en el tenant Grafana del proyecto.
- Activar alertas de `ops/observability/prometheus-rules.yaml` y registrar el canal de notificacion (Slack/Teams/PagerDuty) en el runbook `ops/runbooks/`.

Dia 15. Go live controlado y retro de arranque.
- Promover la feature a produccion detras de feature flag (ver `docs/transversal/90.17-feature-flags.md`) con trafico inicial bajo (5 por ciento).
- Ejecutar el runbook de rollback en ambiente de prueba para validar el procedimiento.
- Retrospectiva de 90 minutos con el squad sobre el arranque y actualizar `CHANGELOG.md` del proyecto con el milestone `v0.1.0-bootstrap`.

<a id="riesgos-y-mitigaciones"></a>
## Riesgos y mitigaciones
- Stack cambia a mitad del plan. Mitigacion: registrar ADR explicito y reinstanciar solo la porcion afectada (`new-service.mjs` puede correrse en paralelo con otro `--dest`).
- Disponibilidad de cloud se retrasa. Mitigacion: desplegar en k3d o kind local con `stacks/<stack>/template/k8s/base/` hasta que cloud este listo.
- Feature canonica es mas grande que un sprint. Mitigacion: dividirla en una rebanada vertical minima (un caso de uso completo, no un CRUD entero) y dejar el resto al siguiente ciclo.
- Secrets de firma o publicacion no disponibles. Mitigacion: cosign keyless con OIDC de GitHub Actions no requiere pre-aprovisionado, y el dashboard puede apuntar a Grafana Cloud free tier para el piloto.

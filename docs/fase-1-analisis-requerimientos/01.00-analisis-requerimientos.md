## Actores
- Visitante/Huesped: navega el sitio publico bilingue y envia solicitudes de reserva.
- Recepcion/Operaciones: consulta reservas, asigna bungalows y ejecuta acciones diarias permitidas.
- Coordinacion/Aprobacion: valida transiciones sensibles y seguimiento operativo.
- Administrador de backoffice: gestiona usuarios, roles y modulos administrativos.
- Editor de contenido: mantiene textos, imagenes y visibilidad publica de bungalows.
- Auditor: consulta historial y evidencia.

## Modulos
- Sitio publico bilingue ES/EN.
- Booking requests y captacion desde la web publica.
- Reservas y monitor operativo.
- Ocupacion y asignacion de bungalows.
- Contenido publico de bungalows.
- Seguridad, sesiones y permisos.
- Historial y auditoria.

## Requerimientos funcionales
| ID | Actor | Requerimiento | Resultado esperado |
|---|---|---|---|
| RF-01 | Visitante/Huesped | Navegar el sitio publico ES/EN | Contenido localizado y consistente por idioma |
| RF-02 | Visitante/Huesped | Enviar una solicitud de reserva desde la web publica | Booking request registrado y trazable |
| RF-03 | Recepcion/Operaciones | Consultar reservas por estado, fecha, canal, responsable y bungalow | Lista filtrable y ordenada |
| RF-04 | Recepcion/Operaciones | Ver detalle y editar una reserva | Datos principales, estado actual y acciones disponibles |
| RF-05 | Recepcion/Operaciones | Asignar bungalow sin solape de ocupacion | Asignacion valida y ocupacion bloqueada |
| RF-06 | Coordinacion/Aprobacion | Ejecutar y aprobar transiciones operativas sensibles | Estado final actualizado con auditoria |
| RF-07 | Editor de contenido | Mantener contenido publico localizado de bungalows y su visibilidad en home | Publicacion controlada desde backoffice |
| RF-08 | Administrador de backoffice | Gestionar usuarios y roles | Accesos actualizados segun RBAC |
| RF-09 | Auditor | Consultar historial de reservas y acciones | Eventos con fecha, usuario y accion |

## Requerimientos no funcionales
- RNF-01 Seguridad: compatibilidad con sesion de backoffice y OIDC/JWT; RBAC para `reservation:*`, `complaint:*` y `admin:users`.
- RNF-02 Auditoria: todo cambio relevante de reserva o asignacion queda registrado.
- RNF-03 Persistencia: PostgreSQL como fuente principal, con fallback local controlado solo para bootstrap/desarrollo cuando no exista `DATABASE_URL`.
- RNF-04 Localizacion: el sitio publico y el contenido de bungalows deben soportar ES/EN sin duplicar rutas manualmente.
- RNF-05 Performance: reservas, booking requests y endpoints publicos responden en menos de 2 segundos para carga nominal.
- RNF-06 Observabilidad: logs estructurados y endpoint de salud operativo para la aplicacion.

## Reglas de negocio
- Solo usuarios con permiso `reservation:write` o superior pueden editar reservas y ejecutar cambios operativos.
- Solo usuarios con permiso `reservation:approve` pueden aprobar transiciones sensibles.
- No se permite asignar un bungalow si existe conflicto de ocupacion para las fechas solicitadas.
- Toda reserva, asignacion o transicion registra usuario, fecha, estado anterior, estado nuevo y motivo.
- Toda solicitud publica entra como booking request trazable antes de convertirse en operacion confirmada.
- El contenido publico visible depende de contenido localizado ES/EN y de banderas administrativas de publicacion/ordenamiento.

## Integraciones
- PostgreSQL `wakaya_erp` con migraciones SQL bajo `db/migrations/`.
- Fallback local controlado en `.data/wakaya-reservations.snapshot.json` y `.data/wakaya-backoffice-auth.snapshot.json`.
- OIDC por configuracion (`OIDC_ISSUER`, `OIDC_JWKS_URL`, `OIDC_AUDIENCE`, `OIDC_ROLES_CLAIM`) para compatibilidad con APIs protegidas.
- Sesion de backoffice persistida para acceso administrativo.
- Integracion de correo/webhook para seguimiento operativo de booking requests cuando aplique.

## Objetivo
Definir un flujo reproducible para transformar UX escrito en Markdown en un prototipo navegable y luego en un mock ejecutable, sin perder trazabilidad con negocio.

Este flujo queda como referencia legacy. El flujo vigente para prototipos navegables es HTML5-first: primero se estructura el producto con `/ux`, luego `/prototype --mode html5`, y Penpot se usa si se requiere formalizacion visual.

## Cuando conviene usar este flujo
- Cuando Product Owner o negocio describen la experiencia primero en texto.
- Cuando se necesita validar rapidamente la UX antes de comprometer arquitectura final.
- Cuando el equipo quiere un puente claro entre descubrimiento UX y un frontend exploratorio.

## Flujo recomendado
### 1. Product Owner (Markdown)
Artefacto base donde se describe:
- objetivo del flujo,
- usuarios y contexto,
- journeys,
- pantallas,
- reglas visibles,
- estados vacios, errores y confirmaciones.

Salida minima:
- `docs/fase-2-ux-ui/02.00-ux-ui.md`
- `docs/fase-2-ux-ui/journeys/*.md` si el flujo ya amerita descomposicion
- `docs/fase-2-ux-ui/pantallas/*.md` si hay pantallas complejas

### 2. IA (estructura UX)
La IA no sustituye la decision de negocio; ayuda a ordenar y completar la estructura UX.

La IA debe producir:
- objetivo de producto y actor por flujo,
- dos alternativas cuando haya ambiguedad relevante,
- arquitectura de informacion,
- mapa de pantallas,
- componentes clave,
- estados de interfaz,
- preguntas abiertas,
- trazabilidad entre requerimientos y experiencia.

Salida minima:
- secciones estructuradas dentro de `02.00-ux-ui.md`, o
- documentos complementarios dentro de `docs/fase-2-ux-ui/`

### 3. Penpot (mockup + prototipo clickable)
Penpot se usa para validar visualmente el flujo y la navegacion antes del mock ejecutable.

El prototipo en Penpot debe cubrir:
- happy path,
- estados vacios,
- errores relevantes,
- feedback de exito,
- variantes de rol si impactan la experiencia.

Salida minima (ubicada en `specs/<feature>/`):
- `specs/<feature>/prototype.md` con descripcion textual del prototipo y enlace a Penpot si existe
- `specs/<feature>/prototype-validation.md` con observaciones convertidas en decisiones trazables (DEC-UX-NNN)
- Cierre de `gate-spdd-approved` una vez que prototype-validation.md tenga todas las observaciones resueltas

Ver checklist de salida: [02.12-checklist-spdd.md](02.12-checklist-spdd.md)

### 4. Angular Mock en Nx (opcional legacy)
Angular Mock ya no es necesario como paso estandar. Si se usa, debe quedar como excepcion documentada para exploracion tecnica, no como requisito de UX.

El Angular Mock no es el producto final. Si se autoriza por excepcion, su rol es:
- validar interaccion real,
- descubrir huecos de UX y datos,
- alimentar arquitectura, APIs y SDD,
- reducir ambiguedad antes de construir el frontend productivo.

Salida minima (los artefactos de SPDD viven en `specs/<feature>/`; los componentes en `frontend/libs/`):
- `specs/<feature>/spdd-frontend.md` con pantallas, estados UI, permisos y contrato mock/API
- `frontend/libs/feature-<nombre>/` para componentes Angular de la feature
- `frontend/libs/data-access-<nombre>/` para store y servicio mock
- `specs/<feature>/ui-test-cases.md` con casos cubiertos y pendientes
- `specs/<feature>/traceability.md` con columna RF/HU → pantalla → componente → endpoint → prueba

## Definition of done del flujo
- El Markdown del Product Owner tiene journeys, pantallas, reglas visibles y estados UX.
- La IA dejo explicita la estructura UX y las preguntas abiertas.
- Si habia ambiguedad relevante, quedaron alternativas y recomendacion.
- `specs/<feature>/product-design.md` existe y `gate-ux-ready` esta declarado. Ver [02.11-checklist-product-design.md](02.11-checklist-product-design.md).
- `specs/<feature>/prototype.md` describe el prototipo y `prototype-validation.md` cierra todas las observaciones como decisiones DEC-UX-NNN.
- `gate-spdd-approved` declarado antes de avanzar a SDD. Ver [02.12-checklist-spdd.md](02.12-checklist-spdd.md).
- Los componentes Angular usan rutas reales, `signals` para estado y el contrato de API queda fijado en `spdd-frontend.md` para que el mock sea reemplazable sin cambios en los componentes.
- Existe trazabilidad en `traceability.md` entre RF/HU, pantallas, componentes, endpoints y pruebas.

## Como conecta con arquitectura y SDD
- Arquitectura usa este flujo para delimitar fronteras entre frontend, backend y contratos.
- SDD usa este flujo para bajar journeys a features, specs funcionales y tareas.
- Para detectar endpoints, DTOs, validaciones y eventos ahora se recomienda usar `prototype-html5/index.html`, `flujo.md` y `decisiones-ux.md` antes de SDD.

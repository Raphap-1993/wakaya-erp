## Objetivo
Dejar un criterio estable para el back office de Wakaya, de modo que login, permisos, tableros, tablas y acciones sigan una misma logica operacional y no vuelvan a mezclar detalle, navegacion y ejecucion en una sola superficie.

## Principios aplicados
- `list-first`: la bandeja principal prioriza lectura rapida, semaforos, filtros y acciones claras.
- `secondary detail`: el detalle profundo vive en una pagina aparte o en modal, no dentro de la misma tabla.
- `permission-aware UI`: si el rol no puede ejecutar una accion, la interfaz no debe sugerir que si puede.
- `explicit denial`: falta de permiso no se comunica como `404`; se comunica como una superficie clara con siguiente paso.
- `operator context`: el shell interno siempre expone quien esta operando y con que rol entra al modulo.
- `status-first`: cada vista operativa debe exponer estado, severidad y siguiente accion antes que texto narrativo.
- `persisted management`: usuarios, sesiones, solicitudes y reservas se leen de la capa persistida real, no de estados efimeros del navegador.
- `live runtime`: las superficies operativas del back office no deben quedar cacheadas como vistas casi estaticas; login, dashboard y modulos internos deben resolver contra el estado actual.

## Patrones de interfaz

### 1. KPIs semaforizados
- `critical`: requiere atencion inmediata.
- `warning`: requiere seguimiento o cobro.
- `success`: estable o resuelto.
- `neutral`: sin actividad o solo contexto.
- Cada KPI debe tener:
  - valor
  - etiqueta corta
  - meta descriptiva de una sola linea

### 2. Tablas operativas
- Cada tabla debe tener una `columna de acciones`.
- La fila no debe depender de click completo para navegar.
- Los chips de estado deben comunicar:
  - estado principal
  - sync o integracion
  - alerta o conflicto
- Los encabezados deben permanecer compactos y con lenguaje operativo.

### 3. Detalle secundario
- `pagina nueva` para gestion completa de una reserva o usuario.
- `modal` para resumen o inspeccion puntual que no necesita cambiar el contexto principal.
- No volver a insertar paneles laterales profundos dentro del listado principal si rompen la lectura del tablero.

### 4. Permisos visibles
- `viewer`: solo lectura.
- `editor`: alta y edicion operativa.
- `approver`: confirmaciones, cobros y cambios de estado sensibles.
- `admin`: gestion de usuarios internos y control total.
- La navegacion lateral y los botones visibles deben depender del permiso real.
- Las acciones mutantes de inbox, sync y reply no deben quedar habilitadas para `viewer`.
- Las acciones de aprobacion final no deben mostrarse como CTA principal para `editor` si el rol real es `approver` o `admin`.

### 5. Estados de acceso
- `login`: formulario directo, persistido y con retorno al modulo pedido.
- `forbidden`: pantalla explicita de permiso insuficiente, con CTA hacia modulos permitidos.
- `not found`: solo para recursos inexistentes, no para faltas de permiso.

## Aplicacion al dominio Wakaya
- `Reservas`: lista operativa con ficha aparte, edicion aparte y ocupacion separada.
- `Solicitudes web`: inbox compacto con apertura de gestion en pagina dedicada.
- `Ocupacion`: grilla semanal con resumen en modal y no detalle incrustado permanente.
- `Usuarios internos`: modulo persistido separado para accesos, roles y activacion.
- `Login y shell interno`: acceso real por sesion, navegacion filtrada por permisos y render dinamico para evitar stale data en rutas internas.

## Referencias externas consultadas
- [`Atlassian Design System · Lozenge`](https://atlassian.design/components/lozenge): confirma el uso de etiquetas compactas para comunicar estado en tablas y listados operativos.
- [`Carbon Design System · Status indicator`](https://carbondesignsystem.com/components/status-indicator/usage/): refuerza la semaforizacion consistente por severidad y progreso operacional.
- [`PatternFly · Modal`](https://www.patternfly.org/components/modal): valida el uso de modal para resumen o confirmacion sin romper el contexto del tablero principal.
- [`PatternFly · Table guidelines`](https://www.patternfly.org/components/table): apoya tablas compactas con columnas claras y acciones visibles para superficies administrativas.
- [`PatternFly · Empty state design guidelines`](https://www.patternfly.org/components/empty-state/design-guidelines): justifica superficies explicitas para `no access`, `no results` y otros estados donde el operador necesita siguiente paso y no solo mensaje.

## Traduccion practica de esas referencias a Wakaya
- Los estados operativos se expresan con `chips` y `badges`, no solo con texto plano.
- Las tablas de back office siempre exponen una `columna de acciones` separada del estado.
- La apertura de gestion completa ocurre en `pagina aparte` y la inspeccion puntual en `modal`.
- El dashboard no se diseña como landing estetica, sino como `superficie de decision`: cantidad, severidad y siguiente paso.
- Las rutas internas que leen estado vivo deben forzarse como `runtime dinamico`, para que no reaparezcan usuarios, reservas o solicitudes ya modificadas en la base persistida.
- Los roles de solo lectura no deben ver CTAs que terminan en `404`, `403` o formularios bloqueados; si no pueden actuar, la celda muestra un estado pasivo como `Solo lectura`.
- El shell debe mostrar `operador activo` y `roles cargados` para que el equipo entienda por que ve ciertos modulos y no otros.
- Si un rol entra a un modulo que no le corresponde, el sistema responde con `forbidden` mas rutas seguras alternativas, no con vacio ambiguo.
- Cuando una grilla detecta conflicto multiple, la superficie principal solo expone el `resumen del conflicto` y el `conteo de cruces`; la comparacion detallada vive dentro del modal o ficha secundaria.

## Anti-patrones a evitar
- Botones de escritura visibles para roles solo lectura.
- Mostrar enlaces de edicion para operadores que no tienen permiso real de escritura y confiar en que la ruta falle despues.
- Mezclar formulario de detalle dentro de la tabla principal.
- Ocultar estados criticos solo en texto corrido sin color ni chip.
- Depender de mock data o seeds de demostracion en el panel operativo.
- Usar una sola pantalla para listar, editar, aprobar y auditar al mismo tiempo.

## Criterio de aceptacion UX del back office
- El usuario entra por login real y aterriza en una superficie persistida.
- La navegacion muestra solo modulos que puede abrir.
- Cada tablero expone KPIs con tono semaforizado.
- Cada listado tiene acciones explicitas por fila.
- El detalle profundo se resuelve con pagina aparte o modal.
- Los permisos alteran UI y API, no solo la ruta.

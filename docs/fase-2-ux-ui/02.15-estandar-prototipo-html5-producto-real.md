## Proposito
Definir el estandar minimo para que un prototipo HTML5 generado por IA se vea y se comporte como una porcion real del producto, no como una maqueta pobre, una landing generica o una tabla de RF.

Este documento aplica a proyectos instanciados desde el template cuando el equipo usa `/prototype --mode html5`.

## Principio rector
El objetivo no es "hacer un HTML". El objetivo es crear evidencia navegable de producto para descubrir problemas de UX, flujo, permisos, estados, datos y validaciones antes de construir frontend productivo.

Un buen prototipo HTML5 debe permitir que un stakeholder recorra una tarea real de inicio a fin sin explicaciones externas.

## Portafolio antes que monolito
Cuando el alcance sea un producto completo, no se debe comprimir todo en un unico prototipo. El agente debe proponer un **portafolio de prototipos por vertical funcional**.

Ejemplos de verticales segun dominio:

| Dominio | Verticales esperadas |
|---|---|
| Streaming/contenido | autenticacion/perfiles, catalogo/player, control parental, busqueda, favoritos/historial, editorial, notificaciones, auditoria |
| SaaS operativo | login, dashboard operativo, lista/trabajo, detalle, aprobaciones, reportes, auditoria, configuracion |
| E-commerce | discovery, ficha producto, carrito, checkout, ordenes, devoluciones, backoffice, soporte |
| Educacion | onboarding, cursos, leccion, evaluacion, progreso, tutor/admin, reportes |
| Salud | admision, agenda, historia, atencion, recetas, facturacion, auditoria y privacidad |

Si el agente decide menos verticales, debe documentar el supuesto en `decisiones-ux.md` o ADR. "Rapidez" no es razon suficiente para empobrecer el prototipo.

## Hub obligatorio
`prototype/index.html` debe ser un hub de producto, no solo una lista de enlaces.

**Estructura estandarizada desde v12.21**: las 10 secciones obligatorias estan
definidas en `plantillas/transversal/PROTOTYPE_HUB.md`. El hub se genera con:

```sh
node scripts/ai-framework-agent.mjs generate-prototype-hub
```

Las 10 secciones son: (0) head + tokens, (1) banner SPDD, (2) hero con stats,
(3) journey por actor, (4) spec cards con invariantes + status pill,
(5) actores del sistema, (6) matriz de cobertura SPDD (regenerada desde la BD),
(7) decisiones transversales, (8) quick links a docs por spec, (9) footer
con stack.

El validador `ci/scripts/check-prototype-hub.mjs` verifica que existe la
cantidad correcta de spec-cards (= specs en `specs/`), tabla de cobertura
con columnas SPDD, banner SPDD, hero con stats, y que no haya headings
con `RF-NN:` o `gate-xxx:` (esos van solo en tablas o invariantes).

> **Nota v12.20**: el "aviso de datos mock" obligatorio dentro de cada prototipo
> se elimino. El prototipo HTML5 ahora debe verse como producto real; la
> condicion de "simulacion / sin backend real" se declara en el hub
> (`prototype/index.html`) y en `prototype-validation.md`, no en cada pantalla.

## Separacion obligatoria: hub/documentacion vs spec HTML5
El hub y la documentacion oficial son el lugar correcto para mostrar trazabilidad tecnica. El prototipo HTML5 de cada spec es el lugar para mostrar producto.

| Artefacto | Debe mostrar |
|---|---|
| `prototype/index.html` | Flujo completo por actor, RF/RN, roles, cobertura, gates, links a specs |
| `docs/fase-2-ux-ui/*` | Metodo, decisiones, criterios, bloqueos, validaciones y trazabilidad |
| `specs/<feature>/*.md` | Product Design, SPDD, contratos, RF, permisos, componentes propuestos y riesgos |
| `specs/<feature>/prototype-html5/index.html` | Experiencia de usuario final, datos mock del dominio, eventos, navegacion, estados y feedback |

Regla: lo tecnico puede existir como comentario o documento de soporte solo si no invade la experiencia visible. Si un stakeholder ve RF, permisos, rutas, gates o nombres de componentes como contenido de pantalla, el prototipo esta mal orientado.

## Estandar visual minimo
Cada prototipo por spec debe verse como una parte real del producto:
- primer viewport reconocible como el producto y su tarea principal;
- patron visual propio del dominio, no dashboard lateral por comodidad;
- navegacion interna con pantallas o vistas;
- datos mock ricos y verosimiles;
- responsive mobile y desktop;
- estado visual consistente entre prototipos del portafolio.

Cada spec debe tener su propia implementacion HTML5 segun su dominio. No se permite una fabrica generica que produzca la misma shell para catalogo, player, control parental, backoffice y observabilidad cambiando solo textos.

La consistencia visual es obligatoria. Diferenciar verticales no significa cambiar de producto. Todos los prototipos deben compartir tokens base de color, tipografia, espaciado, botones, paneles, modales, toasts, formularios y estados. Se permite un color de acento por vertical siempre que la marca, el dorado de accion, fondos, contraste y estilo de componentes se mantengan coherentes.

El HTML visible debe usar nombres de producto, no nombres de metodologia. Si un panel se llama `Recorrido`, `Ajustes de la experiencia`, `Resumen`, `Actividad de ejemplo` o parecido, el agente debe reemplazarlo por una etiqueta del dominio: `Reproduccion`, `Resultados`, `Solicitudes familiares`, `Continuar viendo`, `Progreso diario`, `Historial editorial`, `Prueba de aviso` o `Seguimiento`, segun corresponda.

La forma visual tampoco debe parecer metodologia. Un timeline es valido cuando el dominio lo pide: tracking de pedido, historia clinica, auditoria, seguimiento de caso, expediente, ticket, roadmap, despliegue o actividad operacional. En ese caso debe verse como producto: eventos con fechas, actores, estados, evidencia, filtros, detalle clicable y acciones.

El anti-patron es usar una linea de tiempo generica, pasos numerados o checklist para demostrar que "el flujo existe". Si el dominio no necesita timeline, el avance debe verse como otro componente del producto: progreso de reproduccion, estado de publicacion, cola editorial, solicitudes familiares, resultados, actividad reciente, historial o seguimiento operativo.

Patrones orientativos:
- streaming/contenido: topbar, selector de perfil, hero, filas de contenido, posters, filtros, detalle y player;
- modo infantil: shell distinto, iconos grandes, bajo texto, salida con PIN;
- herramienta operativa: toolbar, filtros, tabla, detalle, acciones, historial y estados;
- backoffice: cola de trabajo, revision, aprobacion/rechazo, confirmaciones y auditoria;
- seguridad/observabilidad: eventos, severidad, actor, fecha, resultado, permisos y trazabilidad.

## Metodo de decision antes de escribir HTML
Antes de generar `index.html`, el agente debe dejar decidida la forma del producto. Esta decision manda sobre la comodidad tecnica de reutilizar una shell.

1. Identificar dominio del spec: streaming, player, busqueda, backoffice, seguridad, educacion, comercio, salud u otro.
2. Identificar actor activo y tarea principal del spec.
3. Elegir un patron de producto reconocido para ese dominio.
4. Definir una interaccion principal con click/evento real en JavaScript vanilla.
5. Definir datos mock del dominio, no datos genericos.
6. Solo despues escribir HTML/CSS/JS.

Plantilla de decision requerida en `decisiones-ux.md`:

```md
## Decision de patron de producto
- Dominio del spec:
- Actor principal:
- Tarea principal recorrible:
- Patron elegido:
- Por que no se usa shell generica:
- Interacciones del prototipo:
- Limitaciones:
```

### Ejemplos de decision
| Spec | Decision incorrecta | Decision correcta |
|---|---|---|
| Catalogo de videos | Sidenav + tabla + formulario + cards iguales para todos los specs | Topbar de marca, selector de perfil, buscador, filtros por edad/tema, posters, detalle en modal, acciones Reproducir/Favorito |
| Reproductor streaming | Panel generico con pasos y boton confirmar | Player 16:9, play/pause, seek bar, volumen, calidad, subtitulos, lista de episodios, autoplay y estados de buffering/error |
| Control parental | Tabla de estados con labels de RF | Centro familiar con hijos, reglas por edad, solicitudes, PIN, aprobacion/bloqueo y salida protegida |
| Backoffice editorial | Dashboard consumidor con posters solamente | Cola editorial, filtros, ficha de contenido, revision doctrinal, aprobar/observar, programar publicacion e historial |
| Seguridad/observabilidad | Cards comerciales sin detalle operativo | Consola operativa con eventos, severidad, actor, sesion, evidencia, escalamiento y auditoria |

### Anti-patron bloqueante
Un prototipo por spec no esta listo si:
- usa la misma estructura visual para streaming, backoffice, seguridad y perfiles;
- muestra `Resumen`, `Actividad de ejemplo`, `Recorrido`, `Ajustes de la experiencia`, `Formulario-spec` o etiquetas parecidas como navegacion principal;
- convierte `loading`, `empty`, `error`, `permission denied`, `success` o `progress` en una pantalla de checklist;
- usa timeline generico, pasos numerados o checklist visible para demostrar flujo sin que sea un patron natural del dominio;
- pone RF visibles dentro de la pantalla de producto para demostrar cobertura;
- tiene botones que solo cambian tabs, pero no ejecutan una accion propia del dominio;
- se ve como dashboard administrativo cuando el spec es de consumo, streaming, player, kids o busqueda.

La cobertura RF, estados formales y gates viven en el hub y documentos SPDD. En el prototipo por spec deben aparecer como comportamiento natural del producto: filtros que dejan resultados vacios, modal de detalle, toast de favorito, player con progreso, bloqueo parental, error recuperable o permiso denegado segun aplique.

### Ejemplo obligatorio por dominio
Para un spec `catalogo-y-reproduccion`, el prototipo correcto no debe abrir con una pantalla de metodologia. Debe abrir con una experiencia de streaming:

```text
Topbar de marca / perfil activo
  -> hero con contenido destacado
  -> buscador por titulo, tema biblico o personaje
  -> filtros por edad, categoria y tema
  -> grid de posters con badges de edad y progreso
  -> click en poster abre modal de detalle
  -> Reproducir abre player o estado de reproduccion
  -> Favorito muestra toast
  -> filtro sin resultados muestra empty state
  -> perfil nino intenta contenido 12+ y ve bloqueo parental
```

Para un spec `reproductor-streaming`, el prototipo correcto debe parecer player:

```text
Header compacto / titulo del episodio
  -> video 16:9 con poster
  -> play/pause
  -> seek bar
  -> volumen
  -> calidad/subtitulos
  -> lista de episodios
  -> buffering simulado
  -> error recuperable
  -> fin de episodio con autoplay
```

## Comportamiento minimo
Cada `specs/<feature>/prototype-html5/index.html` debe incluir:
- link discreto de vuelta al hub (icono o texto suave en el topbar, NO banner que rompa la sensacion de producto);
- navegacion entre vistas;
- happy path;
- error path;
- `permission denied` cuando aplique;
- estado `loading` o `in progress` si hay procesos;
- estado `empty` si hay busqueda/listados;
- validaciones de formulario;
- toast, modal, confirmacion o mensaje recuperable;
- diferencias de experiencia por rol/perfil cuando apliquen, expresadas en lenguaje de producto, no como permisos tecnicos;
- ningun RF, permiso tecnico, ruta interna, nombre de componente, gate o instruccion para IA como contenido visible de pantalla;
- ningun encabezado o label visible de metodologia; todo texto visible debe pertenecer al producto, al dominio o a sus datos mock;
- direccion visual consistente con los demas prototipos del portafolio;
- limitaciones documentadas.

## Bloqueantes
Debe bloquearse o quedar `Listo con observaciones` si:
- parece una plantilla generica;
- solo enumera RF sin flujo;
- el hub es una lista simple;
- no hay prototipos por vertical cuando el alcance es producto completo;
- no hay estados de error;
- no diferencia roles/perfiles cuando el dominio lo exige;
- no muestra restricciones o accesos denegados en lenguaje de producto cuando aplica;
- no hay datos mock representativos;
- no hay `flujo.md` ni `decisiones-ux.md`;
- se declara `gate-spdd-approved` sin validacion humana.

Tambien debe bloquearse si el spec tiene un patron de dominio reconocible y el agente eligio una shell comun por rapidez. Ejemplo: para `catalogo-y-reproduccion`, una tabla con formulario no reemplaza catalogo visual, detalle y player.

## Archivos requeridos por feature visual
```text
specs/<feature>/
  product-design.md
  spdd-frontend.md
  prototype.md
  prototype-validation.md
  ui-test-cases.md
  traceability.md
  prototype-html5/
    index.html
    flujo.md
    decisiones-ux.md
```

## Respuesta esperada del agente
Antes de decir "listo", el agente debe reportar:
- verticales/specs prototipadas;
- rutas creadas o actualizadas;
- pantallas y flujos cubiertos;
- RF/RN cubiertos;
- estados UI cubiertos;
- validaciones y permisos cubiertos;
- gaps pendientes;
- gates que siguen pendientes de validacion humana.

## Propósito
Esta rúbrica permite que un agente o revisor humano clasifique un prototipo HTML5 en cuatro niveles y determine si puede avanzar. El objetivo es eliminar la ambiguedad: o pasa el nivel mínimo o queda bloqueado.

## Regla operativa
`gate-prototype-ready` solo puede marcarse como `Listo para validación` si el prototipo alcanza **nivel 2 o nivel 3**.  
Para productos enterprise o con stakeholders externos se recomienda **nivel 3**.

---

## Nivel 0 — Rechazado

El prototipo no cumple el propósito mínimo. Debe rehacerse antes de cualquier revisión.

| Criterio | Descripción |
|---|---|
| Parece documento HTML | La pantalla se parece a una especificación exportada, no a un producto. |
| Muestra etiquetas metodológicas | RF-, gate-, Formulario-spec, Actividad de ejemplo, Permiso activo, Ruta interna, Componente:, Spec técnica son visibles en pantalla. |
| Sin flujo navegable | No hay transición entre estados o vistas. Un clic no hace nada relevante. |
| Sin estados UI | No hay loading, empty, error, success ni permission denied. |
| Sin datos mock representativos | Textos como "Lorem ipsum", "Datos aquí" o campos vacíos sin ejemplo. |
| Shell genérica para cualquier dominio | La misma estructura serviría para salud, ecommerce, streaming y backoffice sin cambiar nada. |

---

## Nivel 1 — Básico

El prototipo es navegable pero requiere explicación externa para entenderse.

| Criterio | Descripción |
|---|---|
| Tiene navegación | Al menos un flujo de pantalla A a pantalla B funciona. |
| Tiene datos mock | Los campos tienen valores de ejemplo del dominio. |
| Tiene al menos 2 estados UI | Por ejemplo: formulario vacío y resultado con datos. |
| Sirve para explicar | Un revisor puede seguirlo si el agente lo guía verbalmente. |

**Limitación:** No es suficiente para validar UX sin explicación. No avanza a `gate-prototype-ready`.

---

## Nivel 2 — Aceptable

El prototipo permite validar el flujo principal sin explicación externa.

| Criterio | Descripción |
|---|---|
| Tarea principal navegable | El usuario puede recorrer la tarea central de inicio a fin. |
| Estados completos | Tiene loading, empty, error, success y permission denied cuando aplica. |
| Lenguaje del dominio | Todos los labels visibles pertenecen al producto, no a la metodología. |
| Feedback visible | Tiene toast, modal o mensaje recuperable para acciones importantes. |
| Sin etiquetas metodológicas | No aparecen RF, gates, permisos técnicos ni instrucciones para el agente. |
| Datos mock verosímiles | Los datos parecen reales para el dominio. |

**Gate:** Puede avanzar a `gate-prototype-ready` con resultado `Listo para validación`.

---

## Nivel 3 — Producto real

El prototipo parece una porción real del producto. Un stakeholder puede validarlo sin acompañamiento.

| Criterio | Descripción |
|---|---|
| Primer viewport comunica producto | En los primeros 10 segundos se entiende qué producto es, quién lo usa y cuál es la acción principal. |
| Patrón visual correcto para el dominio | Streaming parece streaming, player parece player, backoffice parece backoffice. |
| Datos mock ricos | Múltiples registros con variedad de estados, fechas y actores del dominio. |
| Navegación realista | Hub ↔ feature bidireccional, menús, tabs, breadcrumb o el patrón propio del dominio. |
| Diferencias por rol/perfil | La experiencia cambia según el perfil activo (admin, operador, perfil infantil, etc.). |
| Responsive básico | Funciona en mobile y desktop sin romper layout. |
| Decisiones UX documentadas | `decisiones-ux.md` registra dominio, actor, patrón elegido y razón para no usar shell genérica. |
| Sistema visual consistente | Todos los prototipos del portafolio comparten tokens base: colores, tipografía, botones, modales, toasts. |

**Gate:** Avanza a `gate-prototype-ready` con resultado `Listo para validación`.  
**Recomendado** para enterprise y proyectos con validación con cliente final.

---

## Tabla de decisión rápida

| Pregunta | Si NO → nivel máximo |
|---|---|
| ¿El primer viewport comunica el producto sin explicación? | Nivel 1 |
| ¿Hay una tarea principal navegable de inicio a fin? | Nivel 1 |
| ¿Los estados UI existen como comportamiento natural, no como checklist? | Nivel 1 |
| ¿No hay etiquetas metodológicas visibles en pantalla? | Nivel 0 si las hay |
| ¿Los datos mock son del dominio real? | Nivel 1 |
| ¿El feedback (toast/modal) existe para acciones importantes? | Nivel 1 |
| ¿El patrón visual corresponde al dominio (no es shell genérica)? | Nivel 1 |
| ¿Hay diferencias por rol/perfil visibles? | Nivel 2 |
| ¿El prototipo es responsive? | Nivel 2 |

## Términos prohibidos como texto visible en pantalla

Si cualquiera de estos aparece como **texto visible en la UI** del prototipo de una spec (no del hub), el nivel es automáticamente 0:

```
RF-
gate-
Formulario-spec
Actividad de ejemplo
Permiso activo
Ruta interna
Componente:
Spec técnica
Angular futuro:
```

> **Nota v12.20**: las palabras de "simulación" (`Contrato mock`, `Recorrido
> simulado`, `Proceso asincrono mock`, `Login OIDC simulado`) **ya no son
> prohibidas** — y tampoco se exige el banner "Prototipo de validación / datos
> de demostración". El prototipo HTML5 debe verse como producto real; la
> condición de "simulación" se documenta en `prototype-validation.md` y en el
> hub, no en cada pantalla del prototipo.

Estos términos pueden existir como comentarios de código JavaScript (`//`) o en documentos Markdown de soporte, pero nunca como contenido visible del producto.

## Red flags adicionales por dominio

| Dominio | Red flag de nivel 0 |
|---|---|
| Streaming / contenido | No hay topbar, catálogo visual ni acción de reproducción |
| Player | No hay video 16:9 ni controles de playback |
| Control parental | No hay diferencia de experiencia por perfil (adulto vs niño) |
| Backoffice editorial | No hay cola de revisión ni acción aprobar/rechazar |
| E-commerce | No hay catálogo visual ni flujo de carrito/checkout |
| SaaS operativo | No hay tabla con filtros ni detalle de registro |
| Educación | No hay lección navegable ni progreso |

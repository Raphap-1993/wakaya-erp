## Estilo de arquitectura
Wakaya ERP usa un monolito modular sobre Next.js App Router. La misma aplicacion sirve el sitio publico, el backoffice administrativo y los route handlers de API. El MVP prioriza baja complejidad operativa, trazabilidad de reservas, seguridad compatible con sesion/OIDC y control editorial del contenido publico.

## Frontend
- Sitio publico localizado en `src/app/[locale]/` con rutas ES/EN para home, bungalows y paginas informativas.
- Backoffice administrativo en `src/app/admin/` para reservas, booking requests, ocupacion, bungalows, usuarios y quejas.
- Componentes React y estilos por modulo dentro de la misma aplicacion Next.js.

## Backend
- Route handlers en `src/app/api/` para reservas, booking requests, bungalows, auth, usuarios admin, quejas, salud e integraciones puntuales.
- Logica de dominio concentrada en `src/lib/reservations/`, `src/lib/backoffice-auth/` y `src/middleware/authn.ts`.
- No existe un backend separado del frontend en el repo actual; UI y API se despliegan como un solo artefacto.

## Base de datos
- Base principal: PostgreSQL `wakaya_erp`.
- Migraciones SQL versionadas en `db/migrations/`.
- Tablas operativas principales: `reservation`, `reservation_occupancy`, `reservation_audit`, `booking_request`, `bungalow`, `bungalow_public_content`, `backoffice_user` y `backoffice_session`.
- Cuando `DATABASE_URL` no esta disponible, el proyecto usa fallback local controlado con snapshots en `.data/` para reservas y autenticacion de backoffice.

## Seguridad
- Acceso administrativo principal mediante sesion persistida de backoffice y cookie dedicada.
- Compatibilidad con OIDC/JWT para APIs protegidas via `OIDC_ISSUER`, `OIDC_JWKS_URL`, `OIDC_AUDIENCE` y `OIDC_ROLES_CLAIM`.
- RBAC en codigo con roles `viewer`, `editor`, `approver` y `admin`.
- Permisos canonicos actuales: `reservation:read`, `reservation:write`, `reservation:assign`, `reservation:approve`, `complaint:*` y `admin:users`.

## Contenido publico y localizacion
- El sitio publico opera con rutas localizadas ES/EN bajo `src/app/[locale]/`.
- El contenido editorial de bungalows se persiste en `bungalow_public_content`, con `locale_content`, orden, bandera de home y recursos visuales.
- La publicacion del contenido visible debe controlarse desde backoffice sin depender de un CMS externo ni de despliegues por cada cambio.

## Observabilidad
- Endpoint de salud en `src/app/api/health/route.ts`.
- Logs estructurados con `pino` / `pino-http`.
- Cobertura automatizada con Vitest y Playwright para flujos criticos del producto.

## Despliegue
- Un solo servicio Next.js empaqueta sitio publico, backoffice y APIs.
- La configuracion de base de datos y autenticacion depende de variables de entorno.
- PostgreSQL es la persistencia esperada para entornos operativos; el fallback local queda restringido a bootstrap, desarrollo o contingencia controlada.

<a id="principios"></a>
## Principios
- Modelo explicito: toda entidad se documenta antes de implementarse.
- Migraciones versionadas: cada cambio de esquema es inmutable y ordenable.
- Compatibilidad hacia atras por al menos un release menor (`N-1`).
- Separacion entre datos maestros y datos operativos para simplificar retencion y anonimizacion.

<a id="estado-versus-datos"></a>
## Estado versus datos
- Estado del agregado: vive en la tabla principal y refleja la invariante de negocio.
- Datos auditables: tablas espejo (`*_historial`, `*_auditoria`) alimentadas por eventos de dominio o triggers controlados.
- Lecturas proyectadas: si hay pantallas pesadas, vistas materializadas o tablas derivadas.

<a id="migraciones"></a>
## Migraciones
La plantilla usa Flyway (SQL plano) como referencia canonica. Otros motores equivalentes (Liquibase, Prisma Migrate, Knex) son aceptables si el ADR correspondiente lo justifica.

Reglas:

- Nombre del archivo: `V{YYYYMMDDHHmm}__descripcion_breve.sql`.
- Un archivo por cambio logico. No mezclar esquemas y datos semilla.
- Migraciones idempotentes para operaciones seguras (`CREATE INDEX IF NOT EXISTS`).
- Prohibidas las migraciones que requieren downtime sin ventana planificada.

Ubicacion por stack:

```
stacks/java-monolith/template/src/main/resources/db/migration/
stacks/quarkus-angular/template/backend/src/main/resources/db/migration/
stacks/spring-react/template/backend/src/main/resources/db/migration/
stacks/node-next/template/migrations/            # cuando se agregue node-pg-migrate
```

<a id="convenciones-de-nombres"></a>
## Convenciones de nombres
- Tablas en singular: `expediente`, `usuario`, `auditoria`.
- Columnas en `snake_case`, timestamps en UTC.
- Claves foraneas con el patron `<tabla>_id`.
- Indices con prefijo `ix_<tabla>_<columnas>`.
- Constraints explicitos: `uq_`, `fk_`, `ck_`.

<a id="estrategia-de-rollback"></a>
## Estrategia de rollback
- Cambios aditivos (nueva columna opcional, tabla nueva) requieren solo plan de despliegue.
- Cambios destructivos (drop, rename con perdida) siempre se ejecutan en dos releases: despliegue A marca el campo como obsoleto, despliegue B lo elimina.
- Toda release que incluye migracion declara en `releases/vX.Y.Z.md` la seccion `Datos` con pasos de rollback.

<a id="checklist"></a>
## Checklist
- [ ] Cada migracion tiene ADR o referencia a la feature que la motiva.
- [ ] Las migraciones han sido probadas contra una copia de los datos reales (o sinteticos equivalentes).
- [ ] Existe un procedimiento documentado de rollback.
- [ ] Las migraciones corren automaticamente al arrancar la aplicacion o en un job dedicado del pipeline.

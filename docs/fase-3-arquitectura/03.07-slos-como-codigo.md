<a id="objetivo"></a>
## Objetivo
Expresar los SLOs del servicio como artefactos versionados y revisados con la misma rigurosidad que el codigo: un archivo fuente (`ops/observability/slos.yaml`), un pipeline que valida sintaxis y coherencia, y un generador que traduce la declaracion a reglas de monitoreo concretas.

Evita que los SLOs vivan solo en wikis o dashboards donde nadie los revisa. Da trazabilidad entre la promesa al negocio y las alertas que despiertan al oncall.

<a id="formato-canonico-openslo-v1"></a>
## Formato canonico: OpenSLO v1
Se adopta [OpenSLO v1](https://github.com/OpenSLO/OpenSLO) porque:
- Es neutral al vendor (Pyrra, Sloth, Grafana Labs, Nobl9 lo soportan).
- Separa `Service`, `SLI`, `SLO` y `AlertPolicy` como recursos distintos y reutilizables.
- Permite declarar `ratioMetric` (good/total) o `thresholdMetric` segun el tipo de senal.
- Incluye `alertPolicies` con condiciones por `burnrate`, lo que alinea con el modelo de Google SRE.

Todos los recursos viven en `ops/observability/slos.yaml`. El archivo se instancia como parte del flujo `new-service.mjs`: los tokens `wakaya-erp-api` y `wakaya-erp` se sustituyen automaticamente.

<a id="slos-baseline-incluidos"></a>
## SLOs baseline incluidos
El baseline de la plantilla declara dos SLOs obligatorios para cualquier servicio con trafico sincrono:

| SLO | Target | Ventana | Error budget mensual |
| --- | --- | --- | --- |
| Disponibilidad (no-5xx) | 99.9 por ciento | 28d rolling | 43.2 minutos |
| Latencia p95 bajo 400ms | 95 por ciento de requests | 28d rolling | - |

Para servicios asincronos, anadir SLOs sobre `event-processing-latency` y `consumer-lag`. Para servicios batch, sobre `job-success-rate` y `freshness`.

<a id="alertas-multi-window-multi-burn-rate"></a>
## Alertas multi-window multi-burn-rate
Siguiendo la recomendacion de Google SRE Workbook capitulo 5, la plantilla declara dos policies por SLO:

- `burn-rate-fast` (14.4x, ventana 1h + ventana corta 5m). Page al oncall. Alerta cuando se agotaria 2 por ciento del budget mensual en una hora.
- `burn-rate-slow` (6x, ventana 6h + ventana corta 30m). Abre ticket. Alerta ante deterioro sostenido.

La combinacion de ventanas (una larga + una corta) reduce falsos positivos: solo alerta si el burn rate es alto en ambas.

<a id="generacion-de-reglas-prometheus"></a>
## Generacion de reglas Prometheus
La plantilla incluye `ops/observability/generate-slo-rules.mjs`, un generador Node cross-platform que parsea el OpenSLO y emite reglas Prometheus con:

- Recording rules `slo:sli_ratio:<ventana>:<slo>` para ventanas 5m, 30m, 1h, 2h, 6h, 1d, 3d.
- Alerting rules para `burn-rate-fast` y `burn-rate-slow` con etiquetas `severity`, `slo`.

Uso:
```
node ops/observability/generate-slo-rules.mjs \
    --in ops/observability/slos.yaml \
    --out ops/observability/prometheus-rules-slo.yaml
```

Para flujos productivos mas robustos se recomiendan los generadores oficiales: [Pyrra](https://github.com/pyrra-dev/pyrra), [Sloth](https://github.com/slok/sloth) o [OpenSLO CLI](https://github.com/OpenSLO/OpenSLO/tree/main/cli). El generador interno existe para proyectos que arrancan sin introducir otro componente.

<a id="ciclo-de-vida"></a>
## Ciclo de vida
1. El dueno de servicio abre PR modificando `ops/observability/slos.yaml`.
2. El pipeline valida el YAML y ejecuta `node ops/observability/generate-slo-rules.mjs` para confirmar que genera reglas sin errores.
3. Al merge, un workflow aplica las reglas al Prometheus/Thanos del ambiente correspondiente.
4. El dashboard Grafana consume las recording rules con prefijo `slo:sli_ratio:*`.
5. En cada retro mensual se revisa consumo real de error budget; si se agota repetidamente, se replantea target o se invierte en fiabilidad.

<a id="como-adoptarlo-en-un-proyecto-real"></a>
## Como adoptarlo en un proyecto real
1. Instanciar la plantilla con `new-service.mjs`. El archivo `slos.yaml` viene con tokens sustituidos y SLOs default.
2. Revisar los targets con el product owner: 99.9 por ciento suele ser excesivo para un servicio interno; ajustar a la realidad.
3. Anadir SLOs especificos del dominio (ej. "95 por ciento de expedientes procesados en menos de 2 minutos").
4. Vincular cada SLO con la metrica producto definida en `docs/fase-0-iniciacion/00.08-product-analytics.md`.
5. Verificar en el dashboard `ops/observability/grafana-dashboard-api.json` que el panel de disponibilidad 30d consume la recording rule generada.
6. Registrar el SLO en el runbook del servicio y en la documentacion de ADR-004.

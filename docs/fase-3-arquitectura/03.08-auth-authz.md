<a id="objetivo"></a>
## Objetivo
Dejar explicito el baseline de autenticacion y autorizacion de la plantilla con instanciaciones concretas por stack, de forma que al adoptar la plantilla no haya que redisenar la capa de seguridad desde cero.

<a id="authn-baseline-jwt-firmado-oidc"></a>
## AuthN baseline: JWT firmado OIDC
- Protocolo: OAuth 2.1 / OpenID Connect para login interactivo y Client Credentials para servicio-a-servicio.
- Formato de token: JWT firmado RS256 o ES256. HS256 solo permitido en ambientes dev locales.
- Descubrimiento: `/.well-known/openid-configuration` del IdP con JWKS publico.
- Vida del access token: 10-15 minutos.
- Refresh token: rotacion obligatoria (cada uso invalida el anterior).
- PKCE obligatorio en flujos interactivos.

Claims estandar esperados en el access token:
- `sub` (subject, identificador estable del usuario).
- `aud` (audience, debe incluir el servicio consumidor).
- `iss` (issuer URL del IdP).
- `exp`, `iat`, `nbf`.
- `scope` (string con scopes separados por espacio).
- `tenant_id` o equivalente para multi-tenant.

Claims adicionales custom:
- `roles` (array de strings con roles del usuario).
- `groups` (array de strings con grupos Backstage/LDAP).
- `session_id` (para auditoria y logout distribuido).

<a id="authz-baseline-rbac-mas-claims"></a>
## AuthZ baseline: RBAC mas claims
- RBAC como estructura base: roles agrupan permisos; usuarios se asignan a roles.
- Atributos adicionales (ABAC) se evaluan solo cuando RBAC no basta (ej. "solo puede ver recursos de su region").
- Los permisos se expresan como acciones sobre el recurso de dominio: `reservation:read`, `reservation:write`, `reservation:approve`. Tras `new-service.mjs` se sustituyen por el nombre real del recurso (ej. `case:read`, `customer:read`).
- El token porta roles; el servicio resuelve permisos localmente a partir de la tabla de roles (no se consulta al IdP en cada request).

<a id="identity-provider-recomendado"></a>
## Identity provider recomendado
Evaluacion por tipo de organizacion:
- SaaS B2C con soporte multiregional: Auth0 u Okta Customer Identity.
- Empresa con AD/Azure AD: Microsoft Entra ID (anteriormente Azure AD).
- Self-hosted open source: Keycloak con realm dedicado.
- Startup pequena: Auth0 Free tier hasta 7500 MAU, luego migrar.

El stack no depende del IdP concreto: basta con que exponga OIDC estandar.
La referencia canonica usada en ejemplos y configuraciones del repositorio es `Keycloak` con un `realm` dedicado por producto.

<a id="matriz-de-permisos-canonica"></a>
## Matriz de permisos canonica
La matriz baseline (con el placeholder `reservation` sustituido por el recurso real al instanciar via `scripts/new-service.mjs`):

| Rol | `reservation:read` | `reservation:write` | `reservation:approve` | `admin:users` |
| --- | --- | --- | --- | --- |
| viewer | si | no | no | no |
| editor | si | si | no | no |
| approver | si | si | si | no |
| admin | si | si | si | si |

Ejemplos de instancias reales tras `init-project.mjs --config <empresa>.json`:
- caso "case management": `case:read`, `case:write`, `case:approve`, `admin:users`.
- caso "customer 360": `customer:read`, `customer:write`, `customer:approve`, `admin:users`.

La matriz real debe vivir en un ADR de fase 3 con justificacion por rol, y reflejarse en el codigo como `stacks/node-next/template/src/lib/rbac.ts` o el equivalente del stack elegido.

<a id="instanciacion-por-stack"></a>
## Instanciacion por stack
- Node + Next.js: middleware en `stacks/node-next/template/src/middleware/authn.ts` que valida JWT contra JWKS cacheado, extrae claims y los pone en `request.context.user`. AuthZ en `stacks/node-next/template/src/lib/rbac.ts` consumido por handlers.
- Java monolith (Spring Boot): `SecurityConfig` con `oauth2ResourceServer().jwt()` configurado contra el JWKS del IdP, mas `@PreAuthorize` sobre metodos sensibles.
- Quarkus: `quarkus-oidc` con `mp.jwt.verify.publickey.location` apuntando al JWKS del IdP y anotacion `@RolesAllowed`.
- Spring + React: igual que java monolith en el backend; el frontend usa `react-oidc-context` para flujo PKCE.

<a id="auditoria-de-eventos-de-acceso"></a>
## Auditoria de eventos de acceso
Cada intento de acceso fallido o privilegio elevado genera un evento conforme a `contracts/events/audit-log.schema.json`:
- `eventType=auth.denied`: cuando el token es invalido o el rol no tiene permiso.
- `eventType=auth.privileged`: cuando el usuario ejerce un permiso admin.
- `eventType=auth.session.revoked`: al hacer logout global.

Estos eventos alimentan un panel de seguridad en Grafana y reglas de alerta por anomalias (pico de denies, sesiones sospechosas).

<a id="anti-patrones"></a>
## Anti-patrones
- Emitir tokens sin `aud`: permite reutilizacion entre servicios.
- Firmar con secreto compartido (HS256) en produccion: compromete a todos los servicios con el mismo secreto.
- Hardcodear roles en el codigo sin central de verdad: genera drift.
- Guardar JWT en localStorage sin XSS hardening: preferir cookie httpOnly + SameSite=strict.
- Dejar expiracion larga (mas de 1 hora) sin rotacion: expande la ventana de abuso tras robo.

## Objetivo
Cerrar el flujo MVP de reservas web sin pasarela de pago automatizada:

1. la web publica recibe una solicitud
2. Wakaya envia instrucciones por correo transaccional
3. el cliente responde en el mismo hilo con su comprobante
4. back office revisa adjuntos, responde y confirma manualmente
5. el sistema convierte la solicitud en reserva solo si no existe conflicto operativo

## Componentes
- `PostgreSQL`: fuente real para `booking_request`, `message_thread`, `message_item`, `message_attachment`, `availability_conflict` y `outbound_email`
- `Resend`: salida transaccional inicial desde `reservas@wakayaecolodge.com`
- `Zoho Mail`: continuidad humana del hilo, lectura de respuestas y reply operativo
- `Back office`: panel interno para sync, reply, confirmacion y revision de conflictos

## Flujo end to end
1. `POST /api/public/booking-requests`
   crea `booking_request`, crea `message_thread`, envia el correo inicial por Resend y registra el outbox.
2. El cliente responde al alias `reservas@wakayaecolodge.com`.
3. `POST /api/booking-requests/:id/sync`
   busca el thread en Zoho cuando todavia no existe `provider_thread_id`, sincroniza mensajes, hidrata adjuntos y actualiza `sync_status`.
4. `POST /api/booking-requests/:id/reply`
   responde desde back office dentro del mismo hilo humano usando Zoho.
5. `POST /api/booking-requests/:id/confirm-transfer`
   crea la reserva solo si no existe choque operativo; si existe, registra `availability_conflict` y responde `409 occupancy_conflict`.

## Regla MVP de conflicto
- El bloqueo operativo se dispara cuando la solicitud tiene `requestedBungalowType` y existe una reserva operacional superpuesta para ese mismo bungalow/tipo.
- Si la solicitud todavia no tiene un tipo de bungalow definido, no se bloquea automaticamente la conversion; la decision queda para la asignacion operativa posterior.

## Persistencia nueva
- `booking_request.thread_key`
- `message_thread`
- `message_item`
- `message_attachment`
- `availability_conflict.metadata`

## Decisiones practicas
- No hay checkout con pasarela.
- El comprobante via correo y adjuntos es parte del flujo MVP desde el dia 1.
- El back office no debe salir a JSON crudo: reply, sync y confirmacion se ejecutan desde el panel.
- El correo inicial transaccional y la conversacion humana comparten el mismo alias de negocio.

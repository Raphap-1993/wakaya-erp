<a id="objetivo"></a>
## Objetivo
Convertir requerimientos y SPDD aprobado en especificaciones ejecutables, trazables y listas para construccion front/back.

<a id="entrada-minima"></a>
## Entrada mínima
- vision y alcance definidos
- requerimientos funcionales y no funcionales aprobados
- `gate-spdd-approved` para features visuales
- arquitectura base definida cuando la feature afecte decisiones estructurales

<a id="flujo-estandar"></a>
## Flujo estándar
1. Product Design validado
2. SPDD con prototipo aprobado
3. Spec funcional final
4. Spec tecnica front/back
5. API contract si aplica backend
6. Spec de tareas con plan ejecutable
7. Worktree o rama de feature
8. Implementacion con TDD
9. Revision de codigo
10. Pruebas y QA
11. Validacion

<a id="que-cambia-entre-requerimientos-y-specs"></a>
## Que cambia entre requerimientos y specs
`SDD` no reemplaza el analisis de requerimientos. Lo toma como entrada y lo baja a una feature construible.

| Artefacto | Pregunta que responde | Nivel | Contenido esperado |
|---|---|---|---|
| Requerimiento funcional | que necesita el negocio | producto o capacidad | necesidad, alcance y criterio funcional general |
| Spec funcional | como debe comportarse una feature concreta | feature | actores, flujo, criterios de aceptacion, fuera de alcance, casos borde |
| Spec tecnica | como se implementa la feature | implementacion | contratos, datos, errores, seguridad, observabilidad, riesgos, pruebas |
| API contract | que necesita el backend desde la experiencia aprobada | integracion | endpoint, request, response, errores, permisos y validaciones |
| Spec de tareas | que trabajo se ejecuta | construccion | lista de trabajo concreta para FE, BE, QA, ops u otros roles |

### Regla practica
- Si habla de necesidad del negocio, va en `requerimientos funcionales`.
- Si habla del comportamiento esperado de una feature puntual, va en `spec funcional`.
- Si habla de endpoints, tablas, contratos o componentes, va en `spec tecnica`.
- Si nace de campos, acciones, permisos o errores visibles del prototipo, debe reflejarse en `api-contract.md`.
- Si habla de pasos de implementacion o checklist de trabajo, va en `spec de tareas`.
- Si una tarea sera ejecutada por proveedor IA, debe incluir objetivo, entradas, rutas permitidas, ciclo TDD, comandos de verificacion y evidencia esperada.
- Si la tarea toca frontend, debe enlazar UX, prototipo o mapping Penpot -> Angular para habilitar SPDD Frontend.

### Ejemplo rapido
- Requerimiento funcional: `RF-03 Cambiar estado del expediente con validaciones por rol`.
- Spec funcional: define quien cambia el estado, desde donde, que validaciones ve y que resultados acepta negocio.
- Spec tecnica: define `PATCH /api/expedientes/{id}/estado`, payload, errores, auditoria, seguridad y riesgos.
- Spec de tareas: divide el trabajo en backend, frontend, pruebas y despliegue.

<a id="reglas"></a>
## Reglas
- Cada spec debe referenciar el requerimiento o backlog item que la origina.
- Ninguna feature visual debe cerrar SDD sin `gate-spdd-approved`.
- Si una feature fuerza una decision tecnologica o arquitectonica nueva, debe registrarse ADR.
- La spec tecnica debe cubrir contrato, datos, errores, seguridad, observabilidad, riesgos y estrategia de pruebas.
- Las tareas deben derivarse de la spec tecnica y permitir seguimiento de implementacion.
- Las tareas deben ser pequenas y verificables segun [Flujo de delivery IA para proveedores](../transversal/90.33-flujo-delivery-ia-proveedores.md).
- Las tareas frontend deben declarar dependencia a Fase 2: wireframe, prototipo, sistema de componentes o mapping.
- No se debe pasar a `/build` si `spec-tareas.md` no permite ejecutar una tarea sin adivinar archivos, pruebas o comandos.

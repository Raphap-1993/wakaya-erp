## Objetivo
Definir como construir frontend despues de Product Design, SPDD aprobado y SDD tecnico, usando IA como ejecutor controlado bajo Frontend Dev, UX/UI, QA, Tech Lead y Codex.

## Nivel
Construccion frontend.

## Roles
| Rol | Responsabilidad |
|---|---|
| Frontend dev | implementa componentes, rutas, estados y pruebas |
| UX/UI | valida consistencia con prototipo y sistema de componentes |
| QA | valida criterios, evidencias y regresiones |
| Tech Lead | controla arquitectura frontend y riesgos |
| Codex o proveedor IA | ejecuta tareas chicas con TDD y deja evidencia |

## Flujo canonico de consumo
```text
Product Design + SPDD + Prototype Ready + SPDD Approved + SDD
  -> spec-tareas frontend
  -> worktree/rama
  -> TDD por componente o flujo
  -> implementacion minima
  -> verificacion frontend
  -> review UX/tech
  -> gate-frontend-spdd-ready
  -> gate-4-6
```

## Salidas
- componentes frontend,
- rutas y estados de UI,
- datos mock o integracion segun alcance,
- pruebas unitarias, componente o e2e,
- evidencia de comandos,
- diferencias contra prototipo,
- review y riesgos residuales.

## Reglas de operacion
- No construir frontend si falta `gate-spdd-approved`.
- No construir frontend si falta `gate-prototype-ready` o su excepcion documentada.
- No construir frontend si falta `spec-tareas.md`.
- No introducir componente nuevo sin revisar el mapping Penpot -> Angular.
- No conectar API real durante prototipado HTML5; la API real se conecta solo en construccion aprobada.
- No cerrar tarea con comportamiento visible sin prueba o evidencia manual.
- Si UX y spec se contradicen, se bloquea la tarea y se actualizan preguntas abiertas.

## Checklist de salida
- [ ] Cada tarea tiene ID, objetivo, rutas permitidas y comandos.
- [ ] Cada componente nuevo tiene origen en UX/spec o queda agregado al mapping.
- [ ] Los estados loading, empty, error, success y unauthorized estan cubiertos o justificados.
- [ ] Hay evidencia red-green-refactor o justificacion.
- [ ] Hay typecheck/test/e2e o evidencia browser segun riesgo.
- [ ] Hay review si cambia UX critica, accesibilidad, contrato o seguridad.

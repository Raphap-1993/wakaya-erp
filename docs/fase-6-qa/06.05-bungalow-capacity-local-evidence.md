## Resultado

El reemplazo operativo del inventario individual por cupos agregados quedó
implementado y validado en local. No se ejecutó ningún despliegue a producción.

## Base migrada

- backup previo: `backups-local/2026-07-12-before-bungalow-capacity.dump`;
- formato: custom dump PostgreSQL 16, 196 KB, 179 entradas;
- SHA-256: `3372dbb32a9fa69151b4436a79d69c448b7b9bfd65bd65c0e1b7de55137660bc`;
- dry-run transaccional: 5 categorías, 17 cupos, 14 bloqueos y 7 reservas;
- rollback del dry-run verificado: `bungalow_capacity` no permaneció creada;
- migración aplicada: `012_bungalow_capacity_inventory.sql`;
- resultado persistido: Familiar 5, Individual 5, Matrimonial 4, Doble 2 y Triple 1;
- legado retenido: 17 unidades activas y 14 bloqueos para auditoría/rollback.

El `pg_dump` local era versión 14 y el servidor versión 16. Se usó el binario
16.13 del contenedor `wakaya-erp-postgres` y se verificó el archivo con
`pg_restore -l` antes de migrar.

## Escenarios funcionales cubiertos

- checkout exclusivo y mínimo nocturno del rango;
- solicitudes pendientes no consumen cupo;
- Doble admite dos compromisos y rechaza el tercero;
- bloqueo por cantidad reduce disponibilidad y cancelación la repone;
- reducción por debajo de compromisos devuelve `capacity_below_commitments`;
- asignación, confirmación web y OTA usan cupos agregados y dejan
  `bungalowUnitId` en `null` como legado técnico;
- una reserva solapada no marca agotada una categoría con cupos restantes;
- el calendario público solo bloquea rangos completamente agotados;
- `inventory:manage` permanece exclusivo de Administrador.

## Deuda externa al incremento

`npm run lint` global no está verde: la línea base actual reporta 99 errores y
489 advertencias distribuidos en scripts CI, superficies heredadas del
backoffice y pruebas previas. El código nuevo de `bungalow-capacity`, sus APIs,
UI, pruebas y E2E sí pasa el lint focalizado. Esta deuda no se corrigió para no
mezclar cambios ajenos en un árbol ampliamente modificado.

## Gate de salida

La demo local está disponible y el incremento no debe desplegarse todavía.
Producción requiere aprobación explícita posterior, revalidación del runbook y
una decisión sobre la deuda de lint global antes del go/no-go.

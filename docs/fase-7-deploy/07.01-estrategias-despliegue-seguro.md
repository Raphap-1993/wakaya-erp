## Objetivo
Definir como la plantilla liberar cambios en produccion sin afectar a los usuarios, combinando estrategias progresivas de despliegue, feature flags y plan de rollback automatizado.

## Principios
- Cada release debe ser reversible en menos de cinco minutos.
- Cambios de riesgo alto viajan detras de feature flag y se activan de forma gradual.
- Los gates de promocion usan senales objetivas (tasa de error, latencia, SLO) y no percepcion humana.
- Production nunca recibe una imagen que no haya pasado staging y firma cosign.
- Todo despliegue queda trazado en el repo (GitOps) y en el sistema de feature flags.

## Estrategias soportadas

### 1. Canary con Argo Rollouts
- Imagen promovida en pasos 10% - 25% - 50% - 75% - 100%.
- Cada paso valida un `AnalysisTemplate` (Prometheus) antes de avanzar.
- Metrica de referencia: `success_rate >= 0.995` sobre ventana de cinco minutos.
- Si el analisis falla, el rollout hace `abort` automatico y vuelve a la version estable.
- Ver `ops/k8s/rollouts/canary.yaml` y `ops/k8s/rollouts/README.md`.

### 2. Blue/green
- Indicado para servicios con estado en cache, workers largos o cortes de contrato.
- La version azul sigue sirviendo mientras la verde se calienta y pasa smoke tests.
- El switch es un cambio de `Service.selector` (o del target group del ALB) y dura segundos.
- Rollback es revertir el selector al color anterior.

### 3. Feature-flag-gated rollout
- La funcionalidad se despliega pero queda apagada para todos los segmentos.
- Se activa por cohortes: internos, beta, 5%, 25%, 50%, 100%.
- Fuente de verdad en el proveedor de feature flags via OpenFeature.
- Permite desacoplar "deployment" de "release" y apagar una funcion sin redeploy.
- Ver `docs/transversal/90.17-feature-flags.md`.

## Criterios de eleccion
- Endpoint idempotente y stateless -> canary.
- Cambio de schema de cache o de contrato -> blue/green con dual-write controlado.
- Feature nueva visible al usuario final -> feature-flag-gated sobre canary.
- Migracion destructiva -> dos releases (expansion + contraction) con feature flag intermedio.

## Gates automaticos recomendados
- Tests de contrato en CI (ver [06.00-plan-pruebas.md](../fase-6-qa/06.00-plan-pruebas.md)).
- Analisis Prometheus como parte del rollout.
- Budget de errores mensual antes de autorizar nuevas canary en produccion.
- Chequeo de politicas Kyverno (firma cosign, non-root, limits).

## Rollback
- Rollback automatico: falla el `AnalysisTemplate` del rollout, Argo revierte.
- Rollback manual rapido: `argo rollouts undo` sobre el rollout activo.
- Rollback por feature flag: apagar el flag y dejar la imagen desplegada.
- Todo rollback debe quedar anotado en el runbook de la fase 7 y en el release correspondiente.

## Evidencia esperada en el proyecto
- `ops/k8s/rollouts/` con canary + AnalysisTemplate versionados.
- `ops/k8s/overlays/prod/` referenciando el rollout.
- Entrada en `docs/fase-3-arquitectura/03.03-plan-despliegue.md` alineada con esta estrategia.
- Feature flags activos documentados por release en `releases/vX.Y.Z.md`.

## Base de datos
- `DATABASE_URL` apuntando a PostgreSQL
- migraciones aplicadas: `001_hybrid_reservations.sql`, `002_mail_threads_and_conflicts.sql` y `003_backoffice_auth.sql`
- no depender de `WAKAYA_RESERVATIONS_DB_PATH` ni `WAKAYA_BACKOFFICE_AUTH_PATH` en despliegue operativo

## Correo transaccional
- `RESEND_API_KEY`
- dominio y remitente habilitados para `reservas@wakayaecolodge.com`
- SPF y DKIM de Resend validados en el dominio

## Correo operativo humano
- `ZOHO_MAIL_ACCOUNT_ID`
- `ZOHO_MAIL_ACCESS_TOKEN`
- buzón real activo para `reservas@wakayaecolodge.com`
- SPF y DKIM de Zoho validados en el dominio

## Seguridad y acceso
- `BACKOFFICE_BOOTSTRAP_EMAIL`
- `BACKOFFICE_BOOTSTRAP_NAME`
- `BACKOFFICE_BOOTSTRAP_PASSWORD`
- `BACKOFFICE_BOOTSTRAP_ROLES`
- `BACKOFFICE_SESSION_TTL_HOURS`
- `OIDC_JWKS_URL`
- `OIDC_ISSUER`
- `OIDC_AUDIENCE`

Nota: el build local actual pasa, pero emite advertencias si estas variables OIDC no estan presentes.

## Verificacion minima previa a despliegue
1. `./node_modules/.bin/tsc --noEmit`
2. `npm run build`
3. pruebas focalizadas de reservas, correo y conflictos
4. prueba manual completa:
   - crear solicitud web
   - verificar que aparezca en `/admin/reservations/requests`
   - validar login interno en `/login`
   - validar `/admin/users` y que el roster persista
   - verificar correo inicial
   - responder con adjunto al alias
   - sincronizar hilo
   - responder desde back office
   - confirmar transferencia o validar conflicto

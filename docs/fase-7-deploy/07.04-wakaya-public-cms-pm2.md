## Precondiciones

- commit de `main` con pruebas, TypeScript, ESLint y build aprobados;
- backup verificable de PostgreSQL y del directorio de medios;
- `DATABASE_URL` configurado en el entorno de PM2;
- `WAKAYA_MEDIA_STORAGE_PATH` absoluto, persistente y fuera del directorio de releases, por ejemplo `/srv/wakaya/shared/media`;
- usuario de PM2 con lectura/escritura sobre el directorio compartido y sin acceso de escritura global;
- espacio libre suficiente para conservar el release anterior, backup y nuevas variantes.

## Backup previo

Ejecutar como el usuario operativo y reemplazar los marcadores por rutas explícitas validadas:

```bash
install -d -m 700 /srv/wakaya/backups/<timestamp>
pg_dump --format=custom --no-owner --file=/srv/wakaya/backups/<timestamp>/wakaya.dump "$DATABASE_URL"
tar -C /srv/wakaya/shared -czf /srv/wakaya/backups/<timestamp>/media.tar.gz media
sha256sum /srv/wakaya/backups/<timestamp>/wakaya.dump /srv/wakaya/backups/<timestamp>/media.tar.gz
pm2 save
```

Validar el dump con `pg_restore --list` y registrar el commit/release que PM2 sirve antes de continuar. El backup de base y medios debe compartir el mismo timestamp lógico.

## Preparar release

```bash
git fetch origin main
git worktree add --detach /srv/wakaya/releases/<commit> <commit>
cd /srv/wakaya/releases/<commit>
npm ci
npm run migrate
npm run build
```

No copiar medios dentro del release ni de `.next`. Configurar en PM2:

```text
DATABASE_URL=<postgres-production-url>
WAKAYA_MEDIA_STORAGE_PATH=/srv/wakaya/shared/media
NODE_ENV=production
```

La aplicación standalone debe conservar acceso al directorio compartido. Verificar propietarios con `stat` y una escritura/lectura de prueba ejecutada como el mismo usuario de PM2.

## Activación preparada

La activación se realizará en una ventana posterior, no como parte de este cambio:

```bash
ln -sfn /srv/wakaya/releases/<commit> /srv/wakaya/current
pm2 reload <wakaya-process-name> --update-env
curl --fail --silent http://127.0.0.1:<port>/api/health
```

El healthcheck debe reportar `persistence.contentWrites: "durable"`. Después se deben probar autenticados `/admin/content` y, en ES/EN, Home, Bungalows/listado/detalle, Servicios, Galería, Eventos, Publicaciones, Contacto, Nosotros, FAQ, Testimonios, Políticas, Pet Friendly y Libro de Reclamaciones.

## Rollback

Antes de aceptar nuevas escrituras, se puede volver al release anterior:

```bash
ln -sfn /srv/wakaya/releases/<previous-commit> /srv/wakaya/current
pm2 reload <wakaya-process-name> --update-env
```

Una vez aceptadas escrituras editoriales o nuevos assets, no restaurar ciegamente base o medios por separado. Priorizar roll-forward. Si una corrupción obliga a restaurar, detener escrituras, restaurar PostgreSQL y medios del mismo timestamp en ubicaciones aisladas, verificar referencias/variantes y recién entonces conmutar ambos recursos. Conservar el release y los assets anteriores hasta cerrar la ventana de observación.

## Verificación y observación

- recargar cada página pública después de editar desde el backoffice;
- reiniciar el proceso y repetir lectura para demostrar durabilidad;
- revisar logs de `invalid_payload`, `media_processing_failed`, conflictos de versión y fallos de escritura;
- confirmar que una imagen borrada no vuelve y que un asset aún referenciado no puede eliminarse;
- registrar commit, timestamps, checksums, migraciones aplicadas, salida de health y responsable de la ventana.

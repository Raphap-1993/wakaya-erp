<a id="objetivo"></a>
## Objetivo
Definir que se observa, con que semantica y con que umbrales, de modo que la operacion pueda detectar, diagnosticar y responder a incidentes sin depender de conocimiento tacito.

<a id="los-tres-pilares"></a>
## Los tres pilares
- Logs: estructurados en JSON con un `correlationId` obligatorio. Nivel por defecto `info`; `debug` solo en investigacion puntual.
- Metricas: expuestas via `/metrics` en formato Prometheus, con labels estandar (`service`, `version`, `env`).
- Trazas: generadas con OpenTelemetry; propagacion de `traceparent` W3C en todas las llamadas salientes.

<a id="convenciones"></a>
## Convenciones
- Nombres de metricas en `snake_case`, unidades en el sufijo (`_seconds`, `_bytes`, `_total`).
- Campos de log estandarizados: `@timestamp`, `level`, `message`, `service`, `version`, `env`, `traceId`, `spanId`, `correlationId`, `userId`.
- No se loguean secretos ni PII salvo identificadores anonimizados.
- El `correlationId` se propaga desde el primer punto de entrada (API gateway, SPA) hasta los adapters.

<a id="sli-y-slo-de-referencia"></a>
## SLI y SLO de referencia

| Servicio       | SLI                                         | SLO (30 dias)        |
|----------------|---------------------------------------------|----------------------|
| API Expedientes| disponibilidad `http_requests_total{code!~"5.."}` | 99.5%           |
| API Expedientes| latencia p95 `GET /api/expedientes`         | <= 400 ms            |
| Frontend       | errores de cliente (`window.onerror`)       | < 1% de sesiones     |
| Pipeline CI    | exito en main                                | >= 95% por semana    |

Los SLO se adaptan por proyecto y se registran en `ops/fase-8-operacion/slo.md`.

<a id="alertas"></a>
## Alertas
- Criticas (pager): violacion de SLO sostenida > 5 min, 5xx > 1% por 5 min, caida de health check.
- Mayores (ticket): colas crecientes, tiempo de respuesta degradado, errores de auth aumentan.
- Informativas (log): despliegues, rotacion de secretos, cambios de feature flag.

Cada alerta tiene runbook en `ops/fase-8-operacion/runbooks/<nombre>.md`.

<a id="dashboards"></a>
## Dashboards
Minimos esperados:

- Overview de servicio: RPS, latencia p50/p95/p99, error rate, saturacion.
- Dominio: transiciones de estado por minuto, bandejas activas por usuario, expedientes cerrados por dia.
- Infra: uso de CPU, memoria, conexiones a BD, profundidad de cola.

Definicion como codigo en `ops/fase-8-operacion/dashboards/`.

<a id="checklist"></a>
## Checklist
- [ ] Hay un `correlationId` visible en logs, trazas y respuestas de error.
- [ ] Los stacks emiten metricas Prometheus con labels estandar.
- [ ] Hay al menos un dashboard por servicio y los SLO son revisados cada trimestre.
- [ ] Cada alerta critica apunta a un runbook vigente.

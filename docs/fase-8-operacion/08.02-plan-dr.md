<a id="objetivo"></a>
## Objetivo
Garantizar continuidad del servicio ante fallos mayores: caida regional del proveedor cloud, corrupcion de datos, ransomware o perdida total del cluster Kubernetes. Este plan define objetivos medibles (RTO/RPO), procedimientos ejecutables y ensayos periodicos para validarlos.

No es una politica aspiracional. Cada punto debe estar respaldado por un runbook ensayado y por metricas observables.

<a id="rto-y-rpo-baseline"></a>
## RTO y RPO baseline
| Clasificacion | RTO | RPO | Comentario |
| --- | --- | --- | --- |
| Tier 0 (core critico, ej. identidad) | 30 minutos | 5 minutos | Replicacion sincrona multi-region o failover caliente. |
| Tier 1 (producto principal) | 2 horas | 15 minutos | Replicacion asincrona multi-region, failover tibio. |
| Tier 2 (apps internas) | 8 horas | 1 hora | Backup regular + restore automatizado. |
| Tier 3 (batch / reporting) | 24 horas | 4 horas | Re-ejecucion de jobs desde source of truth. |

La plantilla asume Tier 1 por defecto. Proyectos con requerimientos superiores deben documentar en un ADR especifico (recomendado: ADR-007-disaster-recovery).

<a id="clasificacion-de-escenarios"></a>
## Clasificacion de escenarios
- Escenario A: zona de disponibilidad (AZ) caida. Mitigacion: cluster multi-AZ por defecto, sin accion humana.
- Escenario B: region caida. Mitigacion: failover a region secundaria (runbook `ops/runbooks/dr-failover-region.md`).
- Escenario C: corrupcion logica de datos. Mitigacion: restore puntual desde backup (PITR) + feature flag para aislar trafico afectado.
- Escenario D: ransomware / compromiso masivo. Mitigacion: aislar cluster, restore desde backup inmutable (S3 Object Lock o equivalente), rotar todos los secretos.
- Escenario E: perdida del provider entero. Mitigacion: plan offline de continuidad de negocio (fuera del alcance tecnico; reportar a equipo de riesgos).

<a id="arquitectura-de-referencia"></a>
## Arquitectura de referencia
- Dos regiones activas (primary/secondary) con Argo CD apuntando al mismo repo GitOps.
- DNS por Route 53 / Cloud DNS con health checks que bajan el registro al detectar anomalia regional.
- Base de datos Postgres con replica cross-region (logical replication) o Amazon Aurora Global Database / Cloud SQL Cross-region Read Replicas.
- Almacenamiento de objetos con replicacion cross-region (S3 CRR, GCS replication).
- Secret manager con replicacion cross-region habilitada (AWS Secrets Manager Multi-Region, Google Secret Manager replicas configuradas).
- Event bus (Kafka/MSK/Pub-Sub) con mirroring activo al cluster secundario.

Diagrama recomendado en `likec4/views/dr.view.c4` (crear si no existe en el proyecto).

<a id="estrategia-de-datos"></a>
## Estrategia de datos
- Backups diarios cifrados con KMS, retenidos 30 dias en bucket con Object Lock (WORM).
- Backups semanales con retencion 12 meses.
- Ensayo mensual de restore en ambiente aislado (documentado en `ops/data/backup-restore.md`).
- WAL shipping continuo para RPO de 5-15 minutos segun Tier.
- Checksum verification al aplicar backups (evita restaurar datos corruptos).

<a id="simulacros-gameday"></a>
## Simulacros (GameDay)
Se ejecutan como minimo:
- Trimestral: failover de staging a region secundaria, verificando RTO/RPO reales contra los objetivos.
- Semestral: restore de backup en ambiente aislado y verificacion de integridad.
- Anual: simulacion de ransomware con rotacion completa de secretos y restore desde backup inmutable.

Cada GameDay produce:
- Metrica de tiempo real medido (RTO observado, RPO observado).
- Findings con clasificacion P1/P2/P3.
- Lista de mejoras con duenos y fecha compromiso.
- Entrada en `revisiones/` con la fecha del ejercicio.

<a id="evidencia-y-auditoria"></a>
## Evidencia y auditoria
- Eventos `audit-log` con `eventType=dr.failover.*` para cada accion en runbook.
- Dashboard Grafana con panel dedicado a metricas DR (tiempo desde ultimo backup exitoso, lag de replica, estado de sync cross-region).
- Evidencia de GameDays en `ops/fase-8-operacion/features/` o carpeta dedicada.

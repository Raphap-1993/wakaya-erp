<a id="diferencia-entre-agente-prompt-y-skill"></a>
## Diferencia entre agente, prompt, skill, command y gate
| Artefacto | Para que sirve | Cuando usarlo |
|---|---|---|
| Agente | representa un rol especializado | cuando necesitas una salida completa por dominio |
| Prompt | instruccion reutilizable orientada a una salida | cuando ya sabes exactamente que quieres pedir |
| Skill | capacidad reusable de trabajo o criterio | cuando quieres repetir una forma de ejecutar o revisar |
| Command | entrada por lifecycle de trabajo | cuando quieres activar `/document`, `/plan`, `/ux`, `/prototype`, `/spec`, `/build`, `/test`, `/review` o `/ship` |
| Quality gate | criterio de readiness por bloque de fases | cuando necesitas decidir si una fase puede avanzar |
| Reference | apoyo corto reusable | cuando necesitas criterio adicional sin inflar el prompt o el agent |
| Framework agent | orquestador del framework enterprise AI-first | cuando la solicitud cruza fases, commands, gates, memoria y trazabilidad |

### Regla practica
- Usa `agent` para planificar, disenar, desarrollar, probar o desplegar segun un rol.
- Usa `prompt` para pedir una spec, arquitectura, tests o estimacion concretos.
- Usa `skill` para mantener una capacidad reusable entre multiples features.
- Usa `command` para orquestar agents, prompts, skills y references segun la intencion del trabajo.
- Usa `quality gate` para exigir evidencia antes de cerrar discovery, arquitectura, SDD, QA, release u operacion.
- Usa `enterprise-ai-framework-agent.md` cuando necesites gobernar una solicitud de extremo a extremo dentro del framework.
- Usa `ai/prompts/arranque-desde-fuente-bruta.md` cuando tengas fuente bruta y necesites convertirla en insumos documentales, supuestos y preguntas abiertas.
- Usa `ai/prompts/crear-proyecto-real-desde-template.md` cuando ya existan datos minimos claros para instanciar un proyecto.
- Usa `ai/runbooks/crear-proyecto-real-con-agente.md` cuando necesites orquestacion IA end-to-end con preguntas, gates y validaciones.

<a id="anatomia-minima-recomendada"></a>
## Anatomia minima recomendada
- `Agent`: objetivo, usalo cuando, no lo uses cuando, entradas minimas, salidas, rutas y verificacion.
- `Prompt`: objetivo, usalo cuando, no lo uses cuando, entradas, salida, rutas, verificacion y pedido base.
- `Skill`: objetivo, aplicala cuando, no la apliques cuando, entradas minimas, flujo recomendado, anti-rationalizations, red flags, verification evidence y referencias.
- `Command`: objetivo, fases donde aplica, required inputs, process, anti-rationalizations, red flags y verification evidence.
- `Quality gate`: objetivo, evidencia minima, bloqueantes tipicos, rutas relacionadas y decision esperada.
- `Reference`: checklist corta o criterio reusable para cargar segun la tarea.

<a id="regla-de-instanciacion-para-proyectos-reales"></a>
## Regla de instanciacion para proyectos reales
- `ai/agents/`, `ai/prompts/`, `ai/skills/`, `ai/commands/`, `ai/quality-gates/`, `ai/references/` y `ai/external-agent-skills.md` son artefactos oficiales de la capa de IA del proyecto adoptado.
- `ai/ejemplos/` solo sirve como referencia y no debe copiarse como contenido final del proyecto.
- Si el proyecto adopta otro dominio, stack o estructura, los artefactos de IA deben actualizar sus entradas, restricciones, rutas y salidas esperadas.
- Ningun artefacto oficial de IA debe conservar referencias residuales al caso canonico si ya no aplican.

<a id="restricciones"></a>
## Restricciones
- La IA no debe inventar decisiones tecnicas sin ADR o justificacion.
- La IA debe respetar nomenclatura, fases y rutas canonicas del repositorio.
- La IA debe decidir modo de trabajo en `90.13-modos-de-trabajo.md` cuando la entrada sea ambigua.
- Si la IA trabaja en `modo exploratorio`, debe cerrar con una recomendacion y siguiente paso explicito hacia artefactos formales.
- Si la IA trabaja en `modo estructurado`, debe producir o actualizar directamente artefactos oficiales del repositorio.
- Si una feature tiene UI, no debe pasar a SDD final ni construccion sin prototipo validado o excepcion aprobada.
- Un prototipo HTML5 no es frontend productivo y un prompt Penpot no es prototipo aprobado.
- Los prompts no sustituyen documentos oficiales; deben producir o actualizar artefactos del repositorio.
- Los skills deben reforzar consistencia, no reemplazar analisis ni validacion humana.
- Los ejemplos canonicos no deben copiarse como capa oficial de IA de un proyecto adoptado.

<a id="formato-de-salida-esperado"></a>
## Formato de salida esperado
- Para planificacion: roadmap, estimacion, riesgos y roles.
- Para arquitectura: markdown, ADR y C4.
- Para UX/SPDD: Product Design, SPDD inicial, prototipo HTML5/Penpot, `prototype.md`, `prototype-validation.md` y gates declarados.
- Para SDD: `spec funcional`, `spec tecnica` y `spec de tareas`.
- Para construccion: componentes afectados, pruebas y trazabilidad a specs.
- Para QA y ops: casos, evidencias, checks de deploy y metricas.

## General
- Usa nombres cortos, consistentes y orientados a negocio o capacidad.
- Evita sinonimos para fases, roles, features y artefactos.
- Mantiene una sola nomenclatura para `SDD`, ADR, QA, deploy y operacion.

## Fases
- Usa fases `0` a `8` en todo el repositorio.
- No inventes nombres paralelos para una misma fase.

## Features
- Usa formato `NNN-nombre-feature` en `specs/`, `qa/` y `ops/` cuando aplique.

## ADR
- Nombre de archivo: `ADR-NNN-nombre-corto-kebab.md` dentro de `docs/fase-3-arquitectura/adr/`.
- Titulo H1: `ADR-NNN - Nombre corto legible` (numero y titulo separados por ` - `).
- Secciones obligatorias: `Decision`, `Contexto`, `Opciones consideradas`, `Consecuencias`.
- La plantilla base vive en `plantillas/fase-3-arquitectura/adr.md` y el ejemplo canonico en `ejemplos/fase-3-arquitectura/adr-ejemplo.md`.

## Roles
- Usa los nombres oficiales definidos en `docs/fase-0-iniciacion/00.04-roles-y-responsabilidades.md`.

## Acentuacion y ortografia
- Usa la forma ASCII en titulos, encabezados (H1, H2), breadcrumbs y etiquetas `nav-guided`. Ejemplos correctos: `Indice docs`, `Analisis`, `Operacion`, `Construccion`, `Adopcion`, `Produccion`.
- En prosa de cuerpo se admiten acentos cuando aportan claridad, siempre que el termino principal aparezca escrito igual en el resto del documento y en el indice.
- No mezcles variantes acentuadas y sin acentuar del mismo termino dentro del mismo documento.

## Codificacion de archivos
- Todos los archivos markdown deben guardarse como `UTF-8 sin BOM`.
- No se admite marca de orden de byte (`BOM`, bytes `EF BB BF`) al inicio del archivo.
- Preferir finales de linea `LF`. Evitar mezclar `CRLF` y `LF` dentro de un mismo archivo.

## Estructura minima por documento en `docs/`
- Empieza con un `# Titulo` en linea 1.
- Deja una linea en blanco y agrega la fila de breadcrumbs con enlaces a README principal, Indice docs y, si corresponde, Volver a la fase.
- Incluye el bloque `<!-- nav-guided:start -->`/`<!-- nav-guided:end -->` con `Anterior` y `Siguiente` como enlaces navegables.
- Si un documento cierra el recorrido guiado, `Siguiente` debe enlazar al indice de documentacion o al inicio del recorrido para cerrar el ciclo.

## Recorrido `nav-guided`
- Todos los documentos bajo `docs/` participan del recorrido guiado unico.
- `Anterior` y `Siguiente` siempre deben ser enlaces markdown navegables, nunca texto plano.
- Para una fase con varios documentos, el recorrido debe pasar por todos en orden antes de saltar a la siguiente fase.
- Para una fase con un solo documento guia, se admite que `Siguiente` salte a la siguiente fase (README) sin documentos intermedios. Si se agrega un documento nuevo, el recorrido debe extenderse primero dentro de la misma fase.
- El README de fase actua como primer nodo de la fase y debe apuntar como `Siguiente` al primer documento de la fase, o a la siguiente fase si no existe ningun documento intermedio.

<a id="regla-general"></a>
## Regla general
Cada fase debe dejar entregables minimos, ruta canonica y criterio de salida. Los documentos transversales como roadmap, estimacion y roles se crean en fases tempranas y se actualizan durante todo el proyecto.

<a id="regla-de-instanciacion"></a>
## Regla de instanciacion
La ruta canonica de una fase define el entregable minimo, pero no obliga a que todo el detalle viva en un solo archivo. En proyectos reales, cada fase puede descomponerse en artefactos propios dentro de su carpeta, siempre que el `README.md` de la fase enlace esos artefactos oficiales y que el contenido ya no sea texto de plantilla. Para este criterio, revisa `90.14-instanciacion-fases-proyectos-reales.md`.

<a id="vista-ejecutiva"></a>
## Vista ejecutiva
| Fase | Objetivo ejecutivo | Entregables clave | Resultado esperado |
|---|---|---|---|
| 0 - Iniciacion | Alinear direccion del proyecto | vision, alcance, roadmap, estimacion, roles | proyecto encuadrado |
| 1 - Analisis y requerimientos | Definir que se va a construir | actores, modulos, RF, RNF, reglas, backlog, matriz de huecos | requerimientos aprobados |
| 2 - UX/UI | Validar producto y experiencia antes de construir | Product Design, SPDD, prototipo HTML5/Penpot validado | algo mostrable aprobado |
| 3 - Arquitectura | Definir como se construira y operara | arquitectura, tecnologia, ADR, C4, despliegue | arquitectura aprobada |
| 4 - Spec-Driven Development (SDD) | Bajar SPDD aprobado a features ejecutables front/back | spec funcional, spec tecnica, api-contract, tareas | desarrollo preparado |
| 5 - Construccion | Implementar el incremento | codigo, pruebas, documentacion tecnica y, si aplica frontend, SPDD desde spec + prototipo | incremento construido |
| 6 - QA | Asegurar calidad antes de liberar | plan de pruebas, evidencias, defectos corregidos | calidad aprobada |
| 7 - Deploy | Preparar y ejecutar la salida | pipeline, runbook, rollback, release | produccion lista |
| 8 - Operacion y mejora continua | Medir, operar y evolucionar | monitoreo, metricas, backlog evolutivo | operacion estable |

<a id="notas-de-estandar"></a>
## Notas de estandar
- `docs/fase-0-iniciacion/00.02-roadmap.md`, `docs/fase-0-iniciacion/00.03-estimacion-tiempo-costo.md` y `docs/fase-0-iniciacion/00.04-roles-y-responsabilidades.md` son artefactos transversales.
- La plantilla define minimos; el proyecto real puede descomponer cada fase en artefactos oficiales propios dentro de su carpeta.
- Las rutas fuera de `docs/` son convenciones sugeridas para que la plantilla escale a proyectos reales.
- Si el proyecto usa otra estructura, documenta el cambio en `docs/fase-3-arquitectura/03.00-arquitectura.md` o en un ADR.

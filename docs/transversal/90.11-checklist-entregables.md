## Vista ejecutiva de gates
| Gate | Que valida | Evidencia minima | Decision esperada |
|---|---|---|---|
| Gate 0-1 | El proyecto tiene direccion, alcance y base de planificacion | vision, requerimientos iniciales, roadmap, estimacion, roles | continuar a definicion detallada |
| Gate UX Ready | Fase 2 tiene producto y experiencia validables | objetivo, actor, journeys, estados, prototipo, componentes y mapping | continuar a arquitectura, SDD o mock |
| Gate Prototype Ready | Feature visual tiene prototipo revisable | HTML5 navegable, Penpot, wireframe aprobado o mockup validado con ruta/link/evidencia | continuar a validacion SPDD |
| Gate SPDD Approved | Feature visual tiene prototipo validado antes de SDD final | spec funcional inicial, flujo UX, prototipo, estados, componentes, validaciones, permisos y observaciones aceptadas | continuar a SDD front/back |
| Gate 2-3 | La solucion objetivo esta validada en experiencia y arquitectura | UX validado, arquitectura, tecnologia, ADR, C4, despliegue | continuar a SDD y construccion |
| Gate Frontend SPDD Ready | Frontend construido desde spec + prototipo conserva consistencia | spec, prototipo, mapping, TDD, evidencia browser/test y review | continuar a QA o integracion |
| Gate 4-6 | La ejecucion esta lista y la calidad esta controlada | specs completas, codigo, pruebas, QA, defectos gestionados | continuar a liberacion |
| Gate 7-8 | La salida a produccion y la operacion estan preparadas | pipeline, runbook, rollback, monitoreo, metricas, backlog evolutivo | operar y mejorar |

La version AI-first de estos gates vive en `../../ai/quality-gates/`. Si se ajusta un gate en este documento, debe actualizarse tambien su par en `ai/quality-gates/` para evitar drift.

<a id="consistencia-base"></a>
## Consistencia base
- [ ] La vision, el roadmap, la estimacion y los roles base existen.
- [ ] La nomenclatura de fases y roles es consistente en todos los documentos.
- [ ] Los entregables usan rutas canonicas o documentan su excepcion.
- [ ] Cada fase relevante enlaza sus artefactos oficiales desde su `README.md`.

<a id="vista-operativa-por-gate"></a>
## Vista operativa por gate

<a id="fase-0-1"></a>
## Fase 0-1
- [ ] Vision aprobada.
- [ ] Requerimientos aprobados.
- [ ] El analisis incluye actores, modulos, `RF`, `RNF`, reglas, integraciones, casos de uso y backlog inicial.
- [ ] La matriz de huecos de fase 1 existe y muestra reglas sin RF, RNF debiles, integraciones sin dueno, datos sensibles, permisos, backlog, UX/SPDD y supuestos pendientes.
- [ ] Los huecos abiertos tienen responsable, impacto y decision requerida, o quedaron diferidos con razon.
- [ ] El documento de fase 1 ya esta adaptado al dominio real y no conserva texto instructivo de la plantilla.
- [ ] Roadmap inicial definido.
- [ ] Estimacion aprobada.
- [ ] Equipo asignado.

<a id="fase-2-3"></a>
## Fase 2-3
- [ ] UX validado.
- [ ] `/ux` aplicado cuando hubo journeys, pantallas o prototipo.
- [ ] `/prototype` aplicado en modo HTML5 o Penpot cuando la feature visual necesita algo navegable/revisable.
- [ ] Spec-Driven Product Design documentado si la feature nace de RF/HU o notas de negocio.
- [ ] `gate-prototype-ready` aplicado o bloqueado con razon antes de pedir aprobacion SPDD.
- [ ] `gate-spdd-approved` aplicado para features visuales antes de cerrar SDD.
- [ ] Flujo UX `Markdown -> IA -> HTML5 Prototype -> Penpot si aplica` definido o descartado explicitamente.
- [ ] Arquitectura definida.
- [ ] Decisiones tecnologicas justificadas.
- [ ] ADR iniciales registradas.
- [ ] C4 generado y actualizado.
- [ ] Plan de despliegue definido.

<a id="fase-4-6"></a>
## Fase 4-6
- [ ] Specs listas.
- [ ] Features visuales tienen Product Design, SPDD, prototype/prototype-html5 o link Penpot, prototype validation y traceability en `specs/<feature>/`.
- [ ] Backend afectado por UX tiene `api-contract.md`.
- [ ] Checklist SDD completado.
- [ ] Codigo y pruebas implementados.
- [ ] Si hay frontend, SPDD valido spec + prototipo + mapping antes de construir.
- [ ] Si hay frontend, `gate-frontend-spdd-ready` aprobado o con observaciones aceptadas.
- [ ] Plan de pruebas definido.
- [ ] Evidencias QA disponibles.
- [ ] Defectos criticos resueltos o aceptados explicitamente.

<a id="fase-7-8"></a>
## Fase 7-8
- [ ] Pipeline operativo.
- [ ] Release aprobado y comunicado.
- [ ] Runbook disponible.
- [ ] Rollback definido.
- [ ] Monitoreo activo.
- [ ] Metricas operativas definidas.
- [ ] Backlog evolutivo actualizado.

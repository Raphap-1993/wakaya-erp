<a id="objetivo"></a>
## Objetivo
Mostrar de forma simple como usar `agent`, `prompt` y `skill` en cada fase del repositorio, y como eso se conecta con trazabilidad, consistencia y rutas canonicas.

<a id="entrada-desde-idea-bruta"></a>
## Entrada desde idea bruta
Si el proyecto todavia no tiene documentacion formal y solo existe una idea, oportunidad o problema operativo, primero decide el modo en [90.13-modos-de-trabajo.md](90.13-modos-de-trabajo.md). Este mapa muestra como aterrizar ese modo en fases y artefactos.

```text
Idea bruta
  ->
Modo intake desde fuente bruta
  ->
IA ordena contexto, actores, supuestos y preguntas abiertas
  ->
Primeros borradores de vision/requerimientos
  ->
Decision: seguir discovery, instanciar template o usar agente guiado
```

La ruta oficial para ese caso es [../fase-0-iniciacion/00.10-idea-a-documentacion-inicial-con-ia.md](../fase-0-iniciacion/00.10-idea-a-documentacion-inicial-con-ia.md).

Para ordenar notas, entrevistas, correos o texto libre, usa [../../ai/prompts/arranque-desde-fuente-bruta.md](../../ai/prompts/arranque-desde-fuente-bruta.md). Ese prompt es intake/documentacion inicial; no instancia repositorios ni ejecuta `create-project`.

Para instanciar un proyecto cuando ya existen datos minimos, usa [../../ai/prompts/crear-proyecto-real-desde-template.md](../../ai/prompts/crear-proyecto-real-desde-template.md).

Para guia end-to-end con preguntas, bootstrap, gates y validaciones, usa [../../ai/runbooks/crear-proyecto-real-con-agente.md](../../ai/runbooks/crear-proyecto-real-con-agente.md).

Para transformar solo una idea en documentacion inicial (fases 0-3), usa [../../ai/prompts/transformar-idea-a-documentacion-inicial.md](../../ai/prompts/transformar-idea-a-documentacion-inicial.md).

<a id="como-leer-este-mapa"></a>
## Como leer este mapa
- `Agent`: rol especializado que lidera una salida.
- `Prompt`: instruccion concreta para pedir un entregable.
- `Skill`: capacidad reusable para producir o revisar trabajo.
- `Command`: entrada lifecycle para orquestar trabajo (`/document`, `/plan`, `/ux`, `/prototype`, `/spec`, `/build`, `/test`, `/review`, `/ship`).
- `Gate AI`: criterio AI-first de readiness para cerrar o avanzar bloques de fases.
- `Salida`: artefacto esperado que debe quedar en el repositorio.
- `Trazabilidad`: entrada que alimenta la salida.

<a id="agente-interno-del-framework"></a>
## Agente interno del framework
Cuando una solicitud cruza varias fases o no esta claro que artefacto usar, inicia con `../../ai/agents/enterprise-ai-framework-agent.md`.

Este agente enruta la intencion hacia command, agent, prompt, skill, reference y gate. Tambien puede apoyarse en memoria local descrita en `../../ai/memory/README.md`, manteniendo siempre los archivos del repositorio como fuente oficial.

Si la intencion es crear un proyecto real desde un archivo bruto, primero usa `../../ai/prompts/arranque-desde-fuente-bruta.md` para convertirlo en insumos claros. Luego decide en `90.13-modos-de-trabajo.md` si corresponde bootstrap o agente guiado.

Si la intencion es crear un proyecto real desde la plantilla con datos minimos claros, usa `../../ai/prompts/crear-proyecto-real-desde-template.md`.

Si la intencion es que la IA guie el proceso paso a paso, haga preguntas, aplique gates y cierre con validaciones, usa `../../ai/runbooks/crear-proyecto-real-con-agente.md`.

Si la intencion es convertir una idea, intake, requerimiento o decision en artefactos formales, usa primero `../../ai/commands/document-command.md`. El intake no tiene comando separado: funciona como fuente para `/document`.

<a id="flujo-de-delivery-ia-para-proveedores"></a>
## Flujo de delivery IA para proveedores
Cuando el trabajo pasa a una feature construible, usa [90.33-flujo-delivery-ia-proveedores.md](90.33-flujo-delivery-ia-proveedores.md). Este flujo hace explicito lo que un proveedor IA debe ejecutar:

```text
Product Design
  -> SPDD inicial
  -> /prototype HTML5 o Penpot
  -> gate-prototype-ready
  -> gate-spdd-approved
  -> /spec
  -> spec aprobada
  -> writing-plans
  -> worktree/rama
  -> TDD red-green-refactor
  -> code review
  -> QA/evidencia
  -> cierre branch/PR
```

Regla practica: si un proveedor IA no puede identificar tarea, archivos permitidos, prueba red/green, comando de verificacion y evidencia, la feature aun no esta lista para `/build`.

Regla frontend: si la feature modifica pantallas, el proveedor IA debe identificar tambien prototipo HTML5/Penpot, wireframe, sistema de componentes, mapping Penpot -> Angular, `gate-prototype-ready` y `gate-spdd-approved`. Si no existen, vuelve a `/ux` y `/prototype` antes de cerrar SDD o construir.

<a id="reglas-de-trazabilidad-y-consistencia"></a>
## Reglas de trazabilidad y consistencia
- Ningun `agent`, `prompt` o `skill` debe producir texto huerfano; siempre debe terminar en una ruta canonica del repositorio.
- Toda salida de IA debe indicar que entrada uso: vision, RF, HU, spec o ADR.
- `Prompt` no reemplaza documento oficial; lo ayuda a producir o actualizar.
- `Skill` no reemplaza criterio humano; refuerza consistencia de ejecucion.
- Si una salida de IA cambia arquitectura o tecnologia, debe reflejarse en ADR o en documentos de fase 3.
- `ai/ejemplos/` sirve como referencia; la capa oficial de IA del proyecto debe vivir en `ai/agents/`, `ai/prompts/` y `ai/skills/` ya adaptados al contexto real.

## Ruta corta recomendada
1. Lee `90.00-estandar-ia.md`.
2. Revisa `../../ai/README.md`.
3. Revisa `../../ai/commands/README.md`, `../../ai/quality-gates/README.md`, `../../ai/agents/README.md`, `../../ai/prompts/README.md` y `../../ai/skills/README.md`.
4. Usa `../../ai/ejemplos/README.md` para ver ejemplos del caso canonico.
5. Adapta o crea los artefactos oficiales de `ai/` segun el proyecto real, sin copiar el caso canonico como estado final.
6. Vuelve a este mapa cuando necesites decidir que `command`, `agent`, `prompt`, `skill` o `gate` corresponde a una fase.

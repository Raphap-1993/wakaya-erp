## Objetivo
Explicar que modo de trabajo usar segun el nivel de claridad, el tipo de entrada y el siguiente resultado esperado.

Este documento es conceptual y transversal: decide el camino. Los prompts, runbooks y commands ejecutan el camino.

## Principio general
- El modo estructurado es el modo recomendado por defecto.
- El modo exploratorio sirve para reducir incertidumbre, no para reemplazar entregables formales.
- El modo intake convierte fuente bruta en insumos claros.
- El modo bootstrap instancia el template cuando ya hay datos minimos.
- El modo agente guiado orquesta decisiones, prompts, commands, gates y validaciones.
- Todo modo debe terminar en rutas canonicas, evidencia o preguntas abiertas trazables.

## Mapa rapido de decision
| Situacion | Modo recomendado | Runbook / Command |
|---|---|---|
| Tengo una idea vaga o varias opciones | Exploratorio | `/document`, `/plan` |
| Tengo notas, entrevistas, audios, correos o texto libre | Intake desde fuente bruta | `ai/prompts/arranque-desde-fuente-bruta.md` / `/document` |
| Ya tengo nombre, dominio, stack, ruta destino y datos minimos | Bootstrap desde template | `ai/prompts/crear-proyecto-real-desde-template.md` |
| Quiero que la IA me guie paso a paso | Orquestacion con agente | `ai/runbooks/crear-proyecto-real-con-agente.md` |
| Tengo una feature visual | Product Design + SPDD + prototipo | `/ux` + `/prototype` |
| Tengo prototipo validado | SDD | `/spec` |
| Tengo specs y tareas listas | Construccion | `/build` |
| Quiero validar calidad o riesgos | QA / Review | `/test` + `/review` |
| Quiero liberar o preparar salida | Ship | `/ship` |
| Quiero revisar operacion | Operacion | `/review` + `ai/skills/operations-review.skill.md` |

## Modo exploratorio
### Para que sirve
- discovery rapido cuando el problema todavia es ambiguo,
- evaluacion de opciones tecnicas o funcionales,
- spikes, pruebas de concepto o prototipos descartables,
- conversaciones asistidas con IA para ordenar ideas antes de formalizar.

### Salidas esperadas
- supuestos explicitos,
- opciones evaluadas y recomendacion,
- riesgos visibles,
- preguntas abiertas,
- propuesta de aterrizaje a `vision`, `requerimientos`, `ADR` o `specs`.

### Limites
- no debe cerrar arquitectura critica sin ADR,
- no debe sustituir specs, pruebas o gates,
- no debe iniciar construccion productiva.

## Modo intake desde fuente bruta
### Para que sirve
Convertir informacion desordenada en insumos documentales iniciales.

### Entradas
- notas,
- audios transcritos,
- entrevistas,
- correos,
- actas,
- texto libre,
- backlog preliminar.

### Salidas esperadas
- vision inicial,
- RF/RNF preliminares,
- actores,
- restricciones,
- supuestos,
- preguntas abiertas,
- riesgos,
- features candidatas,
- recomendacion de siguiente modo.

### Referencia operativa
- `ai/prompts/arranque-desde-fuente-bruta.md`
- `ai/commands/document-command.md`

## Modo bootstrap desde template
### Para que sirve
Instanciar esta plantilla como proyecto real cuando ya existen datos minimos.

### Entradas
- nombre del proyecto,
- dominio,
- stack,
- ruta destino,
- config o datos para generarla,
- fuente de negocio resumida.

### Salidas esperadas
- proyecto real creado,
- README/AGENTS adaptados,
- docs iniciales,
- specs reales,
- contracts si aplican,
- capa IA completa,
- gates declarados,
- provider orientation pack,
- validaciones ejecutadas.

### Referencia operativa
- `ai/prompts/crear-proyecto-real-desde-template.md`

## Modo orquestacion con agente
### Para que sirve
Guiar el proceso completo con IA cuando el usuario quiere acompanamiento, preguntas, gates y validaciones.

### Responsabilidad
El agente decide si debe:
- hacer intake desde fuente bruta,
- instanciar desde template,
- pasar a Product Design/SPDD,
- preparar SDD,
- bloquear construccion por gates pendientes.

### Referencia operativa
- `ai/runbooks/crear-proyecto-real-con-agente.md`
- `ai/agents/enterprise-ai-framework-agent.md`

## Modo Product Design + SPDD + prototipo
### Para que sirve
Validar experiencia antes de SDD y construccion.

### Flujo
```text
/ux
  -> Product Design
  -> SPDD inicial
  -> /prototype HTML5 o Penpot
  -> gate-prototype-ready
  -> validacion humana
  -> gate-spdd-approved
```

### Regla
No pasar a SDD final ni construccion frontend productiva sin prototipo validado o excepcion aprobada.

## Modo SDD
### Para que sirve
Convertir lo aprobado en specs tecnicas front/back, contratos, tareas y trazabilidad.

### Referencia operativa
- `docs/fase-4-sdd/04.00-spec-driven-development.md`
- `ai/commands/spec-command.md`

## Modo construccion
### Para que sirve
Implementar tareas pequenas con pruebas, trazabilidad y evidencia.

### Reglas
- No iniciar `/build` sin specs y tareas.
- No iniciar frontend productivo de feature visual sin `gate-spdd-approved`.
- Usar TDD cuando aplique.

## Modo QA / Review / Ship / Operacion
- QA: `/test`
- Revision: `/review`
- Release: `/ship`
- Operacion: `/review` + `ai/skills/operations-review.skill.md`

No existe `/operate` como command principal en esta version; no debe usarse salvo que se agregue formalmente.

## Transicion entre modos
1. Si hay ambiguedad, empieza explorando o haciendo intake.
2. Si hay datos minimos, instancia desde template.
3. Si el usuario quiere guia, usa el agente interno como orquestador.
4. Si hay UI, pasa por Product Design, SPDD y prototipo.
5. Si hay prototipo validado, pasa a SDD.
6. Si hay specs y tareas, pasa a construccion.
7. Si hay codigo, pasa a QA, review, ship y operacion.

## Reglas de consistencia
- No llames exploratorio a trabajo desordenado o sin salida.
- No uses intake para crear repo completo.
- No uses bootstrap para hacer discovery profundo.
- No uses el agente guiado sin decidir primero modo de trabajo.
- No uses exploracion para omitir ADR, SDD, pruebas o checklist cuando ya son necesarios.
- Si el trabajo involucra IA, cruza este documento con `90.00-estandar-ia.md` y `90.12-mapa-ia-por-fase.md`.
- Si el trabajo ya va a implementacion, la referencia principal sigue siendo `docs/fase-4-sdd/04.00-spec-driven-development.md`.

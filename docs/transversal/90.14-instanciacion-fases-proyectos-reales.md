<a id="objetivo"></a>
## Objetivo
Evitar que la documentacion final del proyecto quede redactada como si todavia fuera una plantilla. Este documento explica como convertir `plantillas/` en artefactos reales por fase.

<a id="regla-principal"></a>
## Regla principal
- `plantillas/` contiene moldes reutilizables para trabajar.
- `ejemplos/` contiene referencias de profundidad y forma.
- `docs/fase-x/`, `specs/`, `qa/`, `ops/`, `ci/` y `releases/` contienen artefactos reales del proyecto.
- Ningun entregable oficial debe quedarse en texto instructivo tipo "Describe", "Lista", "Completa" o "Usa esta seccion para".

<a id="diferencia-entre-carpeta-de-trabajo-y-entregable-real"></a>
## Diferencia entre carpeta de trabajo y entregable real
| Tipo | Para que sirve | Que no debe pasar |
|---|---|---|
| `plantillas/` | levantar informacion, discovery, refinamiento | no se publica como entregable final del proyecto |
| `ejemplos/` | entender estructura y nivel de detalle esperado | no reemplaza el analisis real del dominio |
| `docs/fase-x/` | documentar el estado real del proyecto por fase | no debe sonar a instructivo ni a boilerplate |

<a id="patron-de-adopcion-recomendado"></a>
## Patron de adopcion recomendado
1. Usa una o varias plantillas para capturar insumos.
2. Consolida o descompone esos insumos en artefactos reales de la fase.
3. Deja en el `README.md` de la fase el indice de los artefactos oficiales.
4. Mantiene un documento resumen de fase cuando ayude a navegabilidad, aunque el detalle viva en subcarpetas.

<a id="ejemplo-detallado-de-fase-1"></a>
## Ejemplo detallado de fase 1
### Caso incorrecto
`docs/fase-1-analisis-requerimientos/01.00-analisis-requerimientos.md` termina con frases como:
- "Agrupa la solucion en capacidades o modulos funcionales comprensibles por el negocio."
- "Resume los flujos mas importantes que luego pueden aterrizarse en UX y SDD."

Eso sigue siendo plantilla, no documentacion real.

### Caso correcto y simple
- Un solo archivo consolidado:
  - `docs/fase-1-analisis-requerimientos/01.00-analisis-requerimientos.md`
- Contenido ya instanciado:
  - modulos reales,
  - `RF` y `RNF` reales,
  - casos de uso reales,
  - backlog inicial real.

### Caso correcto y escalable
```text
docs/
  fase-1-analisis-requerimientos/
    README.md
    01.00-analisis-requerimientos.md
    modulos/
      modulo-fuentes.md
      modulo-procesos.md
    requerimientos-no-funcionales/
      rnf-base.md
    casos-de-uso/
      UC-01-configurar-fuente.md
      UC-02-configurar-reader.md
    historias-usuario/
      HU-01-crear-fuente.md
      HU-02-ejecutar-proceso.md
    riesgos/
      riesgos-fase-1.md
```

En ese patron:
- `01.00-analisis-requerimientos.md` resume y consolida la fase.
- `modulos/`, `requerimientos-no-funcionales/`, `casos-de-uso/`, `historias-usuario/` y `riesgos/` guardan el detalle real del proyecto.
- `plantillas/fase-1-analisis-requerimientos/` queda solo como apoyo de trabajo.

<a id="ejemplos-por-fase"></a>
## Ejemplos por fase
| Fase | Artefacto minimo | Descomposicion real opcional |
|---|---|---|
| 0 - Iniciacion | vision, roadmap, estimacion, roles | `stakeholders/`, `riesgos/`, `supuestos/` |
| 1 - Analisis y requerimientos | analisis consolidado | `modulos/`, `requerimientos-no-funcionales/`, `casos-de-uso/`, `historias-usuario/`, `reglas/`, `riesgos/` |
| 2 - UX/UI | Product Design, SPDD inicial y prototipo validable | `journeys/`, `wireframes/`, `pantallas/`, `validaciones/`, `prototype-html5/`, links/export Penpot |
| 3 - Arquitectura | arquitectura y decisiones | `adr/`, `diagramas/`, `integraciones/`, `despliegue/` |
| 4 - Spec-Driven Development (SDD) | carpeta por feature | `spec-funcional.md`, `spec-tecnica.md`, `spec-tareas.md` dentro de cada feature |
| 5 - Construccion | codigo y documentacion tecnica | modulos de codigo, notas de implementacion dentro de `docs/fase-5-construccion/` |
| 6 - QA | plan, evidencias, defectos | `casos/`, `evidencias/`, `matrices/`, `defectos/` |
| 7 - Deploy | pipeline, runbook, rollback | `checklists/`, `release-notes/`, `aprobaciones/` |
| 8 - Operacion | monitoreo, metricas, backlog evolutivo | `postmortems/`, `dashboards/`, `incidentes/`, `mejoras/` |

<a id="instanciacion-de-la-capa-ia"></a>
## Instanciacion de la capa IA
En un proyecto real, la capa IA tambien se instancia. No basta con adaptar `agents`, `prompts` y `skills`:

- `ai/commands/` debe reflejar como el equipo usa `/plan`, `/document`, `/ux`, `/prototype`, `/spec`, `/build`, `/test`, `/review` y `/ship`.
- `ai/quality-gates/` debe quedar alineado con los gates documentales de `90.11-checklist-entregables.md`, incluyendo `gate-prototype-ready` y `gate-spdd-approved` cuando hay UI.
- `ai/references/` debe contener checklists cortas del dominio real, no solo criterios genericos.
- `ai/external-agent-skills.md` debe mantenerse como mapeo recomendado, no como dependencia obligatoria.

Para features visuales, la instancia real debe dejar claro:
- `/ux` produce Product Design y SPDD inicial.
- `/prototype` produce o actualiza prototipo HTML5/Penpot revisable.
- `gate-prototype-ready` valida que existe evidencia revisable.
- `gate-spdd-approved` solo se aprueba con validacion humana trazable.

<a id="reglas-de-consistencia"></a>
## Reglas de consistencia
- Cada fase puede tener un documento consolidado, subcarpetas reales o ambos.
- Si una fase se descompone, el `README.md` de la fase debe enlazar sus artefactos oficiales.
- Las subcarpetas de una fase deben contener contenido del proyecto real, no copias literales de `plantillas/`.
- Si el proyecto adopta otra estructura, documenta la equivalencia en un documento transversal o en arquitectura.

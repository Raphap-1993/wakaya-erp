<a id="objetivo"></a>
## Objetivo
Hacer verificable y auditable la seguridad del codigo, las dependencias y los artefactos producidos por la plantilla. Prevenir introduccion de secretos, vulnerabilidades conocidas y artefactos no firmados en entornos superiores.

<a id="principios"></a>
## Principios
- Seguridad como codigo: todas las politicas viven versionadas en el repo, no en tickets.
- Defensa en profundidad: multiples capas redundantes (pre-commit, CI, CD, runtime).
- Fail closed en produccion: sin firma valida ni provenance, no hay despliegue.
- Observabilidad de la cadena: cada artefacto publicado es trazable a su commit, build y firmante.

<a id="capas-de-control"></a>
## Capas de control

| Capa          | Herramientas                                                            | Momento         |
|---------------|-------------------------------------------------------------------------|-----------------|
| Pre-commit    | gitleaks, detect-secrets, format/lint                                   | Dev local       |
| PR / push     | gitleaks, semgrep, dependency-review, tfsec, checkov, grype             | CI              |
| Build         | SBOM (syft), trivy de imagen, cosign sign, SLSA provenance               | CI release      |
| Deploy        | cosign verify, admission policy (Kyverno/OPA), External Secrets          | CD              |
| Runtime       | logs de auditoria, OpenTelemetry, detecciones en SIEM                    | Operacion       |

<a id="cadena-de-suministro-slsa"></a>
## Cadena de suministro (SLSA)
El objetivo declarado de la plantilla es `SLSA Build Level 3`. Implica:

- Build reproducible desde un entorno CI confiable y aislado por job.
- Provenance firmada que referencia el commit, workflow, runner y ejecucion especifica.
- Artefactos firmados con `cosign` keyless contra Fulcio/Sigstore (identidad OIDC del runner).
- Registros de firma y provenance almacenados en el mismo registry (modo `push-to-registry`).

La verificacion en el deploy es obligatoria:

```
cosign verify \
  --certificate-identity-regexp='^https://github.com/<org>/<repo>/' \
  --certificate-oidc-issuer='https://token.actions.githubusercontent.com' \
  ghcr.io/<org>/<repo>/<image>@<digest>

gh attestation verify oci://ghcr.io/<org>/<repo>/<image>@<digest> \
  --repo <org>/<repo>
```

Un admission controller en Kubernetes (Kyverno o Sigstore Policy Controller) debe rechazar imagenes sin firma ni atestacion valida.

<a id="dependencias"></a>
## Dependencias
- `dependabot` levanta PRs semanales por ecosistema (npm, maven, gradle, docker, github-actions).
- `dependency-review-action` bloquea PRs que introducen dependencias con severidad `high` o superior.
- `grype` sobre el SBOM detecta CVEs conocidos en cada push.
- Politica de upgrade: parches de seguridad se integran en <= 5 dias habiles; mayores se evaluan con ADR.
- Se mantiene un `vulnerability-exceptions.yaml` versionado cuando una vulnerabilidad no tiene fix inmediato, con SLA de revision trimestral.

<a id="respuesta-a-incidentes"></a>
## Respuesta a incidentes
- Canal inicial: `security@ejemplo.dev`.
- Playbook de respuesta: contener -> evaluar -> comunicar -> remediar -> postmortem.
- Comunicacion con clientes vigente dentro de ventana regulatoria (72h GDPR, por ejemplo).
- Postmortem publico sin culpas con blameless culture, segun `docs/transversal/90.18-definiciones-operativas.md`.

<a id="checklist"></a>
## Checklist
- [ ] `gitleaks`, `semgrep`, `tfsec`, `checkov`, `grype`, `trivy` corren en cada PR.
- [ ] `dependabot` activo para todos los ecosistemas con auto-merge de patch seguro cuando aplica.
- [ ] Las imagenes publicadas estan firmadas con `cosign` keyless.
- [ ] Cada imagen tiene atestacion SLSA build provenance.
- [ ] El deploy valida firma y rechaza imagenes sin atestacion.
- [ ] Existe politica escrita para excepciones y rotacion de dependencias.
- [ ] Incidentes de seguridad tienen playbook publicado.

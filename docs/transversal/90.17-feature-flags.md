<a id="objetivo"></a>
## Objetivo
Separar el despliegue del codigo del lanzamiento hacia el usuario. Permitir experimentos, rollouts graduales y killswitches sin necesidad de un nuevo despliegue.

<a id="tipos-de-flag"></a>
## Tipos de flag
- Release: activa una feature nueva en produccion. Se elimina cuando la adopcion es total.
- Experimento: asigna variantes a cohortes para medir efecto en metricas de producto.
- Permission: habilita capacidades por rol, plan o usuario.
- Ops / killswitch: apaga rapidamente una feature en incidentes.

<a id="convenciones"></a>
## Convenciones
- Nombre: `area.accion.flag` en `snake_case` (ej. `bandeja.virtual_scroll.enabled`).
- Valor por defecto seguro: `false` para release; la feature solo aparece si alguien la activa.
- Todo flag vive en el servicio central de flags, no hardcodeado.
- Cada flag tiene duefio, fecha de creacion y fecha objetivo de retiro.

<a id="ciclo-de-vida"></a>
## Ciclo de vida
1. Crear flag con valor off en todos los entornos.
2. Desplegar codigo protegido por el flag.
3. Activar progresivamente (dev -> staging -> 1% prod -> 10% -> 50% -> 100%).
4. Cuando el flag alcanza 100% estable por el plazo acordado, eliminar codigo condicional en un PR de limpieza.
5. Auditar trimestralmente flags huerfanos.

<a id="integracion-tecnica"></a>
## Integracion tecnica
- Servicio recomendado para MVP: flags en tabla propia + cache en memoria con TTL corto.
- Servicio recomendado para escala: LaunchDarkly, Unleash, ConfigCat o alternativa equivalente.
- Todo flag se evalua con el `userId` o un contexto estable; nunca por dato sensible.

### OpenFeature como estandar de SDK
Se recomienda usar [OpenFeature](https://openfeature.dev/) como abstraccion cliente en todos los stacks. Beneficios:
- Desacopla el codigo aplicacion del proveedor (Unleash, LaunchDarkly, flagd, ConfigCat, etc.).
- SDK oficiales para Node, Java, Go, Python y mas.
- Se puede iniciar con un provider en memoria en desarrollo y cambiar a un provider remoto sin tocar el codigo de negocio.

Ejemplos de integracion por stack:
- `stacks/node-next/template/src/lib/feature-flags.ts`
- `stacks/java-monolith/template/src/main/java/com/wakaya/erp/featureflags/FeatureFlagsConfig.java`

### Relacion con estrategias de despliegue
- `docs/fase-7-deploy/07.01-estrategias-despliegue-seguro.md` documenta como combinar feature flags con canary y blue/green.
- Un flag nuevo debe quedar listado en `releases/vX.Y.Z.md` con su objetivo, cohortes iniciales y fecha objetivo de retiro.

<a id="checklist"></a>
## Checklist
- [ ] Cada feature nueva declara si requiere flag y su nombre.
- [ ] Los killswitch criticos estan probados en drills de incidente.
- [ ] Los flags retirados se eliminan del codigo dentro del plazo acordado.
- [ ] El catalogo de flags vive versionado o expuesto via servicio con auditoria.

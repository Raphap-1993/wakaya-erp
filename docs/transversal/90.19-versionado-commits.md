<a id="objetivo"></a>
## Objetivo
Unificar como se versiona, etiqueta y comunica cada release de la plantilla y de los proyectos derivados.

<a id="semver-aplicado-a-la-plantilla"></a>
## SemVer aplicado a la plantilla
- Mayor (`X.0.0`): cambios incompatibles que exigen migracion manual en proyectos ya instanciados.
- Menor (`x.Y.0`): nuevas capacidades o documentos que requieren adaptacion ligera.
- Parche (`x.y.Z`): correcciones, aclaraciones, rutas rotas, mejoras internas sin cambio de contrato.

Las reglas para proyectos derivados son equivalentes pero se aplican al contrato publico del producto.

<a id="conventional-commits"></a>
## Conventional Commits
Formato esperado:

```
<tipo>(<ambito opcional>): <descripcion corta en presente>

<cuerpo opcional explicando por que>

<footer opcional: BREAKING CHANGE, Refs, Co-authored-by>
```

Tipos aceptados: `feat`, `fix`, `docs`, `chore`, `refactor`, `perf`, `test`, `build`, `ci`, `revert`.

Ejemplos:

- `feat(specs): agregar feature 004 de notificaciones`
- `fix(docs): corregir rutas rotas en fase-7`
- `chore(ci): unir workflows de calidad y seguridad`

<a id="branches-y-flujo"></a>
## Branches y flujo
- Rama principal protegida: `main`.
- Ramas de trabajo: `feat/<slug>`, `fix/<slug>`, `docs/<slug>`, `chore/<slug>`.
- PRs con linear history cuando es posible; se permite merge commit con titulo canonico.
- Las ramas largas (`release/*`) solo se abren para releases mayores con periodo de endurecimiento.

<a id="changelog"></a>
## CHANGELOG
- Un solo `CHANGELOG.md` en la raiz, en formato Keep a Changelog.
- Cada release tiene entrada con seccion `Added`, `Changed`, `Fixed`, `Removed`, `Security` segun aplique.
- Las entradas se redactan en espanol, con foco en el "que" y el "por que".

<a id="tagging-y-release-notes"></a>
## Tagging y release notes
- Los tags siguen el formato `vX.Y.Z`.
- Cada tag publicado tiene su archivo `releases/vX.Y.Z.md` con resumen, impacto y pasos de upgrade.
- El workflow `release.yml` publica la release en GitHub con el contenido de ese archivo.

<a id="checklist"></a>
## Checklist
- [ ] El mensaje del commit cumple Conventional Commits.
- [ ] El PR esta asociado a un issue o a una spec.
- [ ] Si el cambio afecta contratos, se incrementa la version correspondiente.
- [ ] Se actualiza el CHANGELOG y, si es release, se crea el archivo en `releases/`.

<a id="objetivo"></a>
## Objetivo
Establecer como el equipo disena, ejecuta y cierra experimentos para tomar decisiones de producto y tecnologia basadas en evidencia, y como esto se conecta con releases y feature flags.

<a id="tipos-de-experimento"></a>
## Tipos de experimento
- A/B test (dos variantes).
- Multi-variant (A/B/C/...) con correccion estadistica.
- Rollout gradual (holdout): la variante nueva se expone al 10%, el resto mide baseline.
- Feature on/off (killswitch) sin asignacion aleatoria, para riesgo tecnico.
- Shadow release: codigo nuevo ejecuta en paralelo sin afectar al usuario, se comparan salidas.

<a id="ciclo-de-vida-de-un-experimento"></a>
## Ciclo de vida de un experimento
1. Hipotesis documentada (`Si X, entonces Y se movera Z`), con metrica primaria y secundaria.
2. Diseno: tamano de muestra, potencia (>=80%), alpha (<=0.05), duracion estimada.
3. Instrumentacion: events `experiment.exposure.v1` y `experiment.outcome.v1` con `experimentKey`, `variant`, `userHash`.
4. Activacion via feature flag segmentado.
5. Ejecucion hasta alcanzar significancia o duracion maxima (con limite de cambios concurrentes).
6. Analisis: revisar guardrail metrics (no empeorar negocio ni SLO).
7. Decision documentada en `releases/vX.Y.Z.md`:
   - Promover a 100% (y programar retiro de flag).
   - Revertir (y capturar aprendizaje).
   - Iterar (nuevo experimento).
8. Postmortem breve si el experimento daaa baja calidad o datos sesgados.

<a id="decidir-con-datos-no-con-opiniones"></a>
## Decidir con datos, no con opiniones
- Toda metrica de negocio tiene una definicion precisa versionada en `docs/fase-8-operacion/metricas.md`.
- No se cambian definiciones durante un experimento.
- Se aplica correccion por multiples comparaciones cuando hay mas de dos variantes.
- La metrica primaria gana; las secundarias solo sirven para contexto.

<a id="integracion-con-feature-flags"></a>
## Integracion con feature flags
- Cada experimento vive detras de un flag `experiment.<area>.<nombre>`.
- El provider asigna variante de forma sticky por `userHash`.
- La asignacion se loguea a un topic dedicado (no al stream operativo).
- El cierre del experimento implica limpiar flag y cohortes.

<a id="checklist"></a>
## Checklist
- [ ] Hipotesis, metrica primaria y criterio de parada estan escritos antes del primer deploy.
- [ ] La instrumentacion de exposure esta probada en QA.
- [ ] El analisis se ejecuta con scripts reproducibles, no con screenshots.
- [ ] El resultado queda versionado (PR + release notes + decision log).

<a id="objetivo"></a>
## Objetivo
Definir una anatomia comun para `agents`, `prompts` y `skills` de la capa de IA, de modo que el repositorio no solo diga que artefacto producir, sino tambien como producirlo con una disciplina repetible.

<a id="por-que-existe-esta-guia"></a>
## Por que existe esta guia
La plantilla ya cubre bien fases, rutas y entregables. El siguiente nivel de madurez es que la capa `ai/` tambien deje claro:
- cuando usar cada artefacto,
- cuando no usarlo,
- que referencias cargar,
- como verificar la salida,
- como evitar racionalizaciones del tipo "parece bien" sin evidencia.

<a id="anatomia-minima-por-artefacto"></a>
## Anatomia minima por artefacto

| Artefacto | Secciones minimas |
|---|---|
| Agent | objetivo, usalo cuando, no lo uses cuando, entradas minimas, salidas esperadas, rutas destino, verificacion minima |
| Prompt | objetivo, usalo cuando, no lo uses cuando, entradas minimas, salida esperada, rutas destino, verificacion minima, pedido base |
| Skill | objetivo, aplicala cuando, no la apliques cuando, entradas minimas, flujo recomendado, anti-rationalizations, red flags, verification evidence, referencias |
| Command | objetivo, fases donde aplica mejor, required inputs, process, anti-rationalizations, red flags, verification evidence |
| Quality gate | objetivo, evidencia minima, bloqueantes tipicos, rutas relacionadas |

### Regla practica
- `Agent`: conduce una salida amplia por rol o dominio.
- `Prompt`: pide un entregable concreto.
- `Skill`: fija una forma de trabajar o revisar.

<a id="lifecycle-y-quality-gates"></a>
## Lifecycle y quality gates
La capa de IA del repositorio usa dos mecanismos adicionales de enforcement:

- `ai/commands/` para entrar por lifecycle: `/plan`, `/document`, `/ux`, `/prototype`, `/spec`, `/build`, `/test`, `/review`, `/ship`.
- `ai/quality-gates/` para validar readiness por bloques de fases y subflujos: `0-1`, `ux-ready`, `prototype-ready`, `spdd-approved`, `2-3`, `frontend-spdd-ready`, `4-6`, `7-8`.

La idea practica es:
1. elegir un `command` segun la intencion del trabajo,
2. usar `skills`, `prompts` y `references` minimos,
3. cerrar contra el `quality gate` correspondiente.

<a id="contrato-de-verificacion"></a>
## Contrato de verificacion
Todo artefacto oficial de `ai/` debe reforzar estas reglas:
- declarar la ruta de salida oficial,
- citar la entrada que origina la salida,
- marcar supuestos y preguntas abiertas,
- distinguir claramente propuesta vs decision aprobada,
- pedir evidencia minima antes de dar por cerrada una fase critica,
- resistir racionalizaciones del tipo "lo vemos despues" mediante anti-rationalizations y red flags.

### Evidencia minima segun momento
- Discovery y requerimientos: supuestos, huecos, dependencias y siguiente paso.
- UX/SPDD: producto, journeys, pantallas, accesibilidad, prototipo HTML5/Penpot, criterio minimo UX, sistema de componentes y trazabilidad.
- Arquitectura: trade-offs, RNF, ADR y operacion.
- SDD: trazabilidad a RF/HU, riesgos, pruebas y tareas ejecutables.
- QA, release y operacion: evidencia verificable, smoke checks, rollback, metricas y backlog evolutivo.

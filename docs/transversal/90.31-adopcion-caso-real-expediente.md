<a id="objetivo"></a>
## Objetivo
Adoptar la plantilla canonica (`C:\template\project-template`) en un proyecto real de "Gestion de Expedientes" alojado en `C:\template\caso-real\expediente`. El playbook esta orientado a Windows con PowerShell 7+ y produce un servicio ejecutable, documentacion completa y los entregables por fase del ciclo SDD.

<a id="hipotesis-del-caso"></a>
## Hipotesis del caso
El caso real "expediente" comparte el mismo dominio del caso canonico que ya describe la plantilla. Por eso `apiResourceName=expediente` (singular), `apiResourcePlural=expedientes`, `apiResourcePath=/api/expedientes`. La matriz RBAC sigue `docs/fase-3-arquitectura/03.08-auth-authz.md`: roles de dominio `operador`, `analista`, `aprobador`, `auditor`, `administrador`; permisos `EXPEDIENTE_READ`, `EXPEDIENTE_WRITE`, `EXPEDIENTE_APPROVE`, `EXPEDIENTE_AUDIT`, `ADMIN_CONFIG`.

**Stack elegido: `quarkus-angular`** con **Keycloak** como Identity Provider. Justificacion baseline:

- **Backend Quarkus 3.12** (Java 21): cold start menor a 1s en imagen nativa GraalVM, footprint de memoria reducido, ideal para despliegue en Kubernetes con autoscaling agresivo (HPA + PDB ya configurados en `ops/k8s/`).
- **Frontend Angular 21 (Nx workspace)**: tipado estricto end-to-end, DI nativa, RxJS para flujos reactivos en gestion de expedientes (busqueda, filtros, transiciones de estado), Angular Material para tablas accesibles y formularios complejos.
- **Keycloak self-hosted** como IdP: open source, realm dedicado por producto, mapeo nativo del claim `roles` para `@RolesAllowed`, soporta PKCE para flujo Angular via `angular-auth-oidc-client`.
- **AuthN/AuthZ ya cableado**: el stack incluye `quarkus-smallrye-jwt` + `quarkus-security` + `Permission`/`Role`/`Rbac`/`RbacEvaluator.java` listos para `@RolesAllowed({"operador","analista","aprobador","auditor","administrador"})` en los recursos REST y `RbacEvaluator.can("expediente:approve")` para logica ABAC ad-hoc.
- **Compatibilidad cross-stack**: el job CI `rbac-consistency` valida que la matriz de roles coincida con `node-next`/`java-monolith`/`spring-react`, evitando drift si se agregan stacks adicionales en el futuro.

Esta decision **debe registrarse formalmente como ADR-001** en el repo del proyecto generado (`docs/fase-3-arquitectura/adr/ADR-001-stack-tecnologico.md`) tras llenar el [`90.05-checklist-seleccion-stack.md`](90.05-checklist-seleccion-stack.md) con el contexto especifico (talento del equipo, infra cloud, restricciones regulatorias, SLOs). Si el contexto no respalda la eleccion, considera los stacks alternativos:

| Stack | Backend | Frontend | Cuando preferirlo |
| --- | --- | --- | --- |
| **`quarkus-angular`** (elegido) | Quarkus + Java 21 + Smallrye JWT | Angular 21 (Nx) | Backend con cold start critico (K8s autoscale, serverless) y frontend complejo con DI/forms. |
| `java-monolith` | Spring Boot 3.3 + Spring Security OAuth2 | Sin frontend (agregar `node-next` o `react` aparte) | Equipo Java tradicional, monolito modular sin frontend especifico, ecosistema Spring maduro. |
| `spring-react` | Spring Boot 3.3 + Spring Security OAuth2 | React + Vite | Equipo React, backend Spring tradicional. |
| `node-next` | Next.js 14 (server actions + API routes) | Next.js 14 | Full-stack TypeScript, SSR, edge rendering. |

<a id="prerrequisitos-en-windows"></a>
## Prerrequisitos en Windows
Para `quarkus-angular` (stack elegido):

- **JDK 21** (Eclipse Temurin recomendado): `java --version` debe mostrar 21.
- **Maven 3.9+** o usar el wrapper `./mvnw.cmd` que ya esta en `backend/`.
- **Node.js 20+** y **npm 10+** para el frontend Nx/Angular: `node --version` y `npm --version`.
- **Docker Desktop** activo (Postgres + Redis + Jaeger + Keycloak local via `docker-compose`).
- **PowerShell** 7 o superior (preferible al 5.1; el script `.ps1` corre en ambos).
- **Git** 2.40+: `git --version`.
- **GraalVM 21** (opcional, solo si quieres construir imagen nativa): `gu install native-image`.
- Acceso a la organizacion GitHub donde se publicara el repo `acme/expediente`.

Si tu PowerShell tiene la politica de ejecucion restrictiva, habilita scripts firmados localmente:

```powershell
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
```

## Paso 3 - Validar la config del proyecto real
La plantilla incluye `config\expediente.config.example.json` con los valores listos para el caso real. Ajusta los campos sensibles (organizacion GitHub, dominios DNS reales, IdP, observabilidad, region cloud) y guardalo como `config\expediente.config.json` (sin el `example.`):

```powershell
Copy-Item config\expediente.config.example.json config\expediente.config.json
notepad config\expediente.config.json
```

Campos a revisar minimamente:

- `github.organization` y `github.repository`: organizacion y repo objetivo (ej. `acme/expediente`).
- `terraform.devDomain`/`stagingDomain`/`prodDomain`: dominios reales.
- `auth.oidcIssuer`/`oidcJwksUrl`/`oidcAudience`: tu IdP (Keycloak/Auth0/Entra). Para Keycloak self-hosted en `iam.acme.example`, los valores ya estan plantilleados.
- `dr.primaryRegion`/`secondaryRegion`: regiones AWS reales (ej. `us-east-1` y `us-west-2`).
- `secrets.provider`: `vault`, `aws` o `gcp` segun la organizacion. El default del ejemplo es `vault`.

Valida la config contra el schema:

```powershell
node scripts\validate-template-config.mjs --config config\expediente.config.json
```

Salida esperada:

```
OK. config/expediente.config.json cumple el schema.
```

<a id="paso-4---generar-el-proyecto-real"></a>
## Paso 4 - Generar el proyecto real
La ruta destino es `C:\template\caso-real\expediente`. Asegurate de que NO exista (el script aborta si ya existe).

### 4.1 Comando elegido (`quarkus-angular`)

```powershell
if (Test-Path C:\template\caso-real\expediente) {
    Write-Host "Ya existe; muevelo o borralo antes de continuar."
    exit 1
}

node scripts\new-service.mjs `
    --stack quarkus-angular `
    --config config\expediente.config.json `
    --dest C:\template\caso-real\expediente
```

Equivalente con el wrapper PowerShell nativo:

```powershell
.\scripts\new-service.ps1 `
    --stack quarkus-angular `
    --config config\expediente.config.json `
    --dest C:\template\caso-real\expediente
```

### 4.2 Comandos para los otros stacks (referencia)
Si la decision arquitectonica cambia tras llenar el [`90.05-checklist-seleccion-stack.md`](90.05-checklist-seleccion-stack.md), uno de estos comandos sustituye al de 4.1:

```powershell
# Spring Boot monolitico (sin frontend incluido)
node scripts\new-service.mjs --stack java-monolith --config config\expediente.config.json --dest C:\template\caso-real\expediente

# Spring Boot + React/Vite frontend
node scripts\new-service.mjs --stack spring-react --config config\expediente.config.json --dest C:\template\caso-real\expediente

# Next.js full-stack (TypeScript, server actions + API routes)
node scripts\new-service.mjs --stack node-next --config config\expediente.config.json --dest C:\template\caso-real\expediente
```

### 4.3 Que sucede internamente al correr `new-service.mjs`

1. Valida la config contra el schema (paso 3 implicito).
2. Copia `stacks\<stack>\template\` -> `C:\template\caso-real\expediente`.
3. Sustituye los tokens `reservation`, `com.wakaya.erp`, `team-wakaya`, etc. por los valores de la config (incluye `.java`, `.ts`, `.html`, `.css`, `.scss`, `.yaml`, `.properties`, `.xml`, `.gradle`, `.kts`, `.sql`).
4. Renombra el package Java de `com.wakaya.erp` a `com.acme.expediente` (incluye renombrar directorios).
5. Ejecuta `check-docs.mjs` y `check-template-instantiation.mjs --mode instantiated` sobre la salida.
6. Inicializa git con commit inicial (a menos que pases `--no-git`).
7. Imprime checklist de proximos pasos.

<a id="paso-8---configurar-idp-secrets-y-workflows"></a>
## Paso 8 - Configurar IdP, secrets y workflows

### 8.1 Keycloak (IdP elegido)

Para desarrollo local, anade un servicio Keycloak al `ops\docker\docker-compose.yml`:

```yaml
keycloak:
  image: quay.io/keycloak/keycloak:24.0
  command: start-dev --import-realm
  environment:
    KEYCLOAK_ADMIN: admin
    KEYCLOAK_ADMIN_PASSWORD: admin
  ports:
    - "8180:8080"
  volumes:
    - ./keycloak/realm-expediente.json:/opt/keycloak/data/import/realm-expediente.json
```

Crea el realm export inicial en una ruta del proyecto generado, por ejemplo ops/docker/keycloak/realm-expediente.json, con:

- **Realm**: `expediente`.
- **Client**: `expediente-api` (Confidential, Service Accounts Roles ON, Direct access grants ON, Standard flow ON, audiences include `https://api.expediente.acme.example`).
- **Client roles** del client `expediente-api`: `operador`, `analista`, `aprobador`, `auditor`, `administrador`.
- **Token mapper**: Group/Role membership con claim name `roles` y type `String list`.
- **Frontend client** (opcional): `expediente-web` (Public, PKCE only, redirect URIs `http://localhost:4200/*` para dev).

Para entornos productivos, replicar la misma estructura via Terraform con el provider `mrparkers/keycloak` o el script `keycloakctl`.

Variables de entorno que debe leer el backend Quarkus (vienen de `application.properties`):

```
OIDC_ISSUER=https://iam.acme.example/realms/expediente
OIDC_JWKS_URL=https://iam.acme.example/realms/expediente/protocol/openid-connect/certs
OIDC_AUDIENCE=https://api.expediente.acme.example
OIDC_ROLES_CLAIM=roles
```

Para el frontend Angular, anadir `angular-auth-oidc-client`:

```powershell
cd C:\template\caso-real\expediente\frontend
npm install angular-auth-oidc-client
```

Y configurar el archivo de bootstrap de la app web, por ejemplo apps/web/src/app/app.config.ts:

```typescript
import { ApplicationConfig } from "@angular/core";
import { provideAuth, authInterceptor } from "angular-auth-oidc-client";
import { provideHttpClient, withInterceptors } from "@angular/common/http";

export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(withInterceptors([authInterceptor()])),
    provideAuth({
      config: {
        authority: "https://iam.acme.example/realms/expediente",
        redirectUrl: window.location.origin,
        postLogoutRedirectUri: window.location.origin,
        clientId: "expediente-web",
        scope: "openid profile email roles",
        responseType: "code",
        silentRenew: true,
        useRefreshToken: true,
        secureRoutes: ["https://api.expediente.acme.example"],
      },
    }),
  ],
};
```

### 8.2 Variables y secrets de GitHub
En GitHub:

1. **Variables de organizacion o repositorio** (`Settings` -> `Secrets and variables` -> `Actions` -> `Variables`):
   - `SECRETS_PROVIDER` = `vault` (o `aws`/`gcp`).
   - `VAULT_URL` = `https://vault.acme.example` (si provider=vault).
   - `AWS_ROTATE_ROLE_ARN` = `arn:aws:iam::123456789012:role/expediente-rotate-secrets` (si provider=aws).
   - `AWS_REGION` = `us-east-1`.
   - `AUDIT_BUCKET` = `acme-audit-logs` (para Object Lock del log de rotacion).
   - `DR_KUBECTL_CONTEXT`, `DR_NAMESPACE`, `DR_PROMETHEUS_URL`, `DR_ERROR_RATE_THRESHOLD` (`0.01`).
2. **Secrets de organizacion** (`Settings` -> `Secrets`):
   - `SECRETS_KUBECONFIG` = kubeconfig del cluster productivo (si rotate-secrets debe esperar convergencia ESO).
   - `DR_KUBECONFIG` = kubeconfig del cluster pasivo para gameday-dr.
3. **OIDC trust** (Vault/AWS/GCP) hacia GitHub Actions OIDC con sub claim `repo:acme/expediente:ref:refs/heads/main`.
4. **Habilita workflows**: van en `.github/workflows/`. Manda un primer push a `main` y verifica que pasen `docs`, `ci`, `template`, `rbac-consistency`, `security`. `rotate-secrets` y `gameday-dr` quedan en cron por defecto.

<a id="troubleshooting"></a>
## Troubleshooting

### "Argumento desconocido: --skip-smoke"
Tu copia del template tiene una version anterior de `new-service.mjs`. Actualiza con `git pull --rebase` en `C:\template\project-template`.

### `mvn -q test` falla con nombre de permiso desconocido
El enum `Permission.java` del proyecto real debe usar los nombres de dominio (`EXPEDIENTE_READ`, `EXPEDIENTE_WRITE`, `EXPEDIENTE_APPROVE`, `EXPEDIENTE_AUDIT`, `ADMIN_CONFIG`) alineados con `docs/fase-3-arquitectura/03.08-auth-authz.md`. Si el proyecto fue generado con una version antigua del template (antes de que se alinearan los nombres de dominio), actualiza manualmente `Permission.java` y `Role.java` para usar los nombres de dominio reales y actualiza tambien `RbacTest.java`.

### Tokens `reservation` aparecen en archivos `.java` post-instanciacion
Bug corregido en v12.6.0. Confirma `head -1 C:\template\project-template\README.md` muestra `v12.6.0` o superior.

### `node scripts\new-service.mjs` se queda esperando en `npm run build`
El smoke build de node-next requiere `npm install` previo. Corre con `--skip-smoke` o instala deps:

```powershell
cd C:\template\caso-real\expediente-web
npm install
```

### Backstage muestra `wakaya-erp` literal
El `catalog\` raiz del template requiere ser instanciado via `init-project.mjs` directamente sobre el clone del template (no sobre el proyecto generado). Ejecuta:

```powershell
cd C:\template\project-template
node scripts\init-project.mjs --config config\expediente.config.json --root .
```

Y carga `catalog/all.yaml` desde `app-config.yaml` de Backstage (en Windows la ruta absoluta tipica es C:\\template\\project-template\\catalog\\all.yaml sin backticks porque es texto, no path verificable del repo).

### El validador `check-rbac-consistency` falla post-instanciacion
Si modificaste manualmente la matriz en uno de los stacks, replica el cambio en los otros 3 antes de hacer push. Verifica con:

```powershell
node C:\template\project-template\ci\scripts\check-rbac-consistency.mjs --root C:\template\project-template
```

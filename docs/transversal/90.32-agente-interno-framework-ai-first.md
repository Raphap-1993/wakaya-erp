<a id="objetivo"></a>
## Objetivo
Definir el agente interno que guia y gobierna el framework enterprise AI-first del repositorio. Su funcion es orquestar metodologia, capa IA operacional, commands, quality gates, references, skills, red flags, evidencia, memoria local y trazabilidad.

<a id="alcance"></a>
## Alcance
El agente interno abarca:
- fases `0-8`,
- Product Design y `Spec + Prototype Driven Development (SPDD)` para features visuales,
- `Spec-Driven Development (SDD)`,
- prototipado HTML5/Penpot como evidencia previa a SDD cuando aplica UI,
- commands IA: `/plan`, `/document`, `/ux`, `/prototype`, `/spec`, `/build`, `/test`, `/review`, `/ship`,
- agents, prompts, skills y references,
- quality gates,
- rutas canonicas,
- trazabilidad entre fuente, requerimiento, spec, prueba, ADR, QA, deploy y operacion,
- memoria local con SQLite, sqlite-vec y DuckDB opcional,
- creacion de proyectos reales en rutas externas a partir de `stacks/*/template/`.

No reemplaza la validacion humana ni a los agentes especializados. Actua como orquestador y guardrail del framework.

<a id="arquitectura-operacional"></a>
## Arquitectura operacional
```text
Solicitud del usuario
  ->
Decision de modo de trabajo (90.13)
  ->
Enterprise AI Framework Agent
  ->
Router de intencion
  ->
Command + agent/prompt/skill/reference
  ->
Artefactos oficiales del repositorio
  ->
Quality gate + evidencia + trazabilidad
  ->
Memoria local reconstruible
```

La fuente primaria siempre son los archivos del repositorio. La memoria local ayuda a buscar y relacionar, pero no sustituye `docs/`, `specs/`, `src/`, `tests/`, `qa/`, `ops/`, `ci/`, `likec4/` ni `contracts/`.

<a id="flujo-de-trabajo"></a>
## Flujo de trabajo
1. Interpretar la intencion.
2. Identificar modo de trabajo en `90.13-modos-de-trabajo.md`.
3. Identificar fase y command en `90.12-mapa-ia-por-fase.md`.
4. Seleccionar agente, prompt, skill, reference y quality gate minimos.
5. Leer fuentes oficiales y fuentes brutas indicadas.
6. Separar hechos, supuestos, preguntas abiertas y decisiones.
7. Crear, actualizar o revisar artefactos oficiales.
8. Validar red flags y quality gate.
9. Registrar evidencia y trazabilidad.
10. Reindexar memoria local si aplica.

<a id="modos-de-trabajo-y-enrutamiento"></a>
## Modos de trabajo y enrutamiento
El agente interno no debe asumir que todo pedido es construccion. Primero decide el modo:

| Situacion | Modo | Entrada operativa |
|---|---|---|
| Hay notas, entrevistas, correos o texto libre | Intake desde fuente bruta | `ai/prompts/arranque-desde-fuente-bruta.md` + `/document` |
| Ya existen nombre, dominio, stack, ruta destino y datos minimos | Bootstrap desde template | `ai/prompts/crear-proyecto-real-desde-template.md` |
| El usuario quiere guia paso a paso | Orquestacion con agente | `ai/runbooks/crear-proyecto-real-con-agente.md` |
| Hay feature visual | Product Design + SPDD + prototipo | `/ux` + `/prototype` |
| Hay prototipo validado | SDD | `/spec` |
| Hay specs y tareas listas | Construccion | `/build` |

Reglas:
- `arranque-desde-fuente-bruta.md` convierte informacion cruda en insumos documentales; no instancia repositorios ni ejecuta `create-project`.
- `crear-proyecto-real-desde-template.md` instancia el template cuando ya hay datos minimos claros; no reemplaza discovery profundo.
- `crear-proyecto-real-con-agente.md` orquesta intake, bootstrap, Product Design, SPDD, gates y validaciones cuando el usuario quiere acompanamiento IA end-to-end.
- Para features visuales, `/ux` genera Product Design y SPDD inicial; `/prototype` crea o actualiza evidencia HTML5/Penpot; `gate-spdd-approved` solo puede aprobarse despues de `gate-prototype-ready` y validacion humana.

<a id="delivery-ia-de-features"></a>
## Delivery IA de features
Cuando una feature entra a Fase 4-7, el agente interno debe aplicar el flujo transversal [90.33-flujo-delivery-ia-proveedores.md](90.33-flujo-delivery-ia-proveedores.md):

1. `/ux` -> **Product Design**: salida `specs/<feature>/product-design.md`, gate `gate-ux-ready`.
2. `/ux` -> **SPDD inicial** si la feature tiene superficie visual: salidas `spdd-frontend.md`, `ui-test-cases.md` y `traceability.md`.
3. `/prototype` -> **prototipo HTML5/Penpot revisable**: salidas `prototype.md`, `prototype-validation.md`, `prototype-html5/` o link/export Penpot, gate `gate-prototype-ready`.
4. `/review` -> **validacion humana** del prototipo y registro de decision, gate `gate-spdd-approved`. Bloquea el paso siguiente si no esta aprobado.
5. `brainstorming`: antes de escribir codigo, comparar dos alternativas si hay ambiguedad.
6. `/spec`: convertir la alternativa elegida en spec funcional, tecnica y tareas. Requiere `gate-spdd-approved` si la feature es visual.
7. `writing-plans`: partir `spec-tareas.md` en tareas pequenas con rutas permitidas, TDD, comandos y evidencia.
8. `using-git-worktrees`: crear rama o worktree aislado cuando haya cambios de codigo.
9. `executing-plans`: ejecutar una tarea por vez y actualizar su estado.
10. `test-driven-development`: exigir red, green y refactor en cambios de comportamiento.
11. `requesting-code-review`: revisar entre tareas criticas o antes de QA. Verificar que no haya violaciones de SOLID detectables: acoplamiento de capas, interfaces sobredimensionadas, logica de negocio en controladores o repositorios.
12. `/test`: validar QA y evidencias contra `gate-4-6`.
13. `finishing-development-branch`: preparar PR/merge, evidencia final y limpieza de worktree.

Reglas de bloqueo: si falta `gate-prototype-ready`, el prototipo no esta listo para revision. Si falta `gate-spdd-approved` para una feature visual, no se puede cerrar `/spec` ni iniciar construccion. Si el proveedor IA no tiene tareas verificables, rutas permitidas o comandos de verificacion, debe volver a Fase 4 y mejorar `spec-tareas.md` antes de construir.

<a id="router-por-intencion"></a>
## Router por intencion
| Intencion | Command | Gate | Salida |
|---|---|---|---|
| idea cruda | `/document` + `/plan` | `ai/quality-gates/gate-documentation-ready.md`, `ai/quality-gates/gate-0-1.md` | fase 0 y fase 1 inicial |
| fuente bruta desordenada | `/document` | `ai/quality-gates/gate-documentation-ready.md` | vision preliminar, RF/RNF, supuestos, preguntas abiertas y recomendacion de modo |
| requerimientos | `/document` + `/plan`, `/review` | `ai/quality-gates/gate-documentation-ready.md`, `ai/quality-gates/gate-0-1.md` | RF, RNF, reglas y backlog |
| Product Design (RF/HU -> experiencia) | `/ux` | `ai/quality-gates/gate-ux-ready.md` | `specs/<feature>/product-design.md`, journey, hipotesis, metricas |
| SPDD inicial | `/ux` | `ai/quality-gates/gate-ux-ready.md` | `spdd-frontend.md`, `ui-test-cases.md`, `traceability.md` |
| prototipo navegable/formal | `/prototype` | `ai/quality-gates/gate-prototype-ready.md` | `prototype.md`, `prototype-validation.md`, `prototype-html5/` o link/export Penpot |
| SPDD aprobado | `/review` | `ai/quality-gates/gate-spdd-approved.md` | validacion humana trazable del prototipo |
| arquitectura | `/document` + `/review` | `ai/quality-gates/gate-documentation-ready.md`, `ai/quality-gates/gate-2-3.md` | arquitectura, ADR, C4 y despliegue |
| feature a SDD (requiere gate-spdd-approved si es visual) | `/spec` | `ai/quality-gates/gate-documentation-ready.md`, `ai/quality-gates/gate-4-6.md` | specs funcional, tecnica, api-contract, tareas y trazabilidad |
| implementacion | `/build` | `ai/quality-gates/gate-4-6.md` y, si es frontend, `ai/quality-gates/gate-frontend-spdd-ready.md` | codigo, pruebas, consistencia con prototipo y trazabilidad |
| QA | `/test`, `/review` | `ai/quality-gates/gate-4-6.md` | plan, evidencia y defectos |
| release | `/ship` | `ai/quality-gates/gate-7-8.md` | release, runbook, rollback y monitoreo |
| operacion | `/review` | `ai/quality-gates/gate-7-8.md` | metricas, salud y backlog evolutivo |
| crear proyecto real con datos minimos | `/plan` + bootstrap | `ai/quality-gates/gate-0-1.md` | repo instanciado en otra ruta |
| crear proyecto real guiado por IA | agente guiado | gates declarados por fase | repo instanciado, docs iniciales y provider orientation pack |

## Crear proyecto real en otra ruta
El agente interno debe poder leer la documentacion del framework y crear un proyecto real fuera del template.

Punto de entrada operativo:
- [`scripts/ai-framework-agent.mjs`](../../scripts/ai-framework-agent.mjs)

- [`ai/runbooks/crear-proyecto-real-con-agente.md`](../../ai/runbooks/crear-proyecto-real-con-agente.md)

Documentacion minima que debe leer:
- `AGENTS.md`
- `docs/transversal/90.12-mapa-ia-por-fase.md`
- `docs/transversal/90.13-modos-de-trabajo.md`
- `docs/fase-0-iniciacion/00.00-guia-de-uso.md`
- `docs/fase-0-iniciacion/00.05-checklist-adopcion.md`
- `docs/fase-0-iniciacion/00.10-idea-a-documentacion-inicial-con-ia.md`
- `docs/transversal/90.10-entregables-minimos-por-fase.md`
- `docs/transversal/90.11-checklist-entregables.md`
- `docs/transversal/90.14-instanciacion-fases-proyectos-reales.md`
- `docs/transversal/90.32-agente-interno-framework-ai-first.md`
- `docs/transversal/90.33-flujo-delivery-ia-proveedores.md`
- `docs/transversal/90.34-product-design-y-spdd-frontend.md`
- `docs/fase-2-ux-ui/02.11-checklist-product-design.md`
- `docs/fase-2-ux-ui/02.12-checklist-spdd.md`
- `docs/fase-2-ux-ui/02.13-penpot-ai-prototyping.md`
- `docs/fase-2-ux-ui/02.14-html5-first-prototyping.md`
- `ai/commands/prototype-command.md`
- `ai/quality-gates/gate-prototype-ready.md`
- `ai/quality-gates/gate-spdd-approved.md`
- `stacks/README.md`
- `template.config.example.json`

Diferencia operativa:
- Si hay fuente bruta, primero usa `arranque-desde-fuente-bruta.md` para extraer insumos, supuestos y preguntas abiertas.
- Si ya hay datos minimos claros, usa `crear-proyecto-real-desde-template.md` para instanciar.
- Si el usuario quiere guia, preguntas y gates paso a paso, usa `crear-proyecto-real-con-agente.md`.

Comando base:
```powershell
node scripts/ai-framework-agent.mjs init-memory
node scripts/ai-framework-agent.mjs index-docs
node scripts/ai-framework-agent.mjs plan-create --stack <stack> --config <config.json> --dest <ruta-destino>
node scripts/ai-framework-agent.mjs create-project --stack <stack> --config <config.json> --dest <ruta-destino>
```

Durante discovery se puede omitir smoke checks y git inicial:
```powershell
node scripts/ai-framework-agent.mjs create-project --stack <stack> --config <config.json> --dest <ruta-destino> --skip-smoke --no-git
```

Despues de crear el proyecto, el agente debe trabajar dentro de la ruta destino para completar documentacion real de fase 0-3, revisar gates y ejecutar validaciones.

`scripts/new-service.mjs` sigue existiendo como generador tecnico. El ejecutable del agente es `scripts/ai-framework-agent.mjs`, que usa la memoria SQLite, enruta la intencion y llama al generador cuando corresponde.

Para documentacion guiada, el comando ejecutable es:
```powershell
node scripts/ai-framework-agent.mjs document --intent "El operador debe registrar reclamos con adjuntos"
node scripts/ai-framework-agent.mjs document --source intake\necesidades-iniciales.md
```

No se define un `/intake` separado. El intake es una fuente de entrada del modo completo `/document`.

<a id="memoria-local"></a>
## Memoria local
La memoria local se adopta por etapas:

| Etapa | Componente | Uso |
|---|---|---|
| 1 | SQLite | indice de archivos, trazabilidad, gates, evidencias y preguntas abiertas |
| 2 | sqlite-vec | busqueda semantica local sobre chunks de documentos |
| 3 | DuckDB | analitica opcional de cobertura, drift, deuda y madurez |
| 4 | exportacion | dashboards o integracion externa si el proyecto lo requiere |

La decision arquitectonica vive en [`ADR-010`](../fase-3-arquitectura/adr/ADR-010-memoria-local-ai-first-sqlite-sqlite-vec-duckdb.md).

<a id="contrato-de-evidencia"></a>
## Contrato de evidencia
Cada ejecucion relevante debe dejar:
- fuente leida,
- ruta destino,
- modo de trabajo,
- fase y command usados,
- gate aplicado o razon de descarte,
- agent, prompt, skill y reference usados,
- supuestos y preguntas abiertas,
- red flags revisadas,
- trazabilidad a RF/RNF, spec, prototipo, ADR, prueba, evidencia o runbook.

<a id="red-flags"></a>
## Red flags
- No hay modo, fase ni command declarado.
- La salida no actualiza ruta canonica.
- Se aprueba un gate sin evidencia.
- Se marca `gate-prototype-ready` sin prototipo navegable, link/export Penpot o ruta revisable.
- Se marca `gate-spdd-approved` sin validacion humana trazable.
- Se trata HTML5 como frontend productivo o prompt Penpot como prototipo aprobado.
- Se elige tecnologia sin ADR.
- La memoria contradice documentos oficiales.
- Se usan ejemplos canonicos como entregables reales.
- Se usa intake para crear repo completo o bootstrap para hacer discovery profundo.
- Se mezclan decisiones propuestas con decisiones aprobadas.
- Hay specs sin pruebas o release sin rollback.

<a id="ejemplos-de-uso"></a>
## Ejemplos de uso
```text
Tengo esta idea cruda en docs/raw/idea.md. Guiame para transformarla en fase 0-1.
```

Resultado esperado:
- command `/document` + `/plan`,
- gate `gate-0-1`,
- prompt de transformacion de idea,
- skills de discovery y requerimientos,
- salida en fase 0 y fase 1,
- preguntas abiertas y evidencia.

```text
Estoy listo para construir esta feature?
```

Resultado esperado:
- revision de specs,
- trazabilidad a RF/HU,
- si hay UI, revision de prototipo y `gate-spdd-approved`,
- gate `gate-4-6`,
- bloqueantes si faltan pruebas, tareas o criterios de aceptacion.

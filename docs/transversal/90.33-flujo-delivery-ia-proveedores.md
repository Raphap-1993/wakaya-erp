## Objetivo
Definir el flujo operativo que debe seguir un proveedor IA desde una idea o spec hasta codigo revisado, probado y listo para release.

Este documento complementa las fases `0-8`. No reemplaza SDD ni QA; endurece el tramo donde los agentes suelen perderse: planificar tareas pequenas, trabajar en una rama limpia, ejecutar TDD, pedir revision y cerrar correctamente.

## Flujo canonico
```text
idea / RF / HU / spec
  -> Product Design
  -> SPDD inicial si hay experiencia visual
  -> /prototype HTML5 o Penpot
  -> gate-prototype-ready
  -> validacion humana
  -> gate-spdd-approved
  -> brainstorming
  -> spec aprobada
  -> writing-plans
  -> using-git-worktrees
  -> executing-plans
  -> test-driven-development
  -> requesting-code-review
  -> QA / gate 4-6
  -> finishing-development-branch
  -> ship / gate 7-8
```

## Contrato para proveedores IA
Antes de ejecutar una feature, el proveedor IA debe poder responder:
- que problema resuelve,
- que alternativa fue elegida y por que,
- que specs usa,
- si aplica frontend, que prototipo, wireframe y mapping debe respetar,
- si el prototipo es HTML5, Penpot u otra evidencia aprobada,
- si el prototipo cubre criterio minimo UX: flujo extremo a extremo, estados, validaciones, roles/permisos, datos mock, navegacion y feedback,
- si `gate-prototype-ready` esta aprobado antes de pedir validacion humana,
- si aplica feature visual, que evidencia aprueba `gate-spdd-approved`,
- que tareas concretas ejecutara,
- que archivos puede modificar,
- que pruebas debe escribir primero,
- que comando valida cada tarea,
- cuando debe pedir revision,
- que evidencia deja para el gate,
- como se cierra la rama o worktree.

## Regla de tareas
Cada tarea en `spec-tareas.md` debe ser pequena, concreta y verificable. El formato minimo es:

```md
## T-001 - Nombre

Objetivo:

Entradas:

Archivos permitidos:

Ciclo TDD:
1. Red:
2. Green:
3. Refactor:

Comandos de verificacion:

Evidencia esperada:

Trazabilidad:
```

## Gates reforzados
### Gate prototype-ready
Antes de pedir aprobacion SPDD, debe verificar:
- prototipo HTML5/Penpot revisable,
- ruta, link/export o evidencia registrada en `prototype.md`,
- `prototype-validation.md` preparado para revision humana,
- criterio minimo UX cubierto: flujo extremo a extremo, estados, validaciones, roles/permisos, datos mock, navegacion y feedback,
- limitaciones y decisiones pendientes declaradas.

### Gate 4-6
Ademas de specs, codigo y QA, debe verificar:
- tareas con IDs estables,
- worktree o rama trazable si hubo codigo,
- evidencia red-green-refactor o justificacion,
- revision de codigo en tareas criticas,
- resultado de pruebas por tarea o slice.

### Gate 7-8
Ademas de runbook, rollback y monitoreo, debe verificar:
- rama lista para PR/merge,
- resumen de cambios y riesgos residuales,
- evidencia QA enlazada,
- worktree limpiado o marcado para conservar.

## Red flags
- El proveedor IA recibe una instruccion como "implementa backend" sin tareas verificables.
- No hay ruta canonica para el resultado.
- El plan no dice que pruebas escribir.
- El agente modifica archivos fuera de ownership.
- Se cierra una tarea sin review cuando cambia seguridad, datos, contratos o UX critica.
- Se propone merge sin QA ni rollback.
- Se salta `/prototype` en una feature visual sin excepcion aprobada.
- Se marca `gate-prototype-ready` sin ruta, link/export o prototipo navegable revisable.
- Se cierra SDD o se implementa front/back de feature visual sin `gate-spdd-approved`.

## Modos de trabajo para proveedores IA - Instanciacion vs Construccion

> Esta seccion es critica. La confusion entre estos dos modos es la causa mas frecuente de
> que un proveedor IA genere residuos de dominio, omita la capa IA o inicie construccion
> productiva antes de tiempo.

### Modo A - Instanciacion (antes de gate-spdd-approved)

**Cuando aplica:** el proveedor IA recibe el template y una fuente de negocio bruta, o datos minimos claros, para crear un proyecto real.

**Objetivo:** documentar, estructurar y orientar. NO construir codigo de dominio todavia.

**Flujo:**
```text
1. Leer AGENTS.md, 90.13, 90.32 y el runbook de agente.
2. Decidir modo:
   - fuente bruta -> arranque-desde-fuente-bruta.md
   - datos minimos -> crear-proyecto-real-desde-template.md
   - guia completa -> crear-proyecto-real-con-agente.md
3. Extraer dominio, actores, RF/RNF, supuestos y preguntas abiertas.
4. Generar config y plan de instanciacion.
5. Ejecutar create-project --db <destino>/.agent/ai-framework.db.
6. Completar artefactos por fase (docs/fase-0 a docs/fase-8).
7. Copiar capa IA completa: commands, skills, agents, prompts, references y gates.
8. Crear Product Design y SPDD inicial si hay feature visual.
9. Generar prototipo HTML5/Penpot con /prototype cuando aplique.
10. Declarar gates con estado real: gate-prototype-ready, gate-spdd-approved, gate-4-6, gate-7-8.
11. Generar ai/provider-orientation-pack.md.
12. Ejecutar validaciones (check-docs, check-template-instantiation).
```

**Regla de bloqueo:** No avanzar a Modo B hasta que gate-spdd-approved este APROBADO.

**Senales de que estas en Modo A:**
- No existe prototipo validado en `specs/<feature>/prototype-validation.md`.
- `gate-spdd-approved` figura como PENDIENTE en la tabla de gates.
- El proyecto fue creado en la sesion actual desde una fuente bruta o por bootstrap inicial.

### Modo B - Construccion (despues de gate-spdd-approved)

**Cuando aplica:** el proveedor IA retoma un proyecto ya instanciado, con SPDD aprobado.

**Objetivo:** construir codigo trazable desde specs aprobadas.

**Flujo:**
```text
1. Leer ai/provider-orientation-pack.md del proyecto.
2. Verificar estado de gates.
3. Confirmar que gate-spdd-approved esta APROBADO.
4. Leer spec-funcional.md, spec-tecnica.md, spec-tareas.md de la feature.
5. Seleccionar una tarea pequena de spec-tareas.md.
6. Preparar worktree o rama.
7. Ejecutar ciclo TDD: red, green, refactor.
8. Pedir /review si cambia contrato, seguridad, datos o UX critica.
9. Ejecutar /test para evidencia QA.
10. Cerrar rama y gate 4-6.
```

**Senales de que estas en Modo B:**
- `specs/<feature>/prototype-validation.md` existe y esta aprobado.
- `gate-spdd-approved` figura como APROBADO con fecha y firmante.
- Existen `spec-funcional.md`, `spec-tecnica.md` y `spec-tareas.md` completos.

### Tabla de decision de modo

| Condicion | Modo correcto | Accion |
|---|---|---|
| gate-spdd-approved = PENDIENTE | Modo A | Solo documentar, no construir |
| gate-prototype-ready = PENDIENTE | Modo A | Completar prototipo con `/prototype` |
| gate-spdd-approved = APROBADO + specs cerradas | Modo B | Construir con TDD |
| No existe provider-orientation-pack.md | Modo A | Generarlo antes de continuar |
| Specs tienen carpetas de dominio anterior | Modo A | Limpiar dominio primero |
| ai/commands/ solo tiene README | Modo A | Copiar command files antes de continuar |

### Red flags de confusion de modo
- Iniciar `/build` con gate-spdd-approved PENDIENTE.
- Saltar `gate-prototype-ready` y pedir SPDD aprobado sin prototipo revisable.
- Cerrar `spec-funcional.md` sin prototipo validado para features visuales.
- Aceptar prototipo HTML5 sin flujo, estados, validaciones, roles/permisos, datos mock, navegacion y feedback UX.
- Omitir la capa IA (commands, skills) porque "es solo documentacion".
- Asumir que gates anteriores estan aprobados sin verificar `provider-orientation-pack.md`.
- Dejar residuos del caso canonico del template en el proyecto real.

## Objetivo
Definir la relacion correcta entre Product Design, SPDD, SDD y construccion para que el proyecto tenga algo validable y mostrable antes de construir front/back.

## Relacion correcta
| Momento | Decide o valida | Salida | Gate |
|---|---|---|---|
| Product Design | que experiencia necesita el usuario | claridad de producto y flujo | `gate-ux-ready` |
| Prototipo | materializar esa experiencia en HTML5, Penpot o wireframe validable | prototipo revisable | `gate-prototype-ready` |
| SPDD | cerrar esa experiencia mediante spec inicial + prototipo validado | validacion UX y criterios UI | `gate-spdd-approved` |
| SDD | como se construye lo aprobado | specs funcionales, tecnicas, contratos y tareas | `gate-4-6` |
| Construccion | implementacion front + back | codigo, pruebas y evidencia | `gate-4-6` |

## Flujo completo
```text
RF / HU / notas de negocio
  -> /ux
  -> Product Design
  -> SPDD - spec inicial
  -> /prototype --mode html5 para validacion rapida
  -> /prototype --mode penpot si aplica diseno formal
  -> Prototype Validation
  -> gate-spdd-approved
  -> /spec
  -> SDD - spec funcional + tecnica + api-contract + tareas
  -> /build
  -> Construccion Front + Back
  -> /review + /test
  -> gate-4-6
```

## Regla principal
`/ux` no es un subproducto de `/document`. `/document` formaliza informacion; `/ux` produce Product Design y SPDD para tener algo que mostrar antes de construir.

Ninguna feature visual pasa a SDD final ni construccion sin prototipo validado o decision explicita de no requerir prototipo.

Para velocidad, el framework recomienda HTML5 primero. Para enterprise, Penpot sigue siendo recomendado cuando se requiere diseno formal, colaboracion UX, design system o handoff visual.

## Salidas por feature

## Contrato para proveedores IA
Antes de cerrar SDD o construir, el proveedor IA debe poder responder:
- que RF/HU alimenta la pantalla,
- que prototipo o wireframe debe respetar,
- quien aprobo o acepto observaciones del prototipo,
- que componente Penpot mapea a que componente Angular,
- que tarea de `spec-tareas.md` ejecutara,
- que contrato backend nace de campos, acciones, validaciones, permisos o errores de UX,
- que prueba red escribira,
- que comando verifica la tarea,
- que evidencia dejara para QA y review.

## Vista de capas del framework
SPDD no es una fase rigida adicional; es una practica transversal entre UX y SDD. Las cuatro capas del framework son:

```text
Capa 1 — Discovery / Product Design
  Objetivo: decidir que experiencia necesita el usuario.
  Artefacto principal: product-design.md por feature.
  Gate: gate-ux-ready.

Capa 2 — SPDD
  Objetivo: validar esa experiencia con spec inicial y prototipo.
  Artefactos: spdd-frontend.md, prototype.md, prototype-validation.md, ui-test-cases.md, prototype-html5/ si aplica.
  Gate: gate-prototype-ready para revisar prototipo; gate-spdd-approved para aprobar SPDD.

Capa 3 — SDD
  Objetivo: convertir lo aprobado en specs tecnicas, contratos y tareas ejecutables.
  Artefactos: spec-funcional.md, spec-tecnica.md, api-contract.md, spec-tareas.md, traceability.md.
  Gate: gate-4-6 (entrada).

Capa 4 — Construccion / QA / Deploy
  Objetivo: implementar con evidencia, tests y trazabilidad.
  Artefactos: codigo, pruebas, evidencia QA, runbook, release.
  Gate: gate-4-6 (salida).
```

Estas capas se mapean sobre las fases 0-8 del framework, no las reemplazan.

## Nota de ubicacion de artefactos SPDD
La practica SPDD se define en [Fase 2 UX/UI](../fase-2-ux-ui/02.10-spdd-spec-prototype-driven-development.md), pero los artefactos por feature viven en `specs/<feature>/` porque alimentan SDD directamente. Las plantillas para todos estos artefactos estan en `plantillas/fase-4-sdd/`.

> **Por que existe este documento**: el comando `harvest-trace` y los validadores
> `check-trace-drift` / `check-trace-coverage` necesitan saber donde un RF se
> implementa en el codigo. La unica forma confiable es que el codigo lleve un
> marcador `@trace RF-XX` (o equivalente) en docstrings/comentarios. Sin este
> marcador, la trazabilidad RF -> codigo queda en `planned` aunque el archivo
> exista.
>
> Verificado contra 3 instanciaciones reales: 1/3 agentes (codex) genero
> backend/frontend completos sin un solo `@trace`. La memoria del agente no
> podia conectar las clases con los RF de las specs.

## Patrones aceptados por `harvest-trace`

El regex que cosecha es: `@(trace|covers|implements)\s+(RF-\d+|RNF-\d+|HU-\d+)`.

Funciona en CUALQUIER tipo de comentario (line-comment, block-comment, docstring,
JSDoc, etc.) en CUALQUIER lenguaje. Lo unico que importa es que el archivo sea
texto plano que `harvest-trace` pueda leer.

## Regla minima por feature

**Cada RF/RNF declarado en `traceability.md` debe tener al menos un `@trace RF-XX`
en algun archivo de codigo cuando la feature pase de fase 4 (SDD) a fase 5
(Construccion).**

- Antes de fase 5: la columna `Codigo` de la matriz es `-` y `@trace` es opcional.
- Fase 5+: si la columna `Codigo` tiene un valor, el archivo referenciado DEBE
  llevar `@trace` para que `check-trace-coverage` lo cuente como `implemented`.

## Regla para componentes visuales (React / Vue / Angular / Svelte) — v12.49

**Por que importa**: en componentes visuales hay multiples opciones (componente principal, hooks, handlers, sub-componentes). Sin regla explicita, agentes IA ponen `@trace` en cualquier lado y `check-test-documented` no encuentra la conexion.

**Regla obligatoria**:

1. **Top del archivo** (siempre): JSDoc del `default export` o componente principal.
2. **Hooks/handlers internos**: NO anotar (la anotacion del componente principal cubre todo el archivo).
3. **Sub-componentes en otros archivos**: cada archivo lleva su propio `@trace` en su default export.
4. **Tests del componente**: archivo de test tambien lleva `@trace` en su top (incluso si es el mismo RF).

```typescript
// React — patron canonico
/**
 * Pagina de bandeja de expedientes.
 * @trace RF-02
 * @covers RNF-01
 */
export default function BandejaPage() {
  // No anotar hooks/handlers internos; el componente cubre el archivo.
  const [filters, setFilters] = useState({});
  function handleSearch() { /* ... */ }
  return <main>...</main>;
}
```

```vue
<!-- Vue 3 — patron canonico -->
<script setup lang="ts">
/**
 * @trace RF-02
 * @covers RNF-01
 */
import { ref } from 'vue'
// ...
</script>
```

```typescript
// Angular — patron canonico
/**
 * @trace RF-02
 * @covers RNF-01
 */
@Component({ selector: 'app-bandeja', templateUrl: './bandeja.component.html' })
export class BandejaComponent {
  // Metodos internos: no anotar.
}
```

```html
<!-- Svelte — patron canonico -->
<script lang="ts">
  /**
   * @trace RF-02
   * @covers RNF-01
   */
  import { onMount } from 'svelte';
  // ...
</script>
```

**Regla para tests de componentes**:

```typescript
// bandeja.test.tsx — siempre @covers en el top.
/**
 * @covers RF-02
 */
import { render, screen } from '@testing-library/react';
import BandejaPage from './BandejaPage';

test('lista expedientes filtrados', () => {
  // No anotar test individual; el archivo cubre todo el suite.
  // Excepcion: si el archivo cubre multiples RFs, anotar cada test:
  //   // @covers RF-02
  //   test(...)
  //   // @covers RNF-01
  //   test(...)
});
```

## Donde poner el `@trace`

| Capa | Donde anotar | Por que |
|---|---|---|
| Backend controller | Clase + cada metodo de endpoint | Conecta RF con endpoint expuesto |
| Backend service | Clase + metodo de regla de negocio | Conecta RF con logica de dominio |
| Backend repository | Clase + metodo de query principal | Conecta RNF con perf real |
| Backend test | Clase + cada metodo `@Test` (`@covers RF-XX`) | `check-test-documented` lo valida |
| Frontend component | Top del archivo (JSDoc) | Conecta RF con UI visible |
| Frontend test | Top del archivo (`// @covers RF-XX`) | `check-test-documented` lo valida |
| SQL migracion | Top del archivo + indices criticos | Conecta RNF con perf SQL |
| OpenAPI YAML | Antes del path | Conecta RF con contrato API |

## Comando de cosecha + validacion

```bash
# Cosecha los @trace del codigo a la BD del agente.
npm run memory:harvest-trace

# Verifica que no haya RF con codigo pero sin @trace.
node ci/scripts/check-trace-coverage.mjs --root .

# Consulta a la BD: que RFs estan implementados (codigo+test reales).
npm run memory:query -- --preset rf-implemented

# Que RFs estan planned (declarados pero sin @trace en codigo).
npm run memory:query -- --preset rf-planned
```

## Errores comunes (vistos en proyectos reales)

| Error | Como evitar |
|---|---|
| `@trace` ausente en clases que implementan RFs | Agregar al menos uno por clase de servicio/controller |
| `@trace RF-2` (sin cero) en vez de `@trace RF-02` | Usar formato `RF-NN` con zero-padding |
| `// trace RF-02` (sin arroba) | El regex exige `@trace` o `@covers` o `@implements` |
| Anotar solo en docstrings y no en metodos especificos | Anota en metodos especificos para granularidad por funcion |
| Olvidar `@covers RF-XX` en tests | Los tests SI deben anotar — `check-test-documented` los exige cuando feature pasa fase 6 |

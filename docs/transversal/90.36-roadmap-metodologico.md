### Estado visual del prototipo — semaforo de 5 peldaños (v12.62)

`npm run roadmap:status` calcula, por feature con prototipo, en que peldaño esta.
Es una escalera: cada peldaño exige el anterior. La fuente unica es
`ci/scripts/_lib/prototype-state.mjs` (tambien expuesta en el front embebido y en
`ROADMAP_STATE.json`).

| Peldaño | Semaforo | Significado |
|---|---|---|
| `exists` | 🔴 | Existe `prototype-html5/index.html`, pero no alcanza calidad automatica nivel 2. |
| `auto-quality` | 🔴 | Pasa calidad nivel 2, pero falla anti-trampa (fixtures ocultos, sin golden, estados solo-texto). |
| `visible-product` | 🟡 | Pasa anti-trampa, pero AUN no tiene seccion `## Revision visual humana`. |
| `human-review-pending` | 🟡 | Tiene la seccion, sin aprobar (Resultado pending/blocked). |
| `human-approved` | 🟢 | Revision humana aprobada por un humano real + Fecha + Evidencia revisada. |

**Regla de avance fase 2 → 3 (no negociable)**: una feature con prototipo NO
avanza a fase 3 (Arquitectura) hasta que su estado sea `human-approved`. El
`gate-prototype-human-visual-review` solo lo firma un humano real — `check:prototype-visible-product`
bloquea cualquier `Resultado: approved` firmado por agente/IA. `roadmap:status`
reporta `avance 2→3 BLOQUEADO` mientras quede algun prototipo sin firmar.

**Como saber si esta fase esta completa**: todas las features visuales tienen sus 9 archivos canonicos + nivel 2 minimo + estado `human-approved` (prototype-validation.md firmado por humano) + `gate-spdd-approved`.

## Contrato de ejecucion para agentes (v12.63)

El roadmap dejo de ser solo "sugerencia": `roadmap:next` emite un **contrato de
ejecucion** completo (derivado de `ci/scripts/_lib/phase-contracts.mjs`) y dos
comandos lo hacen operativo y auditable:

```bash
# 1. Obtener el contrato de la siguiente tarea segura (JSON)
npm run roadmap:next

# 2. Renderizarlo como prompt listo para un agente (Codex/OpenCode/Claude/Gemini)
npm run roadmap:prompt -- --agent codex            # > stdout
npm run roadmap:prompt -- --out ai/prompts/next.md  # ademas a archivo

# 3. DESPUES de trabajar, auditar que el agente respeto el contrato
npm run roadmap:audit                               # working tree vs tarea recomendada
npm run roadmap:audit -- --feature 002-... --base origin/main

# 4. Verificar que ROADMAP_STATE.json no esta stale
npm run check:roadmap-state
```

`roadmap:next` ahora incluye, ademas de next_action/blockers/exit_criteria:

| Campo | Significado |
|---|---|
| `recommended_agent` / `allowed_agents` | Perfil de agente que debe tomar la tarea (derivado del `role` de la fase). |
| `touch_policy.allowed_paths` / `forbidden_paths` | Rutas que el agente PUEDE / NO puede tocar (con `<feature>` interpolado). **Enforzable** por `roadmap:audit`. |
| `definition_of_done` | `deliverables` + `checks` de la fase (no se duplica: es el contrato reusado). |
| `transition_request` | Gate que el agente puede **solicitar** (`canApprove: false` — lo firma un humano). |
| `agent_readiness` | Enum ampliado: `ready_for_ai`, `ready_for_ai_with_constraints`, `needs_human_approval`, `blocked_by_dependency`, `blocked_by_missing_context`, … con `reason`. |

**`roadmap:audit` falla (exit 1)** si el diff toca un `forbidden_path`, o si un
gate quedo `approved` firmado por un revisor no-humano (agente/IA) — cierra el
lazo del anti-self-approval. Las ediciones fuera de `allowed_paths` salen como
warning (revisar alcance), no como violacion.

## Agentes en paralelo (multiagente, v12.76)

Para correr varios agentes IA a la vez sin que dos tomen la misma tarea:

```bash
# 1. Cada agente RECLAMA una feature antes de trabajarla (lock cooperativo con TTL)
npm run roadmap:claim -- --feature 002-cambio-estado-expediente --agent codex

# 2. roadmap:next con --agent SALTA las features tomadas por OTRO agente
npm run roadmap:next -- --agent claude     # devuelve la siguiente feature LIBRE

# 3. Al terminar, libera
npm run roadmap:release -- --feature 002-cambio-estado-expediente --agent codex

# Tablero de quien hace que (autogenerado por roadmap:sync)
npm run roadmap:sync   # regenera AGENT_BOARD.md
npm run roadmap:release -- --prune   # libera locks expirados (huerfanos)
```

- Los locks viven en `ai/locks/<feature>.lock.json` (gitignored) con **TTL** (default 4 h): si un agente muere, el lock se auto-libera al expirar.
- `roadmap:next --agent X` solo recomienda features **no tomadas por otro**; si todas estan tomadas, devuelve `needs_human_input` con el detalle.
- `roadmap:audit` reporta locks expirados/huerfanos como warning.
- `AGENT_BOARD.md` (gitignored, regenerable) muestra por feature: fase, estado de prototipo y lock activo.
- Es coordinacion **cooperativa** (no exclusion fuerte): los agentes deben respetar el protocolo claim → trabajar → release. El `touch_policy` + `roadmap:audit` siguen siendo la red de seguridad si dos tocan lo mismo.

## Preparar un release (v12.78)

`ROADMAP_STATE.json` y `AGENT_BOARD.md` son **regenerables** (gitignored, excluidos del ZIP):
no se versionan ni se empaquetan, se generan en cada proyecto con `roadmap:sync`.

Flujo de release en un comando:

```bash
npm run release:prep
#  = roadmap:sync (estado + docs vivos + AGENT_BOARD)
#  + memory:sync  (reindexa la memoria, que queda stale tras roadmap:sync)
#  + check:all
#  + roadmap-sync --check --strict   (ROADMAP_STATE al dia, BLOQUEANTE)
```

- `check:roadmap-state` solo es **warning** por defecto (no rompe pipelines ni clones frescos sin el archivo); **bloquea** con `--strict` / `CHECK_STRICT=1`.
- `pre-flight-gate` en un **proyecto instanciado** exige ROADMAP_STATE al dia + `check:release-binding` + `check:runbook-binding` en strict; en el **template** quedan como warning.

## Reglas del roadmap (no negociables)

1. **No avanzar a fase N+1 sin gate de salida de fase N aprobado**.
2. **Cada gate requiere validacion humana explicita** (no asumir aprobacion). El agente puede SOLICITAR el gate (`transition_request`), nunca aprobarlo.
3. **Respetar `touch_policy`**: no tocar rutas prohibidas de la fase actual. `roadmap:audit` lo verifica.
4. **Si saltas una fase, documentalo en un ADR** (`docs/fase-3-arquitectura/adr/`).
5. **Las fases son por proyecto Y por feature** — una feature puede estar en fase 5 mientras otra esta en fase 2.
6. **Memoria del agente refleja el estado real**: `npm run memory:sync` antes de declarar estado.

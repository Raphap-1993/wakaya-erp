> **Que es esto**: el "contrato" que define para cada fase del roadmap QUE
> puede hacer un agente IA, que NO puede hacer, que debe leer antes, que debe
> actualizar, que debe validar y que debe entregar.
>
> **Por que existe**: el roadmap dice "donde estoy" pero el agente IA tambien
> necesita "que puedo hacer aqui sin romper el metodo". Esta es la guia
> operativa que convierte el roadmap en plan de accion seguro.
>
> **Fuente unica de verdad**: `ci/scripts/_lib/phase-contracts.mjs`.
> Esta doc se considera consumo del agente; el codigo es la fuente.
>
> **Consumido por**:
>   - `npm run roadmap:next` (incluye contract de la fase actual)
>   - `npm run roadmap:prompt` (renderiza el contrato como prompt para un agente)
>   - `npm run roadmap:audit` (verifica el git diff vs el contrato â€” touch_policy + gates)
>   - Panel embebido > pestana Roadmap > click en phase-card
>   - `npm run check:phase-contract` (valida feature cumple contrato)
>   - API `/api/roadmap/contract/<phase>` y `/api/roadmap/contract`

## Como leer cada contrato

Cada fase tiene 6 dimensiones:

| Dimension | Que significa |
|---|---|
| **âś“ Puede hacer** | Acciones permitidas en esta fase sin requerir aprobacion humana adicional. |
| **âś— NO puede hacer** | Acciones prohibidas â€” si un agente las hace, rompe el metodo. |
| **đź“– Debe leer** | Archivos que el agente DEBE leer antes de actuar (no opcional). |
| **âśŹ Debe actualizar** | Archivos que el agente actualizara como resultado de su trabajo. |
| **đź”Ť Debe validar** | Comandos de validacion que el agente debe ejecutar antes de cerrar. |
| **đź“¤ Debe entregar** | Lista de evidencias que justifica avanzar a la siguiente fase. |

Solo HUMANOS aprueban gates. El agente prepara evidencia + ejecuta validaciones; el humano decide.

### Dimensiones enforzables (v12.63)

Ademas de las 6 dimensiones en prosa, cada fase declara campos **enforzables por maquina**:

| Campo | Que significa | Quien lo usa |
|---|---|---|
| `role` / `allowedAgents` | Perfil de agente recomendado/permitido para la fase. | `roadmap:next.recommended_agent` |
| `touchAllow` / `touchForbid` | Globs de rutas que el agente PUEDE / NO puede tocar (con `<feature>` interpolado). | `roadmap:audit` (git diff vs policy) |
| `transition` | Gate que el agente puede **solicitar** (`canApprove: false`). | `roadmap:next.transition_request` |

La `definition_of_done` se DERIVA de `debeEntregar` + `debeValidar` (no se duplica).
`roadmap:audit` falla si el diff toca un `touchForbid` o si un gate quedo `approved`
firmado por un revisor no-humano (agente/IA).

## Como un agente debe usar este contrato

```bash
# 1. Saber donde estoy y que hacer:
npm run roadmap:next

# 2. El JSON devuelto incluye 'phase_contract' con las 6 dimensiones de la fase actual.
# 3. Antes de cambiar archivos, leer los listados en 'debeLeer'.
# 4. Despues de editar, ejecutar los comandos en 'debeValidar'.
# 5. Verificar que la salida coincide con 'debeEntregar'.
# 6. Antes de cerrar sesion:
npm run pre-flight-gate

# 7. Si exit != 0, NO declarar la sesion completa.
```

## Validador opcional

```bash
npm run check:phase-contract -- --feature 002-mi-feature
```

Verifica que una feature en fase N tenga la evidencia que su contrato exige.

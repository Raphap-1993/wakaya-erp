3.1 Inyección por constructor vs setter

⏱ 28 min Intermedio Última actualización: 12 oct 2024 · 👁 4,287 estudiantes · ★ 4.8
Click para reproducir
▶
04:12 / 28:00

¿Por qué inyectar dependencias?

La inyección de dependencias permite que Spring se encargue de crear y conectar los objetos por nosotros. En lugar de escribir new MiServicio() en cualquier lugar, declaramos qué necesitamos y Spring lo provee. Esto hace el código más testeable, desacoplado y configurable.

Inyección por constructor (recomendada)

Es la forma preferida desde Spring 4.3+. El constructor se anota implícitamente con @Autowired cuando es el único, y las dependencias quedan inmutables (final):

// CustomerService.java
@Service
public class CustomerService {
  private final CustomerRepository repository;
  private final NotificationGateway notifier;

  public CustomerService(CustomerRepository repository, NotificationGateway notifier) {
    this.repository = repository;
    this.notifier = notifier;
  }
}
Tip: al ser final, el compilador garantiza que las dependencias se inyectan en el momento de la construcción. Esto previene NullPointerException tardíos.

Inyección por setter

Útil para dependencias opcionales o cuando hay ciclos legítimos. Anota cada setter con @Autowired:

@Service
public class ReportService {
  private AuditTrail audit;

  @Autowired(required = false)
  public void setAudit(AuditTrail audit) {
    this.audit = audit;
  }
}
Cuidado: los setters dejan la dependencia mutable. Usar solo cuando la inyección por constructor no aplica.

Comparativa rápida

  • Constructor → inmutable, obligatoria, ideal por defecto.
  • Setter → opcional, mutable, útil para ciclos.
  • Field injection (@Autowired directo en el campo) → desaconsejada en código productivo; bloquea pruebas unitarias sin contenedor.

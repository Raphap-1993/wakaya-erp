# v12.16.0 - Memoria viva operacional para agentes IA

`v12.16.0` convierte la memoria local del template de un indice de texto en una
memoria estructurada consultable. El objetivo: que un agente IA continue un
proyecto sin perder contexto ni releer n archivos cada vez.

## Problema que resuelve

La BD SQLite ya indexaba documentos y chunks, pero las cinco tablas estructuradas
(`ai_trace_links`, `ai_gate_runs`, `ai_evidence_items`, `ai_decisions`,
`ai_open_questions`) existian en el schema y estaban vacias. La trazabilidad de
`traceability.md` era demasiado debil para responder "que falta implementar de
RF-02". Y la busqueda era un scan en memoria sin ranking.

## Principio

Markdown = fuente de verdad. BD = indice/memoria reconstruible para acelerar al
agente. Si la BD contradice un Markdown, gana el Markdown y se reindexa.

## Cambios principales

### Busqueda FTS5
- `ai/memory/schema-sqlite.sql` agrega la tabla virtual `ai_chunks_fts` (FTS5,
  `unicode61 remove_diacritics`).
- `index-docs` la mantiene sincronizada con `ai_document_chunks`.
- `search` usa ranking `bm25`; si FTS5 no esta disponible cae a un scan de respaldo.

### Comando `sync-memory`
Nuevo comando que parsea los Markdown oficiales y reconstruye por completo las
cinco tablas estructuradas:
- `ai_trace_links`: desde la matriz de `traceability.md`. Cada celda con dato
  genera una relacion consultable (`RF-02 expone GET /api/expedientes`,
  `RF-02 validado-por ExpedienteQueryTest`, ...).
- `ai_gate_runs`: desde la seccion `## Gates` y menciones de gate en specs.
- `ai_evidence_items`: desde secciones `## Evidencia`, la columna `Evidencia` de
  la matriz y `prototype-validation.md`.
- `ai_decisions`: desde los ADR, los `decisiones-ux.md` y las secciones
  `## Decisiones`.
- `ai_open_questions`: desde las secciones `## Preguntas abiertas`.

### Matriz de trazabilidad ampliada
`traceability.md` pasa de `Origen | Product Design | SPDD | SDD | Construccion |
Prueba` al formato vivo:

```
| RF | HU | UX/SPDD | Prototipo | API | BD | Codigo | Test | Estado | Evidencia |
```

mas una seccion `## Gates`. Actualizado en la plantilla, en la instancia canonica
`specs/001-bandeja-trabajo-expedientes/traceability.md` y en los dos sitios de
emision del generador.

### Indexado y status
- `DEFAULT_DOC_PATHS` incorpora `specs/`, `qa/` y los cuatro archivos de contexto
  vivo (`AI_CONTEXT.md`, `PROJECT_MAP.md`, `GLOSSARY.md`, `TRACEABILITY_MATRIX.md`).
- `status` reporta el estado de FTS5 y las siete tablas.
- El modulo `ai-framework-agent.mjs` solo ejecuta `main()` cuando se invoca
  directamente, lo que permite reutilizar sus helpers desde tests.

## Flujo de memoria viva

```text
Markdown vivos -> index-docs  -> ai_documents + ai_document_chunks + ai_chunks_fts
              -> sync-memory  -> ai_trace_links + ai_gate_runs + ai_evidence_items
                                 + ai_decisions + ai_open_questions
              -> search / SQL -> el agente consulta y abre solo los .md relevantes
```

## Validacion

Sobre el propio template:
- `index-docs`: 260 documentos indexados, FTS5 activo.
- `sync-memory`: 24 trace links, 3 gate runs, 29 evidence items, 18 decisions,
  3 open questions.
- `search` con FTS5 devuelve resultados ranqueados por `bm25`.
- `check-docs`: 504 archivos markdown sin hallazgos.

## Alcance de esta entrega

Esta release cubre la **capa de datos** (Nivel 1). Quedan para una segunda pasada:
- Los cuatro archivos de contexto vivo como plantilla + instancia canonica
  (`AI_CONTEXT.md`, `PROJECT_MAP.md`, `TRACEABILITY_MATRIX.md`, `GLOSSARY.md`).
- El front embebido de consulta (generador HTML estatico + servidor Node live).
- Activacion de `sqlite-vec` para busqueda semantica.

## Reglas

- Markdown es la fuente de verdad; la BD es reconstruible con `index-docs` + `sync-memory`.
- Las cinco tablas estructuradas son derivadas: se vacian y repueblan en cada `sync-memory`.
- `framework-agent.db` no se versiona: es memoria local reconstruible.

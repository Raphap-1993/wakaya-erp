# v12.17.0 - Contexto vivo y front embebido de memoria

`v12.17.0` completa la segunda pasada de la memoria viva: los archivos de
contexto que un agente IA lee primero para retomar un proyecto, y un front
embebido para consultar la memoria sin escribir SQL.

## Problema que resuelve

La capa de datos (v12.16.0) ya poblaba trazabilidad, gates, decisiones y
preguntas abiertas. Faltaban dos piezas para que la memoria fuera operativa:
1. Un punto de entrada claro: el agente abria el repo y no sabia en que estado
   estaba el proyecto ni donde buscar.
2. Una forma humana de revisar la memoria sin consultar SQLite a mano.

## Cambios principales

### Archivos de contexto vivo
Cuatro Markdown en la raiz del proyecto, en orden de lectura para un agente IA:
- `AI_CONTEXT.md` — en que estado esta el proyecto ahora: fase activa, features y
  su estado, gates pendientes, proximos pasos, como cargar contexto rapido.
- `PROJECT_MAP.md` — donde vive cada cosa: arbol de carpetas y rutas canonicas
  por tipo de artefacto.
- `TRACEABILITY_MATRIX.md` — matriz global de trazabilidad (rollup de todas las
  features). `sync-memory` la parsea ademas de las `traceability.md` por feature.
- `GLOSSARY.md` — terminos del framework y del dominio, mas sinonimos no permitidos.

Cada archivo existe en tres formas:
- Plantilla en `plantillas/transversal/`.
- Instancia canonica en la raiz del template.
- Emision del generador: `create-project` los crea en cada proyecto real con el
  dominio interpolado.

`AGENTS.md` explica COMO trabaja el agente; `AI_CONTEXT.md` explica EN QUE ESTADO
esta el proyecto.

### Comando `memory-report` (estatico)
Genera `ai/memory/memory-report.html`: un HTML autocontenido con un snapshot
embebido de la memoria. UI por pestañas (trazabilidad, gates, decisiones,
evidencia, preguntas abiertas, documentos) mas busqueda client-side. Portable,
sin servidor, reconstruible; se ignora en Git.

### Comando `memory-serve` (live)
Levanta un servidor Node local que consulta la BD en vivo:
- `/` — la misma UI por pestañas.
- `/api/snapshot` — JSON con stats y tablas estructuradas.
- `/api/search?q=<texto>` — busqueda FTS5 con ranking.

`memory-report` para compartir o revisar; `memory-serve` para consulta en
caliente durante el desarrollo.

## Validacion

Sobre el propio template:
- `index-docs`: indexa los 4 archivos de contexto vivo nuevos, FTS5 activo.
- `sync-memory`: 64 trace links, 5 gate runs, 38 evidence items, 18 decisions,
  5 open questions (la `TRACEABILITY_MATRIX.md` raiz suma trazabilidad global).
- `memory-report`: HTML autocontenido de ~62 KB generado.
- `memory-serve`: `/`, `/api/snapshot` y `/api/search` responden; 404 para rutas
  desconocidas.
- `check-docs`: 513 archivos markdown sin hallazgos.

## Flujo completo de memoria viva

```text
Markdown vivos (AI_CONTEXT, PROJECT_MAP, TRACEABILITY_MATRIX, GLOSSARY,
                docs/, specs/, ai/, qa/)              <- fuente de verdad
   -> index-docs    -> ai_documents + chunks + ai_chunks_fts (FTS5)
   -> sync-memory   -> trace_links + gate_runs + evidence + decisions + open_questions
   -> search / SQL  -> el agente consulta y abre solo los .md relevantes
   -> memory-report / memory-serve  -> revision humana de la memoria
```

## Reglas

- Markdown es la fuente de verdad; la BD y el HTML de reporte son reconstruibles.
- Los cuatro archivos de contexto vivo se mantienen al cerrar cada fase, feature
  o gate; tras actualizarlos se corre `sync-memory`.
- `framework-agent.db` y `memory-report.html` no se versionan: son memoria local.

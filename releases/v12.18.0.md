# v12.18.0 - Busqueda semantica sin dependencias

`v12.18.0` activa la busqueda semantica de la memoria del agente IA sin acoplar
el template a un proveedor de embeddings ni requerir extensiones nativas.
`sqlite-vec` sigue disponible como acelerador opcional para corpus grandes.

## Problema que resuelve

`v12.16.0` y `v12.17.0` dejaron la memoria estructurada y consultable, pero la
busqueda seguia siendo textual (FTS5). La capa semantica existia en el schema
(`schema-sqlite-vec.sql`, `import-embeddings`, `--semantic`) pero requeria
descargar la extension nativa `sqlite-vec` y un proveedor externo de embeddings.

El template no quiere acoplarse a un proveedor especifico ni obligar a instalar
extensiones nativas.

## Solucion

Tres piezas:

1. **Tabla `ai_chunk_embeddings`** (regular SQLite, sin extension). Guarda el
   vector como JSON con `model` y `dim`. Funciona para cualquier dimension y
   cualquier proveedor.
2. **Embedder local determinista** (`local-hash-v1`, dim 256). Combina hashing
   de tokens y trigramas de caracter en un vector L2-normalizado. Cero
   dependencias, reproducible. Encuentra chunks que comparten vocabulario y
   morfologia.
3. **Cosine en JavaScript** sobre la tabla regular. Suficiente para el orden
   de magnitud del template (~2500 chunks, sub-segundo).

`sqlite-vec` queda como **acelerador opcional** documentado para cuando el
corpus crece lo suficiente.

## Cambios principales

### Schema
- Nueva tabla `ai_chunk_embeddings` en `ai/memory/schema-sqlite.sql`.
- `schema-sqlite-vec.sql` aclara que es acelerador opcional, no fuente principal.

### Comando `embed-docs`
```sh
node scripts/ai-framework-agent.mjs embed-docs            # incremental
node scripts/ai-framework-agent.mjs embed-docs --force    # regenera todo
```
Genera embeddings locales para todos los chunks indexados.

### `search --semantic` reconfigurado
- Por defecto: embedder local sobre `--query` + cosine en JS.
- Con `--embedding <vector>`: usa un vector de proveedor externo (de cualquier
  dimension); cosine en JS sobre embeddings de la misma dim.
- Con `--sqlite-vec-extension <ruta>` + `--embedding`: usa la extension nativa
  vec0 como acelerador (camino preservado, opcional).

### Proveedores externos
`import-embeddings` sigue siendo la ruta para cargar vectores de OpenAI, Cohere
u otros. JSONL por linea: `{"chunkId":1, "embedding":[...], "model":"..."}`.

### `status` extendido
Reporta `Embeddings` ademas de las siete tablas anteriores (8 metricas totales).

## Validacion

Sobre el propio template:
- `embed-docs`: 2499 chunks embebidos en menos de un segundo, modelo
  `local-hash-v1`, dim 256.
- `embed-docs` incremental: 0 nuevos cuando ya estan generados.
- `embed-docs --force`: regenera 2499.
- `search --query "..." --semantic`: retorna chunks ranqueados por cosine, con
  scores entre 0 y 1. Para "validar prototipo antes de construir" devuelve
  el chunk del objetivo de SPDD y la skill html5-prototyping como top 2.
- `status`: reporta 265 documentos, 2499 chunks, 2499 embeddings, FTS5 activo.
- `check-docs`: 514 archivos markdown sin hallazgos.

## Flujo completo de la memoria viva

```text
Markdown vivos
  -> index-docs    -> ai_documents + chunks + ai_chunks_fts (FTS5)
  -> sync-memory   -> trace_links + gate_runs + evidence + decisions + open_questions
  -> embed-docs    -> ai_chunk_embeddings (vectores locales o de proveedor)
  -> status        -> 8 metricas
  -> search                -> FTS5 textual con ranking bm25
  -> search --semantic     -> cosine local
  -> memory-report/serve   -> revision humana de toda la memoria
```

## Reglas

- Markdown es la fuente de verdad; la BD y los embeddings son reconstruibles.
- El embedder local no reemplaza a un modelo de lenguaje: encuentra
  similitud lexica y morfologica. Para semantica conceptual real, usa un
  proveedor externo via `import-embeddings`.
- `framework-agent.db` y `memory-report.html` no se versionan.

# v12.19.0 - Memoria viva resistente al rot

`v12.19.0` cierra la propuesta de 5 sprints diseñada para que un agente IA no
pierda contexto al retomar un proyecto. La memoria construida en v12.16-v12.18
ahora se **auto-mantiene** (Sprint 1), **acelera al agente** (Sprint 2), **cierra
el ciclo a codigo real** (Sprint 3), **es mas rica y queryable** (Sprint 4) y se
**sincroniza automaticamente** (Sprint 5).

## Por que esta release

Lo construido antes funcionaba perfecto en la primera lectura — y se atrofiaba
si nadie actualizaba `AI_CONTEXT.md` o si la BD se desactualizaba respecto a los
Markdown. v12.19.0 transforma el sistema de **"vivo si lo mantienes"** a
**"vivo por construccion"**.

## Cambios principales

### Sprint 1 — Anti-rot
- **Schema enriquecido**: `ai_gate_runs` con `actor`, `decided_at`, `evidence_paths`. `ai_decisions` con `tags`, `affects`, `deciders`, `decided_at`. Nueva `ai_session_events`. Migracion ligera idempotente.
- **`SESSION_LOG.md`**: bitacora append-only del agente y el equipo. Plantilla + instancia raiz + emision del generador.
- **Parsers ricos**: `syncGateRuns` agrega autor y fecha desde `git log`; `syncDecisions` parsea frontmatter YAML de ADRs; nuevo `syncSessionEvents`.
- **`regenerate-context`**: reescribe zonas `<!-- auto:start name=X -->` de `AI_CONTEXT.md` desde la BD (features consolidadas, gates pendientes, sesiones recientes, ultima actualizacion). Las secciones humanas se preservan.
- **Freshness indicator**: `status` reporta `STALE` cuando la BD esta atrasada respecto a los Markdown.

### Sprint 2 — Productividad del agente
- **`context-pack --topic <t>`**: bundle JSON con chunks (FTS5+semantico) + trace_links + gate_runs + decisiones + evidencia + preguntas abiertas. Drop-in para el system prompt.
- **`next-task`**: reglas declarativas que devuelven la siguiente tarea desbloqueada (RF con prototipo y sin codigo+test → gate pendiente → pregunta abierta).
- **`diff-since --ref <commit|fecha>`**: cambios en sesiones, gates y decisiones desde un commit o fecha ISO.

### Sprint 3 — Trazabilidad → codigo real
- **`harvest-trace`**: cosecha tags `@trace`, `@implements`, `@covers`, `@fixes` de source code y emite trace_links con confianza 0.8 (inferido).
- **`ci/scripts/check-trace-drift.mjs`**: valida que cada `target_ref` resuelve a un artefacto real (codigo, test, api, bd, prototipo, ...).
- **`ci/scripts/check-trace-coverage.mjs`**: falla cuando un RF/RNF en fase >= 5 no tiene codigo y test.

### Sprint 4 — Memoria mas rica
- **ADR con frontmatter**: parser que extrae `tags`, `affects`, `deciders`, `decided_at`, `status` desde el frontmatter YAML de los ADR.
- **Confidence diferenciada**: declarado (matriz) = 1.0; inferido (harvest-trace) = 0.8. Filtrable en queries.
- **`check-spec-dedup`**: detecta specs semanticamente cercanas (cosine >= 0.85) usando los embeddings locales de v12.18.

### Sprint 5 — Automatizacion
- **`watch`** (3 modos):
  - **default**: `fs.watch` sobre Markdown con debounce 1.5s + banner keep-alive cada 5 min.
  - **`--once`**: una sola sincronizacion y sale (para CI y git hooks).
  - **`--interval 30s|5m|1h`**: polling como alternativa cuando `fs.watch` no es confiable (Docker, editores remotos).
- **`install-hooks`**: instala `.git/hooks/pre-commit` que ejecuta `watch --once` antes de cada commit; no bloquea el commit si falla; `--uninstall` lo remueve. Hace que la memoria se mantenga fresca sin necesidad de tener `watch` corriendo.
- **`template-drift`**: diff entre un proyecto generado y la generacion limpia con su config; reporta `match`, `drift` y `missing` por archivo.

### Fix: alineacion busqueda estatica vs live
- `buildFtsQuery` ahora hace **prefix matching** (`exp*` en vez de `"exp"`). Antes 'exp' encontraba 1 chunk (token literal); ahora encuentra 501 (palabras que empiezan con exp).
- `/api/search` de `memory-serve` usa la nueva funcion `searchAll`, que cubre el **mismo dominio que el modo estatico** (documentos, trace_links, decisiones, gates, preguntas) **y ademas** los chunks FTS5. Presupuesto balanceado: la mitad para chunks, el resto para metadata.
- Resultado: el modo live nunca devuelve menos que el estatico para la misma consulta. Para 'exp' ahora retorna 30 chunks + 26 docs + 4 trace links.

### Consultas predefinidas (memory-query)
Catalogo `MEMORY_QUERY_PRESETS` con 10 preguntas frecuentes deterministas que cubren las preguntas tipicas de un agente IA al retomar un proyecto:

| Preset | Pregunta que responde | Arg |
|---|---|---|
| `docs-for` | ¿Que documentos estan relacionados con RF-XX? | sí (ref) |
| `apis-for` | ¿Que APIs implementan este requerimiento? | sí (ref) |
| `features-pending-qa` | ¿Que features estan pendientes de QA? | no |
| `validated-prototypes` | ¿Que prototipos ya fueron validados? | no |
| `decisions-pending` | ¿Que decisiones estan pendientes? | no |
| `failed-gates` | ¿Que gates fallaron / estan bloqueados? | no |
| `rf-without-code` | ¿Que requerimientos no tienen implementacion? | no |
| `rf-without-test` | ¿Que requerimientos no tienen pruebas? | no |
| `rf-implemented` | ¿Que requerimientos ya estan implementados? | no |
| `decisions-about` | ¿Que decisiones tecnicas se tomaron sobre <X>? | sí (término) |

Disponibles via:
- **CLI**: `node scripts/ai-framework-agent.mjs memory-query --preset <key> [--arg <v>] [--json]`.
- **Endpoint live**: `/api/query?preset=<k>&arg=<v>` (con `/api/presets` para descubrirlos).
- **UI estatica y live**: panel "Consultas rapidas" con botones por preset y un campo de argumento.

### Salida de busqueda mejorada
- Encabezado `N resultados · etiqueta`.
- Chips de desglose por tipo: `chunk: 30 · doc: 26 · trace: 4`.
- Shape uniforme `{t, ref, path, excerpt}` entre estatico y live; ambos muestran excerpt cuando aplica.

## Flujo completo de memoria viva ahora

```text
Markdown vivos                                  fuente de verdad
   |
   |  index-docs    -> ai_documents + ai_document_chunks + ai_chunks_fts (FTS5)
   |  sync-memory   -> trace_links + gate_runs + evidence_items + decisions
   |                   + open_questions + session_events
   |  embed-docs    -> ai_chunk_embeddings (embedder local determinista)
   |  harvest-trace -> trace_links inferidos desde @trace en source
   |
   v
SQLite framework-agent.db                       indice/memoria consultable
   |
   |  status          -> 10 metricas + freshness
   |  regenerate-context -> AI_CONTEXT.md auto-actualizado
   |  context-pack    -> JSON con todo lo relevante a un topic
   |  next-task       -> siguiente accion desbloqueada
   |  diff-since      -> cambios desde un commit/fecha
   |  check-trace-drift / check-trace-coverage (CI)
   |  check-spec-dedup  -> anti-duplicacion semantica
   |  watch           -> sincronizacion automatica al editar
   |
   v
Agente IA arranca en segundos, sin releer n archivos
```

## Reglas

- Markdown sigue siendo la fuente de verdad; la BD es indice reconstruible.
- `regenerate-context` solo toca zonas `<!-- auto:start name=X -->`. Lo humano se preserva.
- Las cinco tablas estructuradas mas `ai_session_events` se vacian y repueblan en cada `sync-memory`.
- `harvest-trace` puede correrse repetidamente: borra solo los links de tipo `source` antes de repoblar.
- Los validadores CI (`check-trace-drift`, `check-trace-coverage`) son no-bloqueantes por defecto; cada proyecto decide si gateara su pipeline con ellos.

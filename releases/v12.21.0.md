# v12.21.0 — `prototype/index.html` estandarizado

`v12.21.0` elimina la última zona del template donde cada agente IA escribía
diferente: el **hub del producto** (`prototype/index.html`). Ahora hay una
estructura fija de **10 secciones**, una plantilla HTML reusable, un comando
autogenerador y un validador.

## Problema que resuelve

Antes, cada proyecto tenía un hub con diferente cantidad de secciones, orden,
densidad y tipo de información. Comparar dos proyectos era imposible. Y los
agentes IA generaban hubs muy distintos: algunos casi vacíos, otros saturados.

El ejemplo externo `plataforma-app-cristiano/prototype/index.html` (1132
líneas) era superior al del template actual (486 líneas) — lo tomamos como
referencia y lo formalizamos.

## Cambios principales

### 1. Estándar de 10 secciones
- **`plantillas/transversal/PROTOTYPE_HUB.md`** documenta cada sección con
  qué información obligatoria/opcional, patrón visual recomendado, y cómo se
  conecta con el resto del framework.

### 2. Plantilla HTML reusable
- **`plantillas/fase-2-ux-ui/prototype-hub.html.tmpl`** trae CSS completo con
  tokens compartidos con los goldens nivel 3, placeholders `{{X}}` para datos
  globales (producto, fecha, contadores) y zonas `<!-- @auto:start name=X -->`
  regenerables.

### 3. Comando `generate-prototype-hub`
```sh
node scripts/ai-framework-agent.mjs generate-prototype-hub
node scripts/ai-framework-agent.mjs generate-prototype-hub --auto-only
```
Lee `specs/`, `decisiones-ux.md`, `traceability.md`, `template.config` y la
BD de memoria, y rellena las 6 zonas auto:
- `@auto:start name=journeys` — una fila por actor con sus specs encadenadas.
- `@auto:start name=specs` — una card por feature con invariantes + status pill.
- `@auto:start name=actors` — una card por actor del producto.
- `@auto:start name=coverage` — matriz SPDD (artefactos × spec).
- `@auto:start name=doclinks` — quick links a `../specs/NNN-feature/`.
- `@auto:start name=stack` — stack tags derivados del template.config.

### 4. Tabla de cobertura híbrida
La tabla principal se regenera desde la BD/repo. El agente puede agregar
tablas adicionales (ADRs, threat-model, controles de seguridad) **fuera** de
las marcas `@auto:start`, sin riesgo de sobrescritura.

### 5. Validador `ci/scripts/check-prototype-hub.mjs`
Verifica:
- Banner SPDD con tags requeridos.
- Hero con `<h1>` + ≥4 stats.
- Spec-cards = N específicos en el repo (cuenta exacta).
- Cada spec-card tiene status pill.
- Tabla de cobertura con ≥7 columnas.
- Footer con stack tags y disclaimer "sin backend real".
- Marcadores `@auto:start` presentes para que el comando funcione.
- Términos prohibidos puramente metodológicos (`Formulario-spec`, `Spec técnica`...).
- **NO bloquea** `RF-NN:` ni `gate-xxx:` en tablas/invariantes — el hub es un
  meta-documento que puede mostrar referencias técnicas. **Sí bloquea** si
  aparecen como heading o spec-title.

### 6. Hub canónico regenerado
`prototype/index.html` del template se regeneró con el comando. Ahora tiene
las 10 secciones, refleja las 4 specs del template (001 bandeja-trabajo, 002
cambio-estado, 003 historial-auditoria, 000-ejemplo) y los 5 actores
detectados desde sus `decisiones-ux.md`.

### 7. Documentación
- `02.15` referencia el estándar de 10 secciones.
- `ai/commands/prototype-command.md` documenta el uso del autogenerador.
- `plantillas/transversal/README.md` lista PROTOTYPE_HUB.md.

## Validación

```
generate-prototype-hub:
  OK · 4 specs · 1 prototipo HTML5 · 5 actores · ⏳ 3 validaciones pendientes

check-prototype-hub:
  Spec-cards en hub:    4
  Specs reales en repo: 4
  Actor-cards:          5
  Journey-rows:         5
  Doc-links:            4
  ✓ Hub cumple las 10 secciones estándar.
```

## Reglas

- El hub se genera con `generate-prototype-hub`, **nunca** a mano.
- Las 6 zonas `@auto:start` se rellenan automáticamente. NO editarlas.
- Cualquier información adicional (tablas, secciones de soporte) va FUERA de
  las marcas auto.
- `--auto-only` regenera solo las zonas auto del hub existente; útil cuando
  el agente quiere conservar customizaciones en zonas no-auto.
- El banner SPDD del hub es el **único banner de "simulación" permitido** en
  el ecosistema (regla v12.20). Los prototipos individuales NO lo llevan.
- El hub PUEDE mostrar `RF-NN:` o `gate-xxx:` como **referencia técnica**
  dentro de tablas/invariantes (es un meta-documento). NO PUEDE mostrarlos
  como heading o spec-title.

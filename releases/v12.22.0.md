# v12.22.0 — Trazabilidad semantica + instanciacion en proyectos reales

`v12.22.0` cierra dos problemas que bloqueaban la adopcion seria del template
en proyectos reales con agentes IA heterogeneos:

1. La matriz reportaba RFs como **"implementados"** aunque el archivo de codigo
   no existiera en el repo (todos los links tenian `confidence=1.0`).
2. No habia una guia clara y un `package.json` canonico para que **cualquier
   agente IA** (Claude Code, Codex, Cursor, Copilot, Gemini Code) supiera como
   instanciar la memoria viva en un repo recien generado.

## El problema (medido sobre el template)

Antes de v12.22, con el corpus del propio template:

```
rf-implemented   → 5 RFs reportados como implementados (FALSO POSITIVO)
check-trace-drift → 18 links a archivos inexistentes (los mismos RFs)
```

Era contradictorio: el preset decia "implementado" y el validador decia "drift".
Causa raiz: `syncTraceLinks` insertaba **todo** lo declarado en la matriz con
`confidence=1.0`, sin verificar si el archivo existia.

## La solucion: `link_status` con 5 estados

`ai_trace_links` gana 3 columnas y un indice por cada una:

```sql
link_status TEXT,    -- planned | implemented | validated | inferred | drift
origin TEXT,         -- markdown-matrix | source-harvest | tool | human
validated_at TEXT    -- ISO date cuando paso a 'validated' (gate aprobado)
```

Semantica:

| `link_status` | Significado                                                       |
|---------------|-------------------------------------------------------------------|
| `planned`     | Declarado en la matriz pero el target_ref **NO existe** en el repo |
| `implemented` | Declarado **Y** existe el archivo/clase/test en el repo           |
| `validated`   | `implemented` **+** Estado de la fila contiene `valid`/`aprob`    |
| `inferred`    | Cosechado desde `@trace`/`@implements` en source (confidence 0.8) |
| `drift`       | Antes existia, ahora ya no                                        |

### Como se calcula

`syncTraceLinks` ahora recibe `root` y llama al nuevo helper
`targetRefExists(root, targetType, targetRef)` por cada celda llenada:

- `codigo`/`test` → busca el nombre en `src/`, `backend/`, `frontend/`,
  `tests/`, `stacks/` por filename o declaracion `class|interface|function|...`
  en multiples lenguajes (Java, TS, Python, Go, Rust, C#, Kotlin).
- `api` → busca la ruta en specs y `contracts/*.yaml`.
- `bd` → busca el nombre de tabla en specs y src.
- `prototipo`/`spdd`/`pantalla` → resuelve el path o lo busca en `specs/`/`docs/`.
- `hu`/`rf`/`rnf`/`estado` → cross-references documentales, siempre `implemented`.

## Cambios adicionales

### Presets endurecidos

- `rf-implemented`, `rf-without-code`, `rf-without-test`: filtran por
  `link_status IN ('implemented','validated')`. Un link planificado ya NO
  cuenta como implementacion real.
- **`rf-validated`** (nuevo): requiere codigo + test con `link_status='validated'`.
- **`rf-planned`** (nuevo): RFs declarados sin artefacto real en el repo.
- **`links-drift`** (nuevo): lista todos los links con `link_status IN ('planned','drift')`.

### `check-trace-coverage.mjs` corregido

Ahora solo cuenta artefactos con `link_status IN ('implemented','validated')`.
Era la fuente del falso positivo: un RF con `Codigo: FooService` (clase que no
existe) pasaba la cobertura como si estuviera implementado.

### Plantilla y emisores actualizados

- `plantillas/fase-4-sdd/traceability.md`: las columnas `Codigo`/`Test`
  empiezan con `-` y la plantilla documenta explicitamente la semantica.
- Los dos emisores de `traceability.md` en `scripts/ai-framework-agent.mjs`
  (lineas 7398 y 8602) generan ahora `-` por defecto + el header de reglas.

### `package.json` raiz con scripts npm canonicos

Nuevo `package.json` en la raiz del template con scripts agente-agnosticos:

```bash
npm run memory:bootstrap       # init-memory + index-docs + sync-memory + embed-docs + regenerate-context
npm run memory:sync            # sync-memory + regenerate-context
npm run memory:query -- --preset rf-implemented
npm run memory:next-task
npm run check:trace-drift
npm run check:trace-coverage
npm run check:all
```

Cualquier agente IA (Claude Code, Codex, Cursor, Gemini, Copilot) puede usar
el mismo flujo sin conocer la ruta del script. Requiere Node 22.x.

### `INSTANCIACION_PROYECTO_REAL.md`

Nueva guia raiz que **cualquier agente IA debe leer primero** al abrir un repo
generado del template. Cubre:

- TL;DR de 3 comandos.
- Los 5 archivos de contexto vivo a leer en orden.
- La semantica `planned`/`implemented`/`validated` con ejemplos.
- Diferencias por agente IA (Claude/Codex/Cursor/Gemini/Copilot).
- Ciclo de cierre de sesion (`SESSION_LOG` + `memory:sync` + `memory:context`).
- Los 4 validadores de CI con su efecto en el merge.
- Errores comunes con su diagnostico.

### Migracion idempotente

`migrateSchema` agrega las 3 columnas nuevas a DBs existentes sin perder datos.
Los DBs preexistentes se actualizan al primer `npm run memory:sync`.

### Memoria viva ahora se emite en proyectos generados

Antes de v12.22, `create-project` emitia los 5 archivos de contexto vivo
(`AI_CONTEXT.md`, `PROJECT_MAP.md`, `TRACEABILITY_MATRIX.md`, `GLOSSARY.md`,
`SESSION_LOG.md`) pero NO emitia ni `package.json` ni la infraestructura
ejecutable (`scripts/ai-framework-agent.mjs`, `ai/memory/schema-sqlite.sql`,
los validadores `ci/scripts/check-*.mjs`). El resultado: la "memoria viva"
del template no funcionaba en el proyecto generado.

Ahora `create-project` copia:

| Archivo                            | Como llega al destino                                              |
|------------------------------------|--------------------------------------------------------------------|
| `AI_CONTEXT.md` + 4 zonas auto     | Plantilla emitida (ya existia + se agregaron los marcadores)       |
| `PROJECT_MAP.md`                   | Plantilla emitida (ya existia)                                     |
| `TRACEABILITY_MATRIX.md`           | Plantilla emitida (ya existia)                                     |
| `GLOSSARY.md`                      | Plantilla emitida (ya existia)                                     |
| `SESSION_LOG.md` con 1ra entrada   | Plantilla emitida (ya existia)                                     |
| **`INSTANCIACION_PROYECTO_REAL.md`** | **Plantilla emitida (v12.22 nuevo)**                              |
| **`package.json` con scripts**     | **v12.22 nuevo, con merge para no pisar stacks**                   |
| **`scripts/ai-framework-agent.mjs`** | **Copiado del template (v12.22 nuevo en la lista)**              |
| **`ai/memory/schema-sqlite.sql`**  | **Copiado del template (v12.22 nuevo en la lista)**                |
| **`ci/scripts/check-*.mjs`**       | **5 validadores copiados del template (v12.22 nuevo)**             |

### Merge inteligente de `package.json`

Para stacks como `node-next` donde Next.js requiere su `package.json` en la
raiz con `dependencies` reales, el generador hace **merge** en lugar de
sobreescribir:

- Si el destino tiene `package.json`: agrega solo los scripts `memory:*` y
  `check:*` que no existan, preserva `name`, `version`, `dependencies`,
  `devDependencies` y scripts existentes (`dev`, `build`, etc.).
- Si no existe: escribe el nuestro tal cual.
- Si existe pero esta corrupto: hace backup a `package.json.pre-v12.22.bak`.

### Cross-link bidireccional con el runbook de creacion

`INSTANCIACION_PROYECTO_REAL.md` (operacion diaria) y
`ai/runbooks/crear-proyecto-real-con-agente.md` (creacion del repo) se
referencian mutuamente y aclaran su momento del ciclo. El runbook ademas
tiene un nuevo "CHECKLIST DE MEMORIA VIVA EN EL PROYECTO REAL" en la fase
de limpieza para que el agente verifique que la capa de memoria llego
completa al destino.

## Reglas (para el agente IA)

- **NO inventes** nombres de clases o tests en la columna `Codigo`/`Test` de
  `traceability.md`. Usa `-` mientras el archivo no exista. Cuando exista,
  reemplaza `-` por el nombre real.
- Antes de declarar un RF como "Implementado" o "Validado" en `Estado`,
  ejecuta `npm run check:trace-coverage` — solo pasa si hay codigo + test
  reales en el repo.
- `npm run memory:query -- --preset links-drift` te da la lista exacta de
  celdas a corregir (nombre inventado o renombre pendiente).
- El preset `rf-validated` te da los RFs realmente listos para release
  (codigo + test + gate aprobado).

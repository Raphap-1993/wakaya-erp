# v12.24.0 — Streaming SSE + boton Detener en el panel de Acciones

`v12.24.0` evoluciona el panel "Acciones" de v12.23 a ejecucion en
**tiempo real**: el stdout/stderr aparece linea por linea en la consola de
la UI mientras el comando corre, y un nuevo boton **Detener** envia
SIGTERM al job en curso. Compatible hacia atras con el modo sync.

## Por que SSE

En v12.23 el endpoint `POST /api/exec` era sincrono: el cliente esperaba a
que el comando terminara para recibir el stdout/stderr completo. Funcionaba
para comandos cortos (status, sync-memory < 5s), pero no para los largos:

| Comando             | Duracion tipica          | Problema sin streaming                            |
|---------------------|--------------------------|---------------------------------------------------|
| `index-docs`        | 1-30s (segun repo)       | Sin feedback durante el procesamiento             |
| `embed-docs`        | 1-60s (segun #chunks)    | Aparenta "colgado"                                |
| `check-trace-drift` | 0.3-2s                   | Aceptable sync                                    |
| `check-docs`        | 2-15s en repos grandes   | Imposible saber por que demora                    |
| `generate-prototype-hub` | 0.5-3s              | Aceptable                                          |

Y mas importante: **no se podia cancelar** un comando que se quedaba
colgado. Habia que ir al terminal del server, encontrarlo y matarlo.

## Que hay nuevo

### 1. Streaming SSE en `POST /api/exec`

El cliente envia `Accept: text/event-stream` y recibe un stream de eventos:

```
event: meta
data: {"actionId":"sync-memory","argv":[...],"pid":1234,"startedAt":...}

event: stdout
data: {"chunk":"OK. Memoria sincronizada (50 archivos)\n"}

event: stdout
data: {"chunk":"  Trace links: 64\n"}

event: exit
data: {"exitCode":0,"signal":null,"durationMs":3201,"timedOut":false}
```

Tipos de evento:
- **`meta`** — `actionId`, argv saneado (paths absolutos reemplazados por `<root>`), pid del child, timestamp ISO.
- **`stdout`** — chunk de stdout (encoding UTF-8).
- **`stderr`** — chunk de stderr (encoding UTF-8).
- **`exit`** — exitCode, signal, durationMs, timedOut.
- **`error`** — mensajes del runtime del server (id desconocido, 409 concurrente, validacion de arg).

### 2. Backwards-compat (modo sync v12.23)

Sin header `Accept: text/event-stream` el endpoint sigue respondiendo
JSON con `{exitCode, stdout, stderr, durationMs, signal, timedOut}` igual
que en v12.23. Cualquier integracion (`curl`, scripts existentes) sigue
funcionando sin cambios.

### 3. Endpoint `DELETE /api/exec`

Cancela el job en curso si existe:

```
$ curl -X DELETE http://127.0.0.1:4319/api/exec -H "Origin: http://localhost:4319"
{"cancelling":"embed-docs","pid":1234}
```

Implementacion: SIGTERM inmediato + setTimeout 3s + SIGKILL si sigue vivo.
Mismo Origin check que POST. Si no hay job: `404 {"error":"No hay job en curso"}`.

### 4. Boton "Detener" en la UI

Aparece en la barra de la consola **solo mientras hay un job corriendo**.
Al pulsarlo:

1. El cliente llama `DELETE /api/exec` (server mata el child directo).
2. El cliente aborta el fetch via `AbortController` (cierra el socket, server detecta y tambien intenta matar como respaldo).

Resultado: el child recibe SIGTERM y la consola muestra
`─ exit null (SIGTERM) · X.Xs ─` en rojo.

### 5. Keep-alive SSE

Cada 15s el server envia `: keepalive\n\n` (comentario SSE). Esto previene
que proxies o el navegador cierren la conexion en jobs muy largos (>30s),
y mantiene `embed-docs` sobre repos grandes sin desconectarse.

## Cambios internos

### `buildExecPlan(root, dbPath, id, arg)`

Nueva funcion compartida que valida id+arg y compone el argv. La usan
tanto `runExecAction` (sync) como `streamExecAction` (SSE).

### `streamExecAction(root, dbPath, id, arg, onEvent)`

Spawn no-bloqueante. Emite eventos via callback `onEvent({type, data})`.
Devuelve Promise que resuelve al `child.on('close')`. Maneja timeout 5min
con SIGTERM + escala a SIGKILL si necesario.

### `cancelExecAction()`

Llama `__execJob.child.kill('SIGTERM')`. Programa `setTimeout(3000, SIGKILL)`
para forzar si SIGTERM no funciona (en Windows Node usa TerminateProcess).

### Cliente: parser SSE manual

No se usa `EventSource` (no soporta POST). En su lugar:

```js
fetch('/api/exec', { method:'POST', signal: ctrl.signal, headers: {Accept:'text/event-stream'}, ... })
  .then(r => r.body.getReader())
  // buffer chunks -> split por \n\n -> parsear event:/data:
  // emitir consoleLine por cada linea de stdout/stderr completa
```

Buffer linea-por-linea: si un chunk llega con `"OK. Foo bar\nOK. Baz"`,
emite `"OK. Foo bar"` y deja `"OK. Baz"` pendiente hasta que llegue el `\n`.

## Validacion

```
TEST 1  SSE status (12 lineas)        → 12 eventos stdout + exit 0 (190ms)
TEST 2  legacy sync sin Accept        → 200 + JSON completo (compat OK)
TEST 3  SSE sync-memory               → eventos stdout incrementales + exit 0
TEST 4  AbortController mid-flight    → server detecta socket close + SIGTERM
TEST 5  DELETE durante job            → 202 {cancelling,pid} + SSE exit SIGTERM
TEST 6  Concurrencia (2 jobs simul)   → segundo recibe SSE event:error 409
```

## Que NO esta todavia (Fase 3)

- **Historial persistente** en `ai_action_runs` (audit trail de runs).
- **Sub-tab Historial** con re-ejecutar.
- **Atajos de teclado** (`Ctrl+Shift+S` = sync, `Ctrl+Shift+R` = report).
- **Progreso estructurado** para comandos largos (ej. "X / Y archivos
  indexados") que requiere modificar los comandos para emitir JSON
  progresivo.

## Migracion desde v12.23

Sin migracion. La BD no cambia. El cliente existente sigue funcionando en
modo sync. Para activar streaming en una integracion externa, agregar
`Accept: text/event-stream` al POST y consumir el stream SSE.

# v12.25.0 — Audit trail + sub-tab Historial + atajos de teclado

`v12.25.0` cierra la Fase 3 del panel de Acciones: cada ejecucion queda
registrada en SQLite (`ai_action_runs`), un nuevo sub-tab "Historial"
muestra los ultimos 50 runs con re-ejecucion en un click, y 6 atajos
de teclado disparan las acciones mas usadas.

## Por que un historial persistente

Antes de v12.25, la consola del panel "Acciones" mostraba solo lo de la
sesion actual del navegador: si refrescabas, perdias todo. Eso impedia:

- Saber cuando fue la ultima vez que se sincronizo la memoria.
- Comparar runs de `check-trace-drift` dia a dia.
- Re-ejecutar un comando con un arg complicado (ej. `diff-since --since
  2026-05-01`) sin re-escribirlo.
- Diagnosticar "el sync fallo ayer" tras cerrar el navegador.

Con la tabla `ai_action_runs` el equipo ya tiene un audit trail real.

## Schema

```sql
CREATE TABLE IF NOT EXISTS ai_action_runs (
  id INTEGER PRIMARY KEY,
  action_id TEXT NOT NULL,
  arg TEXT,
  argv TEXT,                 -- argv saneado (JSON), <root> en vez de path absoluto
  mode TEXT,                 -- 'sync' | 'stream'
  origin TEXT,               -- 'ui' | 'api' | 'cli'
  started_at TEXT NOT NULL,
  finished_at TEXT,
  exit_code INTEGER,
  signal TEXT,
  duration_ms INTEGER,
  timed_out INTEGER,
  cancelled INTEGER,         -- 1 si terminado por DELETE o socket close
  stdout_tail TEXT,          -- ultimos 4KB
  stderr_tail TEXT           -- ultimos 4KB
);
CREATE INDEX IF NOT EXISTS idx_ai_action_runs_started ON ai_action_runs(started_at DESC);
CREATE INDEX IF NOT EXISTS idx_ai_action_runs_action ON ai_action_runs(action_id);
```

Idempotente: las DBs existentes ganan la tabla al primer arranque de
`memory-serve` (la schema-sqlite.sql se ejecuta en cada `ensureDatabase`).

## Sub-tab Historial

El pane "Acciones" ahora tiene dos sub-tabs:

- **Ejecutar** (default): el panel de v12.23/v12.24 con las 4 categorias
  + consola live + boton Detener.
- **Historial**: tabla de los ultimos 50 runs con:
  - **Cuando** (timestamp relativo: `5s`, `2min`, `3h`, `2d`).
  - **Accion** + arg si tuvo.
  - **Duracion** (`337ms` o `3.5s`).
  - **Modo** (badge `sync` o `stream`).
  - **Estado** (badge coloreado: verde `exit 0`, rojo `exit N` / `SIGTERM`, amarillo `cancelled`, gris `corriendo…`).
  - **`↻ Re-ejecutar`** — vuelve al sub-tab Ejecutar y dispara la misma accion con el mismo arg.
  - **Detalle plegable** (`<details>`) con el tail de stdout/stderr del run.

Refresca automaticamente despues de cada accion nueva si esta visible.
Boton `↻ Refrescar` para forzar.

## Endpoint nuevo

```
GET /api/action-runs?limit=N    # 1 <= N <= 200, default 50
```

Devuelve un array ordenado `started_at DESC` con los campos del schema.
Listo para consumir desde scripts o monitoreo externo.

## Atajos de teclado

Funcionan en cualquier tab. Por defecto:

| Atajo               | Accion                                  |
|---------------------|------------------------------------------|
| `Ctrl+Shift+S`      | `sync-memory`                           |
| `Ctrl+Shift+R`      | `memory-report` (regenerar HTML estatico) |
| `Ctrl+Shift+E`      | `regenerate-context` (AI_CONTEXT.md)    |
| `Ctrl+Shift+C`      | `check-trace-drift`                     |
| `Ctrl+Shift+H`      | abrir sub-tab Historial                 |
| `Esc`               | detener job en curso (= boton Detener)  |

Cada atajo (1) abre el tab Acciones, (2) abre el sub-tab "Ejecutar",
(3) dispara `execAction(actionId, '')`.

## Implementacion clave

### Persistencia con tolerancia a fallas

`persistRunBegin(db, plan, mode, origin)` y `persistRunEnd(db, runId, result)`:

- Solo escriben si `db` esta abierto (server live).
- Encierran todo en `try/catch` y retornan silenciosamente si fallan.
- El `INSERT` es minimo (action_id, arg, argv, mode, origin, started_at);
  el `UPDATE` agrega el resultado al final.
- Si el server muere mid-flight, la fila queda con `finished_at = NULL`
  y la UI muestra "corriendo…". Futura mejora: limpieza al boot.

### Tail bounded a 4KB

`tailString(s, 4096)` toma solo el final del output. Para stream, el
`acc.stdout` y `acc.stderr` se mantienen acotados via slice en cada chunk:
`acc.stdout = (acc.stdout + chunk).slice(-TAIL_BYTES)`. Asi una accion
que escriba 100MB no infla la BD.

### Sub-tabs sin reload

`showSubtab(name)` solo toggle de clases CSS. `loadHistory()` se llama
cuando se abre la pestaña por primera vez y despues de cada run nuevo
(si esta visible).

## Migracion desde v12.24

Sin cambios manuales. Al arrancar `memory-serve` con el nuevo
`scripts/ai-framework-agent.mjs`, `ensureDatabase` re-ejecuta el
`schema-sqlite.sql` y crea la tabla `ai_action_runs` en la BD existente.
Los runs anteriores no aparecen (no se conservaron); el historial
empieza desde el primer run con v12.25.

## Que NO esta todavia (futuro)

- **Limpieza al boot**: detectar runs con `finished_at = NULL` viejos
  (>1h) y marcarlos como abandonados.
- **Retencion configurable**: borrar runs > N dias automaticamente.
- **Export del historial** a CSV/JSON desde la UI.
- **Estadisticas agregadas** (avg duration por accion, tasa de exito).
- **Progreso estructurado** ("X/Y archivos indexados") para comandos largos.

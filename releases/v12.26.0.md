# v12.26.0 — Housekeeping del audit trail + Stats + Export

`v12.26.0` cierra la fase de polish del panel de Acciones. La tabla
`ai_action_runs` introducida en v12.25 ahora tiene mantenimiento
automatico (orphan cleanup + retencion), expone agregados por accion
(`/api/action-runs/stats`) y permite descargar el historial completo
en CSV o JSON.

## Housekeeping automatico

`memory-serve` ejecuta `housekeepActionRuns(db)` al boot y luego cada
hora (con `setInterval().unref()` para no bloquear shutdown):

### 1. Orphan cleanup

Filas con `finished_at = NULL` cuyo `started_at` sea anterior a 1h se
marcan:

```sql
UPDATE ai_action_runs
SET finished_at = <now>, cancelled = 1,
    stderr_tail = COALESCE(stderr_tail, '') || '\n[orphan: marcado al boot...]\n'
WHERE finished_at IS NULL AND started_at < <now - 1h>
```

Resuelve el caso "el server se cayo mid-flight y la fila quedo en
'corriendo…' para siempre" que dejaba la UI con un estado mentiroso.

### 2. Retencion configurable

```sql
DELETE FROM ai_action_runs WHERE started_at < <now - RETAIN_DAYS days>
```

Configurable via env `MEMORY_ACTIONS_RETAIN_DAYS` (default `30`). Si se
setea a `0`, se desactiva la purga.

### 3. Output

El boot imprime una linea solo si hubo cambios:

```
Housekeeping audit trail: 1 huerfanos marcados, 1 viejos purgados.
```

## Endpoint nuevo: estadisticas agregadas

```
GET /api/action-runs/stats
```

Por cada `action_id` devuelve:

| Campo          | Significado                                                 |
|----------------|-------------------------------------------------------------|
| `total`        | runs totales                                                |
| `ok`           | `exit_code=0 AND cancelled=0`                              |
| `fail`         | `exit_code!=0 AND cancelled=0`                             |
| `cancelled`    | `cancelled=1` (SIGTERM/SIGKILL via DELETE o socket close)  |
| `avg_ms`       | promedio de duration_ms                                     |
| `min_ms`/`max_ms` | extremos                                                |
| `p50_ms`/`p95_ms` | percentiles (calculo JS, sin window functions)         |
| `success_rate` | porcentaje 0-100 (`100 * ok / total`)                      |
| `last_run`     | ISO timestamp del run mas reciente                          |

Ordenado por `total DESC, action_id ASC`.

## Endpoint nuevo: export completo

```
GET /api/action-runs/export?format=csv
GET /api/action-runs/export?format=json
```

Descarga TODO el historial (no solo los 50 mas recientes). Cabeceras:

```
Content-Type: text/csv; charset=utf-8           # o application/json
Content-Disposition: attachment; filename="action-runs-2026-05-16T03-36-14.csv"
```

CSV con quoting RFC4180 (escapa `"`, `,` y saltos de linea). JSON
pretty-printed con `null` literal en campos vacios.

## Sub-tab "Stats" en la UI

Tercer sub-tab del pane Acciones (junto a "Ejecutar" y "Historial"):

- **Banner agregado**: 4 KPIs grandes — Total runs, Exitos (verde),
  Fallos (rojo), Success rate global.
- **Tabla por accion** con 12 columnas:
  `Accion · Total · OK · Fail · Cancel · Avg · p50 · p95 · Min · Max ·
  Success% · Ultima`.
- **Coloreado de success_rate**: verde >=95%, neutral 70-94%, rojo <70%.
- Boton `↻ Refrescar` y **auto-refresh** al terminar cualquier run nuevo
  si el sub-tab esta visible.

## Botones de Export en Historial

Junto al `↻ Refrescar`, dos botones-link:

```html
<a href="/api/action-runs/export?format=csv"  download>⬇ Export CSV</a>
<a href="/api/action-runs/export?format=json" download>⬇ Export JSON</a>
```

El atributo `download` + el header `Content-Disposition` hacen que el
navegador guarde el archivo con el timestamp automatico, sin cuelgues.

## Atajo nuevo: `Ctrl+Shift+T`

Abre directamente el sub-tab Stats desde cualquier tab. Total atajos:

| Atajo              | Accion                          |
|--------------------|--------------------------------- |
| `Ctrl+Shift+S`     | `sync-memory`                   |
| `Ctrl+Shift+R`     | `memory-report`                 |
| `Ctrl+Shift+E`     | `regenerate-context`            |
| `Ctrl+Shift+C`     | `check-trace-drift`             |
| `Ctrl+Shift+H`     | sub-tab Historial               |
| **`Ctrl+Shift+T`** | **sub-tab Stats (nuevo)**       |

## Validacion

```
TEST 1  Stats agregados (5 acciones)            -> totales, p50/p95, last_run OK
TEST 2  Export CSV con quoting RFC4180          -> header attachment + body valido
TEST 3  Export JSON pretty-printed              -> header attachment + indent 2
TEST 4  Orphan: row 3h ago, finished_at=NULL    -> tras restart: cancelled=1
TEST 5  Recent row (30s ago, finished_at=NULL)  -> intacto (dentro de cutoff 1h)
TEST 6  Retencion MEMORY_ACTIONS_RETAIN_DAYS=7  -> row de 10d purgada (12 -> 11)
```

## Migracion desde v12.25

Sin cambios manuales. La proxima vez que arranque `memory-serve`:

- Las DBs existentes ya tienen la tabla `ai_action_runs` (de v12.25); no
  hay nuevas columnas.
- El housekeeping correra y limpiara silenciosamente (solo imprime si
  hubo cambios).
- Para cambiar el periodo de retencion: `MEMORY_ACTIONS_RETAIN_DAYS=60
  npm run memory:serve` o `0` para desactivar.

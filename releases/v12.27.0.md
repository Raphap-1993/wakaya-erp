# v12.27.0 — Filtros en Historial + Barra de progreso live

`v12.27.0` cierra dos huecos de UX en el panel de Acciones:

1. **Historial filtrable**: tres selects en vivo (Accion, Estado, Periodo)
   sobre la lista. Antes habia que scrollear los 50 ultimos runs a mano.
2. **Barra de progreso en la consola**: cuando una accion emite lineas
   `[progress] X/Y label`, el cliente las renderiza como **una sola barra
   actualizable** en lugar de spammar la consola con N lineas de texto.
   `index-docs` y `embed-docs` ya las emiten.

Bonus: el primer filtro `status=fail` descubrio un bug de v12.23
(`check-prototype-html5` apuntando a un path inexistente). Fixed.

## Filtros en Historial

Nueva barra encima de la tabla con 3 selects:

| Filtro    | Opciones                                                              |
|-----------|-----------------------------------------------------------------------|
| Accion    | `— todas —` + ids del historial real + ids del catalogo               |
| Estado    | `solo OK` / `solo fail` / `solo cancelled` / `solo corriendo`         |
| Periodo   | `ultima hora` / `ultimas 24h` / `ultimos 7d` / `ultimos 30d`          |

El select de **Accion se autopopula** (no hay que escribir nada). Contador
de runs filtrados en la esquina: `12 runs` o `100+ runs` si toca el limite.
Cambiar cualquier filtro recarga la lista en vivo (1 fetch).

### Endpoint actualizado

```
GET /api/action-runs?action_id=<id>&status=ok|fail|cancelled|running&since=<periodo>&limit=N
```

`since` acepta:
- forma corta: `1h`, `24h`, `7d`, `30d`, `2w`, `1m`
- fecha ISO: `2026-05-01`

WHERE compuesto con parametros bind, sin string interpolation.

## Barra de progreso en consola

El cliente parsea cada chunk de stdout y detecta el patron:

```
[progress] <done>/<total> <label>
```

En vez de imprimir N lineas:

```
[progress] 0/2517 chunks
[progress] 50/2517 chunks
[progress] 100/2517 chunks
[progress] 150/2517 chunks
...
```

renderiza **una sola barra unificada** con porcentaje + ancho animado:

```
┌────────────────────────────────────────────────────┐
│ chunks: 1250/2517 (50%)                            │
│ ████████████████████░░░░░░░░░░░░░░░░░░░░░░░░░░░░  │
└────────────────────────────────────────────────────┘
```

Estilo: gradient cyan (`#06B6D4`) -> green (`#10B981`), transition de
`.25s ease` en el ancho. Cuando llega al 100% la barra "se cierra" y la
siguiente linea de progreso abre una nueva. Cualquier linea normal
tambien cierra la barra activa.

## Progreso en comandos del agente

### `index-docs`

```
[progress] 0/265 archivos
[progress] 13/265 archivos
[progress] 26/265 archivos
[progress] 39/265 archivos
...
[progress] 265/265 archivos
OK. Documentos indexados: 3. Sin cambios: 262. FTS5: si. DB: ...
```

Tick cada 25 archivos (o cada 5%, lo que llegue primero). Bug fix
colateral: la version anterior hacia `continue` en archivos sin cambios,
lo que rompia el contador. Refactor a un loop unificado que procesa
TODOS los archivos y separa el conteo de `indexed` vs `skipped`.

### `embed-docs`

```
[progress] 0/2517 chunks
[progress] 50/2517 chunks
...
[progress] 2500/2517 chunks
[progress] 2517/2517 chunks
OK. Embeddings locales generados: 2517. Total en BD: 2517. ...
```

Tick cada 50 chunks. Antes parecia colgado durante 4-5s sobre el corpus
del template (2517 chunks). Ahora hay feedback continuo.

## Bug fix: `check-prototype-html5` path

La accion `check-prototype-html5` (del whitelist v12.23) apuntaba a
`ci/scripts/check-prototype-html5.mjs` que no existe. El archivo real
es `check-html5-prototype-quality.mjs`. Esto causaba que la accion
fallara con `MODULE_NOT_FOUND` en el server.

**Fix aplicado en 3 lugares**:
- `EXEC_ACTIONS` (whitelist del panel UI)
- `LIVE_FRAMEWORK_FILES_TO_COPY` (lista que el generador copia al
  proyecto real generado)
- `scripts` en el `package.json` raiz

## Self-discovery: el audit trail justifica su existencia

El bug se descubrio en el primer test del filtro `status=fail` del
Historial: aparecio un `check-prototype-html5` con stderr
`Error: Cannot find module ...`. Sin el audit trail + filtros, este
bug habria seguido latente hasta que un agente IA lo invocara desde el
panel y se preguntara por que no funciona.

Conclusion: la inversion en v12.23 (panel) -> v12.25 (audit trail) ->
v12.26 (stats/export) -> v12.27 (filtros) compone una herramienta de
observabilidad que ya empezo a pagar.

## Migracion desde v12.26

Sin cambios manuales. Restartear `memory-serve` y abrir el navegador.
La barra de filtros aparece automaticamente en el sub-tab Historial,
los progress bars aparecen al ejecutar `index-docs` o `embed-docs`.

## Que NO esta todavia (futuro)

- **Persistir el filtro en URL** (`?actions_action=sync-memory&actions_status=fail`).
- **Progreso en otros comandos** (sync-memory por trace-link, check-docs
  por archivo, etc.).
- **Dashboard de tendencias**: chart de success_rate diario/semanal.
- **Alertas** si check-trace-drift falla N veces seguidas.

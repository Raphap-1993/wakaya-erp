# v12.28.0 — Tendencias (sparklines) + Alertas + URL persistente

`v12.28.0` agrega las dos piezas que faltaban para que el audit trail
(v12.25-v12.27) sea un sistema de observabilidad utilizable, no solo
almacenamiento:

1. **Tendencias**: sparklines de 14 dias por accion, integrados en Stats.
2. **Alertas**: banner rojo arriba del pane Acciones cuando hay fallos
   consecutivos. Detectado por una nueva query.
3. **URL persistente**: los filtros del Historial van en la query string.
   `?act_action=embed-docs&act_status=fail&act_since=24h` es un link
   compartible.

## 1. Sparklines de 14 dias en Stats

Cada fila de la tabla de stats ahora tiene una columna `14d` con un
sparkline SVG inline:

- **1 barra por dia** de los ultimos 14.
- **Altura proporcional** al volumen de runs ese dia (normalizado al max
  de la accion).
- **Color por estado dominante**:
  - verde `#10B981` — todos OK
  - rojo `#DC2626` — algun fail
  - amarillo `#F59E0B` — solo cancelled (sin OK ni fail)
  - gris `#E5E7EB` — sin runs
- **Tooltip nativo**: `2026-05-16: 10 runs (5 ok / 5 fail / 0 cancel)`

Implementacion: SVG inline puro, sin librerias. Renderizado en JS:
`sparklineSvg(series)` -> string.

## 2. Endpoint `/api/action-runs/trend`

```
GET /api/action-runs/trend?days=N&action_id=X
```

- `days` default 14, max 60.
- `action_id` opcional (filtra a una sola accion).
- Devuelve `[{action_id, days:[...], series:[{day, total, ok, fail, cancelled, avg_ms}, ...]}, ...]`.
- Rellena dias sin runs con ceros (longitud consistente).
- Sin window functions: `GROUP BY substr(started_at,1,10)`.

## 3. Endpoint `/api/action-runs/alerts`

```
GET /api/action-runs/alerts?last=N&min=K
```

- `last` default 5 (window de los ultimos N runs).
- `min` default 3 (umbral de fallos consecutivos).
- Devuelve solo acciones con `consecutive_failures >= min` al inicio
  de la ventana (ie. los mas recientes).
- Por cada alerta: `{action_id, consecutive_failures, window, last_success, latest_started_at}`.

Algoritmo: para cada `action_id` distinto, leer los ultimos N runs
`ORDER BY started_at DESC`, contar fallos consecutivos desde el inicio,
romper al primer OK/cancelled. Si `streak >= min`, emitir alerta.

## 4. Banner de alertas

Si `/api/action-runs/alerts` devuelve resultados al cargar el pane
Acciones, aparece un banner rojo (left-border `#B91C1C`) arriba de los
sub-tabs:

```
⚠ 4 ACCIONES CON FALLOS CONSECUTIVOS
─────────────────────────────────────────────────────────
check-docs:             5 fallos consecutivos · nunca tuvo OK    [↻ re-ejecutar] [ver historial]
check-prototype-html5:  4 fallos consecutivos · ultimo OK: 22min [↻ re-ejecutar] [ver historial]
check-trace-drift:      4 fallos consecutivos · nunca tuvo OK    [↻ re-ejecutar] [ver historial]
context-pack:           4 fallos consecutivos · nunca tuvo OK    [↻ re-ejecutar] [ver historial]
```

Cada item tiene dos atajos:
- **`↻ re-ejecutar`**: cambia al sub-tab Run y dispara la accion.
- **`ver historial`**: abre el sub-tab Historial con el filtro
  `action_id=` preseteado para inspeccionar los runs.

El banner se hidrata en background al cargar el catalogo de acciones.

## 5. Auto-discovery: la alerta paga su existencia

El primer arranque del endpoint sobre el historial real detecto **4
acciones con fallos consecutivos**:

- `check-trace-drift`: 4 fails. **Esperado** — el template tiene 18
  drift links conocidos (RF-04 -> bd:transicion_estado, etc.) que
  validan que el detector funciona.
- `check-docs`: 5 fails. **A revisar** — debe haber un test runner que
  invoca el script con un working dir distinto al CLI directo.
- `check-prototype-html5`: 4 fails. Eran de **antes del fix v12.27**
  cuando el path estaba mal; ya no se reproducen.
- `context-pack`: 4 fails. **Latente real** — no esta documentado y
  hay que diagnosticar.

Sin el banner de alertas, estos 4 patrones seguirian invisibles. Ahora
saltan a la vista al abrir el panel. Esa es la justificacion del
endpoint.

## 6. URL persistente para filtros

Al cambiar un filtro del Historial, la URL se actualiza in-place via
`window.history.replaceState`:

```
/                                        ← sin filtros
/?act_action=embed-docs                  ← solo embed-docs
/?act_status=fail                        ← solo fallos
/?act_action=sync-memory&act_status=fail&act_since=24h
```

Al cargar la pagina con esos parametros, los filtros se aplican
automaticamente Y se abre directamente el tab Acciones > Historial
(en lugar de Trazabilidad).

Casos de uso:
- Compartir un link a "todos los fallos de `embed-docs` del ultimo dia"
  con un compañero.
- Marcar la pestaña con tu filtro frecuente.
- Pegar un link desde el banner de alertas al historial filtrado.

## Migracion desde v12.27

Sin cambios manuales. Restartear `memory-serve`:

- `/api/action-runs/trend` y `/api/action-runs/alerts` aparecen.
- Stats sub-tab agrega la columna `14d` con sparklines.
- Si hay >=3 fallos consecutivos en alguna accion, el banner aparece
  al abrir el pane Acciones.
- Los filtros del Historial empiezan a persistir en URL desde el
  primer cambio.

## Que NO esta todavia

- **Charts de tamaño completo** (no solo sparklines): graficas con ejes
  y leyendas para drill-down de una accion.
- **Webhook/notificacion externa** cuando aparece una alerta nueva
  (ej. Slack, email).
- **Snooze de alertas conocidas** ("este check-trace-drift es esperado,
  no me lo muestres mas").
- **Comparacion entre periodos** (ultimo 7d vs 7d anteriores).

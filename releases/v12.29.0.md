# v12.29.0 — Dashboard de tendencias (sub-tab Tendencias)

`v12.29.0` cierra finalmente el pendiente arrastrado desde v12.26: el
**dashboard completo de tendencias** con line chart de success_rate y
stacked bars de volumen, no solo los sparklines pequeños de v12.28.

## Por que aparece esta version

El usuario pregunto "¿el Dashboard de tendencias donde lo veo?" y la
respuesta honesta era: no existia. La columna `14d` de sparklines en
Stats (v12.28) cubria "¿hubo runs este dia?" pero no "¿cual es la
tendencia de success_rate de las ultimas 4 semanas?". v12.29.0 cierra
ese hueco.

## Donde verlo

```bash
npm run memory:serve
# abrir http://localhost:4319
# tab "Acciones" → sub-tab "Tendencias"
# o directamente: Ctrl+Shift+D
```

## Endpoint nuevo

```
GET /api/action-runs/dashboard?days=N&action_id=X
```

- `days`: 7..180, default 30.
- `action_id`: opcional (sin = agregado de todas).
- Devuelve:
  ```json
  {
    "days_window": 30,
    "action_id": null,
    "days": ["2026-04-17", ..., "2026-05-16"],
    "series": [
      { "day":"2026-04-17", "total":0, "ok":0, "fail":0, "cancelled":0, "avg_ms":null, "success_rate":null },
      ...
    ],
    "compare": {
      "current":  { "total":81, "ok":51, "fail":27, "cancelled":3, "success_rate":63, "window_days":7 },
      "previous": { "total":0,  "ok":0,  "fail":0,  "cancelled":0, "success_rate":null, "window_days":7 },
      "delta_volume_pct": null,
      "delta_success_pp": null
    },
    "top_actions": [ { "action_id":"check-docs", "total":10, "ok":0, "fail":10, "success_rate":0 }, ... ]
  }
  ```

`series` siempre tiene `days` elementos (rellena ceros). `compare` toma
la mitad reciente vs la mitad anterior del rango; con datos solo en la
mitad reciente, `delta_*` es `null`.

## Componentes visuales

### Banner de comparacion (3 KPIs)

- **Success rate**: valor actual + delta en pp (puntos porcentuales),
  verde si sube, rojo si baja.
- **Volumen**: total de runs en el periodo actual + delta % vs anterior.
- **Fallos**: total de fallos + delta % vs anterior. Semantica
  invertida: subir es rojo (mal), bajar es verde (bien).

### Line chart de success_rate

- SVG 720x200, viewport responsive.
- Gridlines horizontales en 0/25/50/75/100%.
- Etiquetas Y con %.
- Linea cyan con puntos por dia con datos.
- Los dias sin runs (`success_rate=null`) **rompen la linea** en lugar
  de saltar a 0% (que daria una impresion equivocada).
- Tooltip nativo via `<title>` por cada punto: `2026-05-16: 63% (51/81)`.

### Stacked bars de volumen

- SVG 720x180.
- Bara apilada por dia: verde (ok) + rojo (fail) + amarillo (cancel).
- Altura proporcional al max del periodo (no a 100, asi 1 dia con 50
  runs no aplasta dias con 5).
- Tooltip nativo con totales del dia.

### Tabla Top 8

Acciones con mas runs en el periodo, ordenadas por volumen. Success
rate coloreado verde >=95% / neutro 70-94% / rojo <70%.

## Implementacion

- **Sin librerias externas**: helpers `buildLineChart(series, opts)` y
  `buildStackedBars(series, opts)` son SVG strings construidos a mano.
- **Sin window functions**: el agregado por dia usa `GROUP BY
  substr(started_at,1,10)`, compatible con SQLite anterior a 3.25.
- **Mismo patron que el resto del audit trail**: respeta el lock 127.0.0.1,
  los endpoints estan bajo `/api/action-runs/*`, los controles persisten
  estado via DOM (no en URL — no se pidio compartir links de tendencias,
  pero se puede agregar en una version futura).

Sobre los datos reales del template:
- success_rate global 7d: **63%** (81 runs, 51 ok).
- Peor accion: `check-docs` con 10/10 fallos en el dia.
- Mejor accion (con volumen): `sync-memory` con 6/7 ok (1 cancel).

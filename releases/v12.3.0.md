# v12.3.0 - PII redaction default, threat modeling, contract governance, chaos y consistencia RBAC

[README principal](../README.md) | [Indice docs](../docs/README.md) | [Volver a releases](README.md)

## Resumen
`v12.3.0` cierra las brechas ALTO identificadas en [`revisiones/2026-04-25-revision-post-v12-2.md`](../revisiones/2026-04-25-revision-post-v12-2.md). El foco es operacionalizar tres capas de gobernanza que faltaban: privacy by default (redaccion PII obligatoria en logs), governance de contratos publicos (gating de breakings), y reliability practice (threat modeling y chaos engineering como parte del ciclo). Tambien resuelve el drift cross-stack RBAC con un check automatico, y deja un workflow scheduled para GameDays DR programados.

## Destacado
- **PII redactor wirado al logger por defecto**:
  - node-next: nuevo `src/lib/logger.ts` que envuelve `console`/stdout, aplica `redact()` a cada payload y emite JSON line con `ts`, `level`, `event`. `logger.unsafe` queda como opt-out explicito. Tests en `tests/unit/logger.test.ts`.
  - Spring stacks: nuevo `PiiMaskingConverter` (Logback) + `logback-spring.xml` con regla `safeMsg` que aplica `PiiRedactor` a cada mensaje formateado antes de escribirlo.
  - Quarkus: nuevo `PiiLogFilter` (`@LoggingFilter("PII")`) que aplica `PiiRedactor` sobre cada `LogRecord` y sus parametros antes de emitir; activado en `application.properties` via `quarkus.log.console.filter=PII`.
- **Threat modeling 90.25**: documento transversal con metodologia STRIDE, plantilla de threat model por feature (`specs/<feature>/threat-model.md`), crosswalk con DPIA y mecanismo de revision en CI.
- **Contract governance 90.26 + check automatico**:
  - Documento con politica de SemVer del contrato, listas explicitas de cambios breaking vs non-breaking, deprecation 90 dias, patron expand-contract.
  - Nuevo `ci/scripts/check-openapi-diff.mjs` (Node, sin libs externas) que extrae paths/methods/responses/required/security via regex y reporta breakings entre `--base <ref>` y `--head <WORKING|ref>`. Detecta: paths/operations eliminados, response codes removidos, required nuevo en request body, security schemes removidos.
  - Job `contract-governance` en `.github/workflows/template.yml` que falla en PRs sin label `contract-breaking-approved` cuando hay breakings.
- **Chaos engineering 90.27**: documento transversal con principios, catalogo de 10 experimentos baseline (CE-1 a CE-10), plantilla de GameDay, cadencia y gating por entorno (dev libre / staging mensual / dr-test trimestral / prod excepcional con doble aprobacion).
- **Consistencia RBAC cross-stack**:
  - Nuevo `ci/scripts/check-rbac-consistency.mjs` que parsea las 4 implementaciones (TS `ROLE_PERMISSIONS` + 3x Java `Role.java`), normaliza a `{ role -> Set<permission> }` y exige paridad estructural. Reporta `role_missing`, `role_extra`, `permission_missing`, `permission_extra`.
  - Job `rbac-consistency` en `template.yml` que bloquea PRs con drift.
- **GameDay DR programado**: nuevo `.github/workflows/gameday-dr.yml` con `schedule` mensual contra region `dr-test`, `workflow_dispatch` con inputs `environment`, `tier`, `post_failover`, `vars` para `DR_KUBECTL_CONTEXT`, `DR_NAMESPACE`, `DR_PROMETHEUS_URL`, `DR_ERROR_RATE_THRESHOLD`, `secrets.DR_KUBECONFIG` para acceso al cluster pasivo. Si verify-dr falla en cron, abre issue con label `gameday-dr` para tracking.

## Compatibilidad
- Cambios aditivos. La cadena nav-guided se extiende: 90.24 -> 90.25 -> 90.26 -> 90.27 -> docs/README.
- node-next: el nuevo `logger.ts` es opcional para servicios existentes; no rompe `console.log`. Para activarlo, sustituir cada `console.log({...})` por `logger.info("event", {...})`.
- Spring stacks: el `logback-spring.xml` redefine el patron de salida. Si el servicio ya tenia `logback-spring.xml` propio, hay merge manual: agregar el `conversionRule` y cambiar `%msg` por `%safeMsg`.
- Quarkus: el filter `PII` se activa por una sola linea en `application.properties`. Para opt-out, eliminar la linea `quarkus.log.console.filter=PII`.
- El job `contract-governance` solo se activa en PRs; si no hay `origin/main` en el contexto del PR, falla suave (warning) sin bloquear. Para introducir un breaking aprobado, etiquetar el PR con `contract-breaking-approved`.

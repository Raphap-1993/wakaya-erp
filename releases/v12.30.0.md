# v12.30.0 — Cierre de los 8 pendientes del panel de Acciones

`v12.30.0` cierra todo el backlog acumulado del panel de Acciones desde
v12.27 hasta v12.29. Es una version grande pero focalizada: cada cambio
elimina una papeleta de la lista de "pendiente" del CHANGELOG anterior.

## Los 8 items + 1 bug fix de regalo

### 1. Filtros adicionales en Historial (`mode` + `slow`)

Dos selects nuevos en la barra del Historial:

- **Modo**: `solo sync` / `solo stream`. Filtra por como se invoco la
  accion (HTTP sync vs SSE streaming).
- **Lentos (>=)**: `1s` / `3s` / `5s` / `10s` / `30s`. Filtra runs
  cuyo `duration_ms` sea mayor o igual.

Persisten en URL: `?act_mode=stream&act_slow=3000`.

### 2. URL persistente para Dashboard de Tendencias

Antes solo Historial tenia URL persistente. Ahora Tendencias tambien:

```
/?trend_action=embed-docs&trend_days=90
```

Al cargar con esos params, abre directamente Acciones > Tendencias con
los filtros pre-aplicados. Tiene prioridad sobre los `act_*` que abren
Historial.

### 3. Drill-down al click en barra del stacked chart

Cada barra de volumen del stacked chart es ahora un `<g>` clickeable
con `data-day="YYYY-MM-DD"`. Al hacer click:

1. Cambia al sub-tab Historial.
2. Pre-llena el filtro `since` con la fecha del dia clickeado.
3. Pre-llena el filtro `action_id` con el que estaba en Tendencias.

Tambien: click en la columna "Accion" de la tabla "Top acciones por
volumen" filtra Tendencias a esa accion individual.

### 4. Comparacion por accion individual en el banner

El compare-banner ya filtraba por `action_id` en el endpoint, pero la
UI no dejaba claro si comparaba todas las acciones o una sola. Ahora
hay una cabecera explicita arriba del banner:

```
comparando embed-docs · ventana actual: 7d · ventana previa: 7d
```

o

```
todas las acciones (agregado) · ventana actual: 15d · ventana previa: 15d
```

### 5. Export PNG del dashboard

Boton `⬇ PNG` por cada chart (line + stacked bars). Implementacion sin
librerias:

1. `XMLSerializer().serializeToString(svg)` → string SVG.
2. `new Blob([svgStr], { type: "image/svg+xml" })` → Blob.
3. `URL.createObjectURL(blob)` + `new Image()` → carga el SVG.
4. `canvas.getContext('2d').drawImage(...)` con escala 2x + fondo blanco.
5. `canvas.toBlob(pngBlob => ...)` → otra Blob URL.
6. `<a download="dashboard-<aid>-<chart>-<timestamp>.png">` triggered click.

Resultado: descarga inmediata de un PNG 1440x400 nitido.

### 6. Webhook externo Slack/email

```bash
export MEMORY_ALERTS_WEBHOOK_URL=https://hooks.slack.com/services/T.../B.../...
# opcional, default 'slack':
export MEMORY_ALERTS_WEBHOOK_FORMAT=slack   # o 'json' para payload generico
npm run memory:serve
```

`memory-serve` al boot + cada 5min computa alertas activas y POSTea
cada una NUEVA al webhook. Formato Slack-compatible:

```json
{
  "text": "⚠ *check-docs*: 5 fallos consecutivos",
  "blocks": [
    { "type": "section", "text": { "type": "mrkdwn",
      "text": "⚠ *check-docs*: 5 fallos consecutivos en ventana de 5 runs.\nUltimo OK: nunca.\nUltimo intento: 2026-05-16T04:30:34.153Z." } }
  ]
}
```

Tabla nueva `ai_alert_notifications (action_id, fingerprint, notified_at,
webhook_status)` previene re-notificar. Fingerprint = `consecutive_failures::started_at_day`.

### 7. Snooze de alertas

Para silenciar alertas esperadas (ej. `check-trace-drift` que falla
by-design en el template):

- Tabla nueva `ai_action_snoozes (action_id PK, reason, snoozed_at, expires_at)`.
- 3 endpoints:
  - `GET /api/action-runs/snoozes` — lista.
  - `POST {action_id, duration: "24h"|"7d"|"30d"|"forever", reason?}` — silenciar.
  - `DELETE ?action_id=X` — despertar.
- `/api/action-runs/alerts` filtra acciones silenciadas a menos que
  `?include_snoozed=1` (entonces aparecen con `snoozed: true`).
- UI: cada item del banner trae 3 links: `🔕 silenciar 24h / 7d / forever`.
  `forever` pide razon via `prompt()`.

Resultado en el template real:
- Antes: banner con 4 alertas (check-docs, check-prototype-html5, check-trace-drift, context-pack).
- Tras snooze de check-trace-drift + check-prototype-html5: banner con **1 alerta** real (context-pack — bug latente sin documentar).

### 8. Progreso estructurado en mas comandos

- **`sync-memory`**: emite 7 lineas `[progress] N/6 fases (X: count)`:
  ```
  [progress] 0/6 fases (sincronizando 50 archivos)
  [progress] 1/6 fases (traceLinks: 64)
  [progress] 2/6 fases (gateRuns: 5)
  [progress] 3/6 fases (evidence: 38)
  [progress] 4/6 fases (decisions: 18)
  [progress] 5/6 fases (openQuestions: 5)
  [progress] 6/6 fases (sessionEvents: 15)
  ```
- **`check-docs`**: emite `[progress] N/total archivos` cada 25-50 archivos
  (cada 5% o lo que llegue primero). En el template: ~530 archivos.

La UI renderiza una sola barra que avanza, no spam de lineas.

### Bug fix bonus: `acceptsDb` flag

Descubierto durante el testing del progress de `check-docs`. La invocacion
via API exec siempre añadia `--db <path>` al argv. Esto funcionaba para
`check-trace-drift` y `check-trace-coverage` (que parsean `--db`), pero
**los 3 validadores restantes** (`check-docs`, `check-prototype-hub`,
`check-html5-prototype-quality`) lo interpretaban como un argumento
posicional (root) y se rompian con `readdirSync` sobre un archivo.

Fix: cada accion declara `acceptsDb: true|false`. `buildExecPlan` solo
agrega `--db` cuando corresponde. Los 3 validadores ahora pasan
correctamente desde la UI.

## Atajos de teclado (sin cambios)

Total: 8 atajos (S/R/E/C/H/T/D + Esc). No se agregaron nuevos en esta version.

## Migracion desde v12.29

Sin pasos manuales para los usuarios. Los nuevos schemas
(`ai_action_snoozes`, `ai_alert_notifications`) se crean automaticamente
en cualquier DB existente al primer arranque de `memory-serve` via
`CREATE TABLE IF NOT EXISTS`.

Para activar webhook: setear `MEMORY_ALERTS_WEBHOOK_URL` antes de arrancar.

## Que NO esta todavia

- **Filtros multi-select** en Historial (hoy solo single-select).
- **Webhook con plantilla custom** (hoy solo `slack` o `json` plano).
- **Alertas por umbral de duracion** (ej. "si embed-docs supera p95+50%").
- **Persistencia de la URL para sub-tab Stats** (Historial y Tendencias ya
  persisten, Stats no porque no tiene filtros).
- **Dashboard combinado de varias acciones a la vez** (hoy es una o todas).
- **Boton "limpiar snoozes expirados"** (hoy se acumulan; futuro:
  housekeeping cada hora los purga).

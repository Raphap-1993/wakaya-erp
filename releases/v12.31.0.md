# v12.31.0 — Coherencia matriz ↔ memoria ↔ validadores

`v12.31.0` cierra las 3 inconsistencias detectadas en una revision externa
del estado del template tras v12.30. La arquitectura conceptual
planned/implemented/validated de v12.22 era correcta, pero la
implementacion aun tenia falsos positivos que invalidaban las preguntas
basicas del agente IA.

## El problema concreto

La revision externa pregunto: "¿Estos archivos implementan RF-02?". La
respuesta deberia ser un `NO` claro porque `ExpedienteQueryService` no
existe en `src/`. Pero:

1. La matriz canonica `TRACEABILITY_MATRIX.md` llevaba el nombre
   aspiracional, incumpliendo su propia regla "usar `-` mientras no exista".
2. `check-trace-drift` reportaba 18 drifts (correctos pero ruidosos).
3. `check-trace-coverage` decia "OK. Todas las features tienen cobertura
   completa" — **engañoso**: en realidad nunca encontraba RFs porque
   `evidence_ref` no estaba calificado por scope.
4. La tabla `ai_trace_links` tenia duplicados porque el mismo link aparecia
   en `TRACEABILITY_MATRIX.md` raiz Y en `specs/.../traceability.md`.

Sin estas correcciones, un agente IA podia leer mensajes contradictorios
de la BD/validadores y desviarse.

## Los 4 fixes

### 1. Matrices canonicas alineadas con la regla

Las 2 matrices del template (raiz + per-feature) ahora usan `-` donde
corresponde:

```diff
- | RF-02 | ... | ExpedienteQueryService | ExpedienteQueryTest | Prototipo validado | ...
+ | RF-02 | ... | -                      | -                   | Prototipo validado | ...
```

Los nombres futuros (`ExpedienteQueryService`, etc.) viven ahora donde
corresponde: en `spec-tareas.md` de cada feature.

### 2. `evidence_ref` calificado con scope

Antes:
```sql
INSERT INTO ai_trace_links(..., evidence_ref, ...)
-- evidence_ref = "prototype-validation.md"  ← sin scope
```

Ahora:
```sql
-- evidence_ref = "specs/001-bandeja-trabajo-expedientes/prototype-validation.md"
```

Logica en `syncTraceLinks`:
```javascript
const scopeMatch = file.rel.match(/^(specs\/[^/]+)\//);
const scope = scopeMatch ? scopeMatch[1] : null;
if (scope && evidenceRaw && !evidenceRaw.includes("/")) {
  evidenceRef = `${scope}/${evidenceRaw}`;
}
```

### 3. `source_file` + dedup logico en `ai_trace_links`

Nueva columna `source_file` registra de que Markdown vino cada link:

| source_file | links |
|---|---|
| TRACEABILITY_MATRIX.md | 15 |
| specs/001-bandeja-trabajo-expedientes/traceability.md | 18 |

`syncTraceLinks` ahora:
- Procesa primero las matrices per-feature (mas especificas).
- Dedupica por `(source_ref, target_type, target_ref)` con un `Set`.
- Cuando el global TRACEABILITY_MATRIX.md tiene un link ya visto por la
  per-feature, se descarta (no se inserta duplicado).

Resultado: **trace links bajan de 64 a 33** (-48%) sin perder informacion.

### 4. `check-trace-coverage` arreglado

Antes (engañoso):
```sql
WHERE source_type IN ('RF', ...) AND evidence_ref LIKE ?
-- params: ['%specs/001-bandeja.../%']  ← evidence_ref no traia scope
-- resultado: 0 filas siempre -> "OK" falso
```

Ahora:
```sql
WHERE source_type IN ('RF', ...)
  AND (source_file LIKE ? OR evidence_ref LIKE ?)
-- params: ['specs/001-bandeja.../%', '%specs/001-bandeja.../%']
-- resultado: encuentra los RF reales y verifica codigo + test
```

Verificado: ahora reporta correctamente:
```
specs/001-bandeja-trabajo-expedientes :: RF-02  faltan: codigo, test
specs/001-bandeja-trabajo-expedientes :: RNF-01  faltan: codigo, test
specs/001-bandeja-trabajo-expedientes :: RNF-03  faltan: codigo, test
```

(Sigue siendo "fail" pero por la razon correcta: el template no tiene
codigo real, no porque el validador este roto.)

### 5. AI_CONTEXT.md con version auto-regenerable

```diff
- - Version actual: v12.20.0
+ - Version actual: <!-- auto:start name=version -->v12.31.0<!-- auto:end -->
```

`regenerate-context` lee `package.json` y reemplaza la zona. Aplicable
tanto a la instancia raiz como a la plantilla que emite el generador.

### 6. check-ai-artifacts: 4 hallazgos -> 0

Agregadas las secciones `## No lo uses cuando` y `## Verificacion minima`
a:
- `ai/prompts/generar-prototipo-html5-desde-spdd.md`
- `ai/prompts/generar-prototipo-html5-ejecutable.md`

## Respuesta clara a la pregunta original

**¿Estos archivos implementan RF-02?**

Con v12.31:
```
$ npm run memory:query -- --preset rf-implemented
0 resultados                          ← claro: ningun RF tiene codigo + test reales

$ npm run memory:query -- --preset rf-planned
RF-04                                  ← solo RF-04 esta planificado (con un target real planeado)

$ npm run check:trace-drift
DRIFT: 1 trace links apuntan a artefactos que no existen.
  RF-04 --bd--> transicion_estado     ← senal real, no falso positivo

                                       ← reporta los huecos reales por feature
```

Tres validadores y la BD ahora dan respuestas coherentes y verificables.

## Migracion desde v12.30

Sin pasos manuales. `ensureDatabase` agrega `source_file` a `ai_trace_links`
en DBs existentes via `ALTER TABLE ADD COLUMN`. La proxima `sync-memory`
re-puebla todos los links con `source_file` y dedup.

Si tu proyecto real tiene `traceability.md` con nombres aspiracionales en
`Codigo`/`Test`, considera limpiarlos a `-` para evitar drifts en
`check-trace-drift`. Los nombres futuros viven en `spec-tareas.md`.

## Que NO esta todavia

- **`harvestTraceFromSource` con source_file = path real al codigo**: hoy
  guarda `source_file = rel` (igual que `evidence_ref`); util pero podria
  separarse mejor cuando el harvest reciba mas trabajo.
- **`check-trace-coverage` con phase explicita**: hoy infiere fase 5+ por
  existencia de `spec-tareas.md`. Podria leer una columna `phase` real
  de `ai_gate_runs` para precision.
- **`AI_CONTEXT.md` con seccion auto:start para Stack** (hoy hardcodea
  Node 20+ pero el template ya pide Node 22+).

# v12.32.0 — Cierre de 8 pendientes (coherencia + panel)

`v12.32.0` cierra los **3 pendientes que dejé en v12.31** (coherencia) y
los **5 pendientes del panel de Acciones desde v12.30** en una sola
version cohesiva.

## Los 8 items

### 1. `harvestTraceFromSource` separa `source_file` de `evidence_ref`

Antes (v12.31): ambos guardaban `rel` (la ruta del archivo). Inútil.

Ahora:
- `source_file = src/foo.ts` ← path puro
- `evidence_ref = src/foo.ts:42 @trace RF-02` ← cita con linea + tag

Permite drill-down al numero de linea exacto desde la consulta `docs-for`.

### 2. `check-trace-coverage` con phase explicita

Antes solo inferia fase 5+ por `existsSync(spec-tareas.md)`. Heuristica
fragil. Ahora:

1. Lee `ai_documents.phase` (poblada por `index-docs` desde paths y headings).
2. Convierte a numero: `fase-5`, `5`, `construccion` -> 5.
3. Si CUALQUIER `.md` del scope tiene `phase >= minPhase`, la feature
   esta activa.
4. Fallback: heuristica de `spec-tareas.md` para BDs viejas o casos limite.

```javascript
function phaseToNumber(phaseStr) {
  // soporta '5', 'fase-5', 'construccion', etc.
  if (/iniciaci/.test(s)) return 0;
  if (/analisis|requerimientos/.test(s)) return 1;
  // ... etc
}
```

### 3. `AI_CONTEXT.md` con zona `auto:start name=stack`

Antes:
```
- Stack: Node 20+ ...     ← hardcodeado, congelado mientras el template pedia 22+
```

Ahora:
```
- <!-- auto:start name=stack -->
Node >=22.0.0 · stacks de referencia: java-monolith, node-next, quarkus-angular, spring-react
<!-- auto:end -->
```

`regenerate-context` lee en orden:
1. `template.config.json` -> `project.stack` (proyectos reales).
2. `package.json` -> `engines.node`.
3. `stacks/` -> directorios disponibles.

### 4. Filtros multi-select en Historial (CSV)

UI con `<select multiple>`. URL persiste como CSV:

```
/?act_action=status,sync-memory&act_status=ok,fail&act_mode=stream
```

Endpoint actualizado para parsear CSV en `action_id`, `status`, `mode`:

```sql
WHERE action_id IN (?, ?)
  AND ((exit_code = 0 AND cancelled = 0) OR (exit_code != 0 ...))
  AND mode IN (?)
```

Helper `multiCsv(id)` y `setMulti(id, csv)` compartidos entre `loadHistory`,
`writeFiltersToUrl`, `readFiltersFromUrl`, `populateActionFilter`.

### 5. Webhook con plantilla custom

```bash
export MEMORY_ALERTS_WEBHOOK_URL=https://discord.com/api/webhooks/...
export MEMORY_ALERTS_WEBHOOK_TEMPLATE='{"content":"🔥 {action_id} fallo {consecutive_failures}x desde {last_success}"}'

# O en archivo (mejor para JSON complejos):
export MEMORY_ALERTS_WEBHOOK_TEMPLATE_FILE=ai/memory/discord-template.json
```

Placeholders soportados: `{action_id}`, `{consecutive_failures}`, `{window}`,
`{last_success}`, `{latest_started_at}`, `{kind}`, `{detail}`, `{timestamp}`.

Si no se define plantilla, sigue el formato Slack/JSON anterior. Util para
Discord, Teams, Mattermost, PagerDuty, n8n, custom CRM.

### 6. Alertas por umbral de duracion (`kind: 'duration-threshold'`)

Nuevo tipo de alerta. Para cada accion:

1. Calcula `p95` de las duraciones de runs OK **historicos** (excluyendo el actual).
2. Si requires `>= 5` samples para estabilidad.
3. Si `last.duration_ms > p95 * (1 + slow_pct/100)`, emite alerta:

```json
{
  "kind": "duration-threshold",
  "action_id": "embed-docs",
  "detail": "ultimo run 10000ms supera umbral 405ms (p95 historico=270ms +50%)",
  "duration_ms": 10000,
  "p95_ms": 270,
  "threshold_ms": 405,
  "latest_started_at": "2026-05-16T07:50:54.142Z"
}
```

Parametros:
- Endpoint: `?slow_pct=50&slow_min_samples=5`
- Env (ticker): `MEMORY_ALERTS_SLOW_PCT=50`

UI: cada alerta `slow` se renderiza con `🐢` y el detalle visible.

### Bug fix critico de este feature

Primer intento: `okDurations` incluia el run actual -> comparacion
circular, nunca disparaba. Fix:
```diff
- WHERE action_id = ? AND exit_code = 0 ...
+ WHERE action_id = ? AND exit_code = 0 AND started_at < ? ...
+ -- params: [aid, last.started_at]
```

### 7. URL persistente para sub-tab activo

`showSubtab(name)` ahora hace `replaceState(?act_subtab=<name>)`. El
default `run` no se escribe (URL limpia). Al cargar con
`?act_subtab=stats` se abre directamente Acciones > Stats.

Combinado con v12.28 (filtros Historial) y v12.30 (filtros Tendencias):
- `/?act_subtab=stats` -> sub-tab Stats.
- `/?act_action=embed-docs&act_status=fail` -> sub-tab Historial filtrado.
- `/?trend_action=sync-memory&trend_days=90` -> sub-tab Tendencias filtrado.

### 8. Housekeeping purga snoozes expirados

```sql
DELETE FROM ai_action_snoozes
WHERE expires_at IS NOT NULL AND expires_at < <now>;
```

Se ejecuta al boot + cada hora. Los snoozes `forever` (expires_at NULL)
se preservan. Boot log:

```
Housekeeping audit trail: 0 huerfanos marcados, 0 viejos purgados, 1 snoozes expirados.
```

## Migracion desde v12.31

Sin pasos manuales. La proxima vez que arranque `memory-serve`:

- `regenerate-context` agrega la zona `stack` a `AI_CONTEXT.md` (si no tiene marcador, lo agregas a mano una vez).
- `check-trace-coverage` empieza a usar phase explicita (mas estricto).
- `harvest-trace` empieza a poblar `evidence_ref` con la cita semantica.
- Las URLs viejas (con un solo valor en `act_action`) siguen funcionando — CSV es compatible.
- Webhook sigue funcionando con format Slack si no defines plantilla.

## Que NO esta todavia

- **Alertas combinadas** ("falla 3 veces Y duracion subio 50%").
- **Snooze por kind** (snoozear solo `duration-threshold` de una accion,
  mantener `failure-streak` activa).
- **Webhook con retry/backoff** (hoy es fire-and-forget; si el webhook
  esta caido, la alerta se pierde para ese fingerprint).
- **Dashboard exportable como markdown** (hoy solo PNG).
- **Comparacion entre 2 acciones lado a lado** en Tendencias.

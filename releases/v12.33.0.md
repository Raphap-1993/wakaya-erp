# v12.33.0 — Template CI-green (cierre de revision externa)

`v12.33.0` cierra los 4 puntos que la revision externa de v12.31
identifico como bloqueantes para confiar el template a agentes IA sin
supervision. Ahora `npm run check:all` pasa en verde sobre el template
canonico (sin codigo real, lo cual es el caso de uso correcto).

## Los 4 puntos cerrados

### 1. Drift legitimo `RF-04 → bd:transicion_estado` resuelto

**Antes**: la matriz declaraba `transicion_estado` como BD de RF-04,
pero la tabla no estaba documentada en ninguna spec. `check-trace-drift`
reportaba 1 drift correcto pero ruidoso.

**Ahora**: `specs/002-cambio-estado-expediente/spec-tecnica.md` tiene
una seccion "Modelo de datos" formal con la tabla:

```markdown
## Modelo de datos

Tabla `transicion_estado` ...

| Columna | Tipo | Notas |
|---|---|---|
| id | UUID | PK |
| expediente_id | UUID FK | apunta a `expediente.id` |
| estado_origen | TEXT | catalogo de estados |
| estado_destino | TEXT | catalogo de estados |
| motivo | TEXT | obligatorio |
| evidencia_path | TEXT NULL | adjunto si aplica |
| usuario_id | UUID FK | apunta a `usuario.id` |
| ocurrida_en | TIMESTAMPTZ | indice descendente |
| correlation_id | TEXT | trazabilidad |

Indices: `(expediente_id, ocurrida_en DESC)` y `(estado_destino)`.
Restricciones: `(estado_origen, estado_destino)` validados contra la
maquina de estados. La fila es append-only (no UPDATE/DELETE).
```

`targetRefExists('bd', 'transicion_estado')` la encuentra en `specs/`,
el link pasa a `link_status='implemented'` y `check-trace-drift` reporta
**0 drifts**.

### 2. `check-trace-coverage`: eliminado el fallback heuristico

**El bug**: la version v12.32 todavia decia "si existe `spec-tareas.md`,
asumir fase >=5". Pero `spec-tareas.md` es de fase **4** (SDD planning),
no de fase 5 (construccion).

El validador marcaba como construccion features que aun estaban en SDD
y exigia codigo/test cuando no correspondia.

**Fix**: el validador ahora solo confia en:

```javascript
// (1) phase explicita de ai_documents
const docs = db.prepare(`SELECT phase FROM ai_documents WHERE path LIKE ?`).all(`${scope}/%`);
for (const d of docs) {
  if (phaseToNumber(d.phase) >= minPhase) return true;
}
// (2) existencia REAL de codigo en src/<feature>/ o backend/<feature>/
for (const candidate of ["src", "backend", "frontend"]) {
  if (existsSync(join(root, candidate, slug))) return true;
}
return false;
```

`spec-tareas.md` solo dice "termine la planificacion SDD", NO "empece
a construir". El validador ya no lo confunde.

### 3. Separacion `check:template` vs `check:project`

Antes habia un solo `check:all` que mezclaba checks que no aplican
al mismo target:

| Comando            | Que cubre                                                              | Cuando aplica           |
|--------------------|------------------------------------------------------------------------|-------------------------|
| `check:template`   | `check:docs`, `check:prototype-hub`, `check:ai-artifacts`, `check:markdown-paths` | template canonico (debe ser verde) |
| `check:project`    | `check:trace-drift`, `check:trace-coverage`                            | proyectos reales con codigo |
| `check:all`        | `check:template && check:project`                                      | ambos                   |

Sobre el template canonico (sin codigo), `check:project` ahora tambien
queda en verde porque:
- `check:trace-drift` -> 0 drifts (transicion_estado resuelto).
- `check:trace-coverage` -> "no hay features en fase >=5" (correcto:
  el template no tiene `src/<feature>/`).

### 4. Preset nuevo `rf-not-implemented` (sin ambiguedad)

**Antes**: para responder "¿que RFs me faltan implementar?" habia que
correr 2 presets (`rf-without-code`, `rf-without-test`) y unir
mentalmente. `rf-planned` confundia porque solo devolvia RFs con
artefactos fantasma en la matriz, no todos los huecos reales.

**Ahora**:
- `rf-not-implemented` -> **union** de rf-without-code + rf-without-test
  ("cualquier hueco"). Una sola query.
- `rf-planned` -> aclarada la label a "Requerimientos con artefacto
  declarado pero inexistente (drift)". Es un caso especifico, no el
  general.

## Para proyectos generados

`create-project` ahora emite el `package.json` con los mismos scripts
nuevos. `LIVE_FRAMEWORK_FILES_TO_COPY` incluye los 2 validadores
faltantes (`check-ai-artifacts.mjs` y `check-markdown-paths.mjs`).

Un proyecto recien generado puede correr `npm run check:template` desde
el dia 1 y obtener verde.

Cuando ese proyecto empiece a escribir codigo en `src/<feature>/`,
`check:project` empezara a exigir cobertura — comportamiento esperado.

## Migracion desde v12.32

Sin pasos manuales. La proxima `npm run memory:sync` re-pobla los links
con la nueva semantica. Los scripts `check:template` y `check:project`
aparecen en el `package.json` actualizado.

Si tu proyecto real tiene un drift legitimo (un BD/codigo declarado en
la matriz pero no documentado en ninguna spec), tienes 3 opciones:

1. **Documentarlo** en `spec-tecnica.md` (preferido, como hicimos con
   `transicion_estado`).
2. **Cambiarlo a `-` en la matriz** (si aun no esta decidido).
3. **Aceptar el drift** como senal de trabajo pendiente.

## Que NO esta todavia

- **`check:project` con phases granulares** (hoy solo distingue fase
  <5 vs >=5; podria distinguir fase 4 vs 5 vs 6 con cobertura distinta
  por fase).
- **Plantilla de spec-tecnica con `## Modelo de datos`** obligatorio
  para que cada tabla declarada tenga doc automaticamente.
- **CI workflow YAML preconfigurado** que use `check:template` en PR de
  template y `check:project` en PR de codigo.

# v12.34.0 — Convencion BD documentada (replicable a proyectos reales)

`v12.34.0` cierra un hueco de **adopcion**: el patron `## Modelo de datos`
con `` Tabla `<nombre>` `` que introdujimos en v12.33 para resolver el
drift de `transicion_estado` vivia solo en el ejemplo canonico, sin
explicacion ni enforcement. Cualquier otro agente generando un proyecto
real no tenia forma de saber que debia replicarlo.

v12.34 lo hace **discoverable, replicable y validable**.

## Las 4 capas

### 1. Plantilla canonica (`plantillas/fase-4-sdd/spec-tecnica.md`)

La seccion `## Modelo de datos` paso de ser una linea generica
(`Entidades, tablas, indices...`) a ser declarativa:

```markdown
## Modelo de datos

> Regla v12.34+ (validador `check-bd-documented`): cada tabla declarada en
> la columna `BD` de `traceability.md` de esta feature **debe** aparecer aqui
> como `Tabla \`<nombre>\`` con su descripcion de columnas.

Plantilla canonica (copiar por cada tabla):

Tabla `<nombre_tabla>` (...)

| Columna | Tipo | Notas |
|---|---|---|
| id | UUID | PK |
...

Indices: `(...)`. Restricciones: ... La fila es <append-only|mutable>.

Si la feature no toca BD, escribe explicitamente:

> Esta feature no introduce ni modifica tablas. Consume `<tabla>` documentada
> en `specs/<otra-feature>/spec-tecnica.md`.
```

### 2. Generator emission

`create-project` ahora emite la seccion completa en cada nueva
`spec-tecnica.md` con una tabla concreta basada en `ctx.apiResourceName`:

```markdown
## Modelo de datos

> Regla: toda tabla declarada en la columna `BD` debe aparecer aqui...

Tabla `case` (cuando se implemente la entity vivira en
`src/backend/infrastructure/cases/`...):

| Columna | Tipo | Notas |
|---|---|---|
| id | UUID | PK |
| numero | TEXT | unico, formato CASE-YYYY-NNNN |
| estado | TEXT | catalogo de estados |
| prioridad | TEXT | enum (alta/media/baja) |
| responsable_id | UUID FK | apunta a `usuario.id` |
| fecha_actualizacion | TIMESTAMPTZ | indice descendente |

Indices: `(estado, prioridad)` y `(responsable_id, fecha_actualizacion DESC)`.
Restricciones: `numero` UNIQUE.
```

Cualquier feature nueva nace con doc canonica desde el primer commit.

### 3. Validador nuevo `check-bd-documented.mjs`

Para cada `target_ref` con `target_type='bd'` en `ai_trace_links`,
verifica que exista la marca canonica `` Tabla `<nombre>` `` (o
`Tabla **<nombre>**`) en algun `specs/**/spec-tecnica.md`.

```bash
$ npm run check:bd-documented
OK. 6 tablas declaradas, 5 documentadas en spec-tecnica.md. Cero huerfanos.

# Si falta alguna:
$ npm run check:bd-documented
BD SIN DOCUMENTAR: 1 tabla(s) declaradas en la columna BD de algun
traceability.md no aparecen como `Tabla \`<nombre>\`` en ningun spec-tecnica.md.

  - "transicion_estado" declarada por RF-04 en specs/.../traceability.md

Fix: agregar una seccion `## Modelo de datos` con `Tabla \`<nombre>\`` en
el spec-tecnica.md de la feature que la genere.
```

**Es mas estricto que `check-trace-drift`** porque exige la marca
canonica, no cualquier mencion suelta del nombre en el texto.

### 4. Documentacion en `INSTANCIACION_PROYECTO_REAL.md`

Nueva seccion "Convencion BD" con:
- Ejemplo completo de `transicion_estado`.
- Las 3 opciones cuando aparece una BD nueva:
  1. Documentarla con marca canonica (preferido).
  2. Referenciar otra spec si vive ahi.
  3. Usar `-` en la matriz si aun no esta decidida.
- Comandos: `npm run check:bd-documented` y `npm run check:project`.

Cualquier agente IA que abra el repo encuentra el patron en la guia
obligatoria de instanciacion.

## Por que NO bastaba `check-trace-drift`

`check-trace-drift` (v12.22) usa `targetRefExists('bd', ...)` con esta
logica:

```javascript
case "bd": {
  const needle = v.split(/[\s(]/)[0].toLowerCase();
  const files = [...listMarkdownFiles(root, ["specs"]), ...listSourceFiles(root)];
  return files.some((f) => {
    try { return fs.readFileSync(f, "utf8").toLowerCase().includes(needle); }
    catch { return false; }
  });
}
```

Eso busca CUALQUIER ocurrencia del nombre en cualquier archivo de
`specs/`. Un parrafo casual con la palabra "transicion_estado"
(`"En el futuro consideraremos transicion_estado como ..."`) lo marca
como implementado, sin que exista una entity real declarada.

`check-bd-documented` resuelve esto exigiendo la marca canonica
`` Tabla `<nombre>` `` o `Tabla **<nombre>**`. Una mencion casual ya no
cuenta. La marca solo aparece cuando un humano (o agente) decidio
formalmente que esa tabla existe.

## Integracion en el pipeline

| Lugar | Cambio |
|---|---|
| `package.json` root y emitido | nuevo `check:bd-documented` standalone + agregado a `check:project` |
| `EXEC_ACTIONS` | nueva accion `check-bd-documented` en categoria validador, `acceptsDb: true` |
| `LIVE_FRAMEWORK_FILES_TO_COPY` | `.mjs` copiado a proyectos generados |
| Panel UI | aparece en el grupo "Validadores" del tab Acciones |
| Banner alertas | si falla > 3 veces seguidas, dispara alert |

## Aplicado al template canonico

`specs/001-bandeja-trabajo-expedientes/spec-tecnica.md` ahora documenta
formalmente:

- **Tabla `expediente`** (8 columnas, 3 indices, UNIQUE en numero, NOT NULL
  condicional en responsable_id).
- **Tabla `rbac`** (matriz rol-permiso con scope opcional, clave compuesta).

`specs/002-cambio-estado-expediente/spec-tecnica.md` (de v12.33) sigue
documentando:

- **Tabla `transicion_estado`** (9 columnas, append-only, indices para reportes).

Resultado:
```
6 tablas declaradas en la matriz (con duplicados)
5 tablas unicas documentadas con marca canonica
0 huerfanos
check:all -> verde end-to-end
```

## Migracion desde v12.33

Sin pasos manuales para usuarios. Las DBs existentes funcionan igual; el
validador es **lectura pura** sobre `specs/`.

Si tu proyecto real ya tiene BDs declaradas en la matriz sin doc
canonica, corre:

```bash
npm run check:bd-documented
```

Por cada hallazgo, agrega la seccion canonica al `spec-tecnica.md` de la
feature que genera la tabla, siguiendo
`plantillas/fase-4-sdd/spec-tecnica.md` como referencia.

## Que NO esta todavia

- **`check-bd-documented` con verificacion de columnas minimas** (hoy
  solo verifica que aparezca `` Tabla `<nombre>` ``; podria exigir
  `id`, `PK`, indices declarados).
- **`check-api-documented`**: equivalente para columna `API` (que cada
  endpoint declarado en la matriz tenga doc formal en `api-contract.md`).
- **`check-test-documented`**: equivalente para columna `Test` cuando
  llegue fase 5.
- **Plantilla de `api-contract.md`** con regla similar.

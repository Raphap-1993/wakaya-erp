# v12.36.0 — Validadores semanticos + spec-funcional obligatoria + concurrency

`v12.36.0` cierra los 5 pendientes de v12.35. Cada validador pasa de
"verificar presencia" a "verificar propiedades semanticas": tipos SQL
reconocidos, endpoints en OpenAPI, tests con @trace al RF, secciones
funcionales explicitas.

## Los 5 items + 1 hardening

### 1. `check-bd-documented` con tipos SQL conocidos

Antes exigia tabla markdown + PK + indices. Ahora ademas valida que
cada columna use un tipo SQL del whitelist:

```
TEXT, VARCHAR, CHAR, STRING,
UUID,
INT, INTEGER, BIGINT, SMALLINT, SERIAL, BIGSERIAL,
DECIMAL, NUMERIC, REAL, FLOAT, DOUBLE,
BOOLEAN, BOOL,
DATE, TIME, TIMESTAMP, TIMESTAMPTZ, DATETIME,
JSON, JSONB, ARRAY, ENUM,
BYTEA, BLOB, BINARY
```

Acepta tipos con tamaño (`VARCHAR(255)`, `DECIMAL(10,2)`) y modifiers
en la celda (`NULL`, `NOT NULL`, `FK`, `UNIQUE`, `REFERENCES`).

Si un agente IA inventa `STR` o `TIMESTAMP_NEW`, el validador reporta:

```
BD INCOMPLETAS: 1 tabla(s) ...
  - "expediente" en specs/001/spec-tecnica.md:
      * columnas con tipos no reconocidos: created_by -> "STR" (no es tipo SQL conocido)
```

### 2. `check-api-documented` con validacion OpenAPI

Si existe `contracts/api/openapi.yaml` o `specs/<feature>/openapi.yaml`,
exige que cada endpoint declarado en la matriz aparezca tambien ahi
(no solo en `api-contract.md`).

Parser ligero del YAML sin dependencias externas:

```javascript
// Path: indentado 2-4 espacios, termina en ':'
const pathM = line.match(/^(\s{2,4})(\/[A-Za-z0-9_\-/{}.]+):\s*$/);
// Method: 'get:' / 'post:' indentado MAS profundo que el path
const methM = line.match(/^(\s+)(get|post|put|patch|delete|head|options):\s*$/i);
```

Compat hacia atras: si no existe ningun `openapi.yaml`, la verificacion
extra se salta (v12.35 behavior).

Resultado:
```
OK. 6 endpoints declarados, 7 entradas documentadas en api-contract.md,
6 en openapi.yaml. Cero huerfanos.
```

### 3. `check-test-documented` con verificacion de `@trace RF-XX`

Ademas de exigir archivo real de test, ahora exige que ese archivo
contenga `@trace RF-XX`, `@covers RF-XX` o `@implements RF-XX` que
matchee el RF asociado:

```javascript
// En el archivo de test:
// @covers RF-02 — bandeja muestra solo expedientes del rol activo
test("bandeja respeta scope del rol", async () => { ... });
```

Si el archivo existe pero no menciona el RF:

```
TESTS SIN @trace: 1 test(s) existen pero no referencian el RF que cubren.
  - specs/001-bandeja-trabajo-expedientes :: ExpedienteQueryTest
      esperado: RF-02
      archivos: tests/ExpedienteQueryTest.ts

Fix: agregar comentario `@trace RF-02` o `@covers RF-02` en el archivo de test.
```

Cierra el ciclo: matriz -> nombre del test -> archivo -> @trace -> RF.

### 4. `check-ai-artifacts` extendido con `spec-funcional.md`

Cada `specs/<feature>/spec-funcional.md` ahora debe tener:

- `## Requerimientos`
- `## Criterios de aceptacion`
- `## Reglas de negocio`

Sin estas, falla CI. La justificacion: el agente IA no puede responder
"que RF cubre esta feature" sin alucinar si la spec funcional no los
declara explicitamente.

### 5. Plantilla `spec-funcional.md` con regla v12.36+

Nueva seccion `## Requerimientos` con ejemplo:

```markdown
## Requerimientos

> Cada `RF-NN` y `RNF-NN` que aparezca en `traceability.md` de esta feature debe
> estar declarado aqui con su redaccion completa.

- **RF-01**: <accion observable que el sistema debe permitir>.
- **RF-02**: <otra accion observable>.
- **RNF-01**: <atributo de calidad medible — performance, seguridad, disponibilidad>.
```

Tabla `## Criterios de aceptacion` con columna "Estado". `## Reglas de
negocio` con guia de redaccion declarativa. Patron paralelo al
spec-tecnica.md de v12.34 y api-contract.md de v12.35.

### 6. Workflows con `concurrency` mas agresivo

```yaml
# v12.36: concurrency mas agresivo — cancela runs anteriores del mismo PR
# (o del mismo branch en pushes a main) cuando llega un commit nuevo.
concurrency:
  group: template-checks-${{ github.event.pull_request.number || github.ref }}
  cancel-in-progress: true
```

Aplicado a `template-checks.yml` y `project-checks.yml`. Cada push nuevo
a un PR cancela el run anterior del mismo PR. Ahorra minutos de CI y
acelera feedback en iteraciones rapidas.

## Migracion desde v12.35

Sin pasos manuales. Si tu proyecto real:

- Tiene tablas con tipos no estandar: `check:bd-documented` te dira que
  columna ajustar.
- Tiene endpoints en api-contract pero no en openapi.yaml: agrega los
  paths al openapi (el validador da el snippet exacto).
- Tiene tests sin `@trace`: agregalos como comentario al inicio del
  archivo de test.
- Tiene specs-funcional sin las 3 secciones: copia la plantilla de
  `plantillas/fase-4-sdd/spec-funcional.md`.

Los workflows YAML se actualizan automaticamente al hacer merge.

## Que NO esta todavia

- **`check-bd-documented` con verificacion de FK targets reales**
  (que `usuario_id UUID FK -> usuario.id` exija que `usuario` exista).
- **`check-api-documented` con shape de request/response** (no solo
  que el path este, sino que el body schema sea compatible).
- **`check-test-documented` con coverage real** (lcov/coberture) ademas
  del @trace.
- **Plantilla `spdd-frontend.md` con regla equivalente** para
  componentes visuales obligatorios.
- **`check-runbook-documented`** para feature en fase 8 (ops).

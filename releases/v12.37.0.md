# v12.37.0 — FK targets + OpenAPI shape + lcov + spdd-frontend + runbooks

`v12.37.0` cierra los 5 pendientes del backlog en un release cohesivo.
Los validadores existentes pasan de "verificar presencia" a "verificar
relaciones referenciales", y se agrega un validador nuevo para el ciclo
de vida en produccion (`check-runbook-documented`).

## Los 5 items

### 1. `check-bd-documented` con FK targets reales

Antes validaba tipos SQL conocidos. Ahora ademas detecta columnas FK y
exige que la tabla referenciada exista documentada.

Patrones reconocidos en celdas `Tipo` o `Notas`:

```
- apunta a `usuario.id`
- FK -> usuario.id
- FK usuario.id
- REFERENCES usuario(id)
```

Si la tabla no esta documentada con `Tabla \`<nombre>\`` en ningun
spec-tecnica.md (y no es builtin: `usuario`, `user`, `auth`, `role`,
`permission`), reporta:

```
BD INCOMPLETAS: ...
  - "expediente" en specs/001/spec-tecnica.md:
      * FKs que apuntan a tablas no documentadas: responsable_externo -> proveedor (tabla no documentada)
```

### 2. `check-api-documented` con shape de request/response

Parser YAML extendido captura el bloque completo de cada `method:`
(parameters, requestBody, responses). Validaciones:

- Todo endpoint debe tener `responses:` con al menos un status code.
- POST/PUT/PATCH adicionalmente requieren `requestBody:` con schema o $ref.

```
API SIN SHAPE: 1 endpoint(s) en openapi.yaml sin request/response definidos.
  - "POST /api/expedientes" (RF: RF-01):
      * 'responses:' presente pero sin status codes (esperado 200/201/204/4xx/etc.)
      * POST requiere 'requestBody:' con schema (este es write method)
```

### 3. `check-test-documented` con coverage lcov

Si existe `coverage/lcov.info` (o variantes), parsea SF/LH/LF/BRH/BRF
y agrega la cobertura por feature scope (suma de archivos bajo
`src/<slug>/`, `backend/<slug>/`, `frontend/<slug>/`).

Si la cobertura agregada esta debajo del minimo (`--min-coverage`, default
70%), reporta:

```
COVERAGE INSUFICIENTE: 1 feature(s) con cobertura de linea < 70% en lcov.info.
  - specs/001-bandeja-trabajo-expedientes (RF RF-02): 58.3% (140/240 lineas, minimo 70%)

Fix: agregar tests o bajar el umbral con `--min-coverage <N>`.
```

Sin lcov.info, el check no rompe (compat hacia atras).

### 4. Plantilla `spdd-frontend.md` con regla v12.37+

Secciones obligatorias (validadas por `check-ai-artifacts`):

- `## Componentes principales` — Cada componente reusable con responsabilidad en una linea.
- `## Estados UI` — Loading/Empty/Error/Success/Unauthorized/Permission denied con trigger + feedback visual.
- `## Permisos visibles` — Cada accion visible mapea a un permiso de la matriz RBAC.

Patron paralelo a `spec-tecnica.md` (v12.34), `api-contract.md` (v12.35)
y `spec-funcional.md` (v12.36). Sin esto, el frontend y el prototipo se
inventan componentes/estados/permisos sin source-of-truth.

### 5. `check-runbook-documented` nuevo validador

Para features en fase >=8 (Operacion), exige que exista
`ops/fase-8-operacion/features/<slug>/runbook.md` (o variantes) con las
4 secciones que un operador necesita en una madrugada:

```
## Procedimiento normal
## Procedimiento de fallo
## SLO / SLI
## Contactos
```

Si phase <8, no aplica (runbook aspiracional es normal antes de
produccion). Detecta features fase 8 por phase explicita en ai_documents
o por existencia real de runbook.md en alguna ubicacion ops/.

## Migracion desde v12.36

Sin pasos manuales. Si tu proyecto real:

- Tiene FKs a tablas no documentadas: `check:bd-documented` te dira
  cual + en que columna.
- Tiene endpoints en openapi.yaml sin shape minimo: `check:api-documented`
  lo reporta con la lista de huecos.
- Genera reportes de coverage en `coverage/lcov.info`: el check se
  activa automatico. Si no quieres exigirlo, deja el archivo fuera o
  pasa `--min-coverage 0`.
- Tiene features en produccion sin runbook: `check:runbook-documented`
  te dira que crear y donde.

Los GH Actions workflows se siguen disparando con paths-filter; los
nuevos scripts npm se ejecutan dentro de `check:project`.

## Que NO esta todavia

- **`check-bd-documented` con verificacion de columna referenciada**
  (hoy solo valida que la tabla exista; podria validar que `usuario.id`
  exista como columna en `usuario`).
- **`check-api-documented` con validacion de schemas completos**
  (hoy verifica que haya schema/$ref; podria parsear y validar tipos).
- **`check-test-documented` con coverage por RF** (hoy agrega por feature;
  podria mapear lineas a RF especifico usando `@trace` granular).
- **`check-runbook-documented` con SLO numerico verificable**
  (hoy verifica que la seccion exista; podria validar que tenga
  `latency p95 <= Xms` parseable).
- **Plantilla `prototype-validation.md` con regla equivalente** para
  cerrar el ciclo prototipo -> validacion humana.

# v12.38.0 — Cierre de revision externa: gates por feature, freshness, paths

`v12.38.0` cierra los 3 puntos que la revision externa identifico sobre
v12.36 (y que v12.37 no abordo porque estaba enfocada en validadores
nuevos). El mas critico afecta lo que el agente IA "ve" sobre el estado
del proyecto: gates por feature dejaron de invisibilizarse.

## El problema critico que se cierra

Antes:

```
$ npm run memory:query -- --preset failed-gates
# o
$ cat AI_CONTEXT.md | grep gate
- _(no hay gates pendientes registrados)_
```

Cuando la matriz global `TRACEABILITY_MATRIX.md` tenia:

```
| 001-bandeja-trabajo-expedientes | gate-prototype-ready | Listo para validacion | ... |
| 002-cambio-estado-expediente    | gate-prototype-ready | Pendiente             | ... |
| 003-historial-auditoria-expediente | gate-prototype-ready | Pendiente          | ... |
```

`syncGateRuns` deduplicaba las 3 filas por clave `gate::scope` y como el
scope era el archivo (`TRACEABILITY_MATRIX.md` vacio/root), solo se
guardaba una entrada — el primer `gate-prototype-ready` que encontraba.

Resultado: el agente IA leia AI_CONTEXT.md y veia "no hay gates
pendientes" cuando la realidad era que 002 y 003 si tenian uno cada uno.

Ahora:

```
$ npm run memory:sync && grep -A4 'gates-pendientes' AI_CONTEXT.md
- `gate-prototype-ready` en `specs/002-cambio-estado-expediente` — Pendiente (ultimo: Natan, 2026-05-16)
- `gate-prototype-ready` en `specs/003-historial-auditoria-expediente` — Pendiente (ultimo: Natan, 2026-05-16)
```

## Los 3 cambios

### 1. (CRITICO) `syncGateRuns` usa columna `Feature`

```javascript
// v12.38: si la tabla tiene columna 'feature', cada fila usa su propio
// scope (`specs/<feature>`) en vez del scope del archivo. Esto evita que
// 3 filas de TRACEABILITY_MATRIX.md con el mismo gate pero distinta
// feature se deduplicaran a una sola entrada con scope global.
const featureIdx = table.headers.findIndex((h) => h === "feature" || h === "spec");
...
let rowScope = scope;
if (featureIdx >= 0) {
  const featureCell = cleanCell(row[featureIdx]);
  if (featureCell && /^[0-9]/.test(featureCell)) {
    const slug = featureCell.replace(/^specs\//, "");
    rowScope = `specs/${slug}`;
  }
}
const key = `${gate.toLowerCase()}::${rowScope}`;
```

`ai_gate_runs` pasa de 3 filas (con scope colapsado) a 5 filas (con scope
correcto): 3 para 001 (diferentes gates) + 1 para 002 + 1 para 003.

### 2. Path Windows en docs/transversal/90.31

```diff
- Y carga `C:\template\project-template\catalog\all.yaml` desde `app-config.yaml` ...
+ Y carga `catalog/all.yaml` desde `app-config.yaml` ... (en Windows la ruta
+ absoluta tipica es C:\template\project-template\catalog\all.yaml sin
+ backticks porque es texto, no path verificable del repo).
```

`check-markdown-paths` interpretaba el path absoluto Windows entre
backticks como ruta del repo y fallaba porque no existe relativamente.
Ahora usa `catalog/all.yaml` (ruta real) en backticks + la ruta Windows
como texto plano para que siga siendo informativa pero no se valide.

### 3. `memory:bootstrap` y `memory:sync` no dejan STALE

Antes:
```
"memory:bootstrap": "... && embed-docs && regenerate-context"
```

`regenerate-context` modifica `AI_CONTEXT.md` al final. El proximo
`status` reportaba `STALE` porque el markdown era mas reciente que el
ultimo indexado.

Ahora:
```
"memory:bootstrap": "... && embed-docs && regenerate-context && index-docs"
"memory:sync":      "sync-memory && regenerate-context && index-docs"
```

El segundo `index-docs` es barato (re-indexa solo los archivos
modificados; en este caso `AI_CONTEXT.md`). Agrega ~50-200ms y garantiza
`Freshness: fresca` al final.

## Migracion desde v12.37

Sin pasos manuales. La proxima `npm run memory:sync` re-pobla la BD con
la nueva semantica de gates por feature. Los proyectos generados con
`create-project` desde v12.38 reciben los scripts `memory:*` corregidos.

## Por que no se cerro en v12.37

v12.37 estaba enfocado en validadores **nuevos** (FK targets, OpenAPI
shape, lcov, spdd-frontend, runbooks). La revision externa apuntaba a 3
problemas **existentes** que estaban afectando la confiabilidad de la
memoria. Son trabajos paralelos: ambos eran necesarios.

## Que NO esta todavia

- **`syncGateRuns` con detall por origen** (saber si un gate viene de la
  matriz global o de la per-feature; hoy solo guarda el archivo fuente).
- **Validador `check-gates-mentioned`**: verificar que cada feature en
  `specs/` mencione al menos un gate en su `traceability.md`.
- **AI_CONTEXT.md con `## Decisiones recientes`** auto-regenerable.
- **`memory:bootstrap --quick`** que salte el segundo index-docs cuando
  el agente solo quiere refrescar BD sin contexto.

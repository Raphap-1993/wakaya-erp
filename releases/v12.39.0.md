# v12.39.0 — Freshness sin falso STALE + embeddings completos

`v12.39.0` cierra los 2 hallazgos finales de la revision externa de
v12.38. Ambos son detalles sutiles que solo se veian en `status`/logs
pero erosionaban la confianza en la memoria viva.

## Fix 1: falsos `STALE` por precision de timestamp

### Diagnostico

SQLite guarda `updated_at` con `CURRENT_TIMESTAMP` en precision de **segundos**:

```
2026-05-17 06:25:04
```

El filesystem reporta `mtimeMs` en **milisegundos**:

```
2026-05-17T01:25:04.094Z
```

Cuando `regenerate-context` modifica `AI_CONTEXT.md` justo antes de
`index-docs`, los ~500ms de diferencia generaban `STALE` perpetuo
aunque la BD si tenia los datos.

### Fix

`memoryFreshness` ahora aplica tolerancia explicita de **2000ms**:

```javascript
const FRESHNESS_TOLERANCE_MS = 2000;

function memoryFreshness(root, db) {
  // ...
  const delta = newestSource - lastIndexed;
  const stale = delta > FRESHNESS_TOLERANCE_MS;
  return {
    stale,
    lastIndexed: ...,
    newestSource: ...,
    toleranceMs: FRESHNESS_TOLERANCE_MS,
    deltaMs: delta,  // diagnostico
  };
}
```

Si `newestSource - lastIndexed <= 2000ms`, no se marca como stale.

### Validacion

```
$ npm run memory:bootstrap
$ node scripts/ai-framework-agent.mjs status
Freshness:       fresca (ultimo indexado 2026-05-17T06:25:04.000Z; markdown mas reciente 2026-05-17T01:25:04.094Z)
```

Antes: `STALE` por 94ms. Ahora: `fresca`.

## Fix 2: embeddings incompletos por orden incorrecto

### Diagnostico

La cadena anterior era:

```
init-memory → index-docs → sync-memory → embed-docs → regenerate-context → index-docs
                                              ↑
                                  embed-docs aqui ya corrio
```

Problema: `regenerate-context` modifica `AI_CONTEXT.md` y `SESSION_LOG.md`,
lo que crea chunks nuevos cuando el segundo `index-docs` re-indexa. Pero
`embed-docs` ya habia terminado ANTES, asi que esos chunks nuevos
quedaban sin embedding.

Resultado tipico:
```
Chunks:      2560
Embeddings:  2550   ← 10 chunks huerfanos
```

### Fix

Reordenar para que `embed-docs` sea el **ultimo** paso:

```
init-memory → index-docs → sync-memory → regenerate-context → index-docs → embed-docs
                                                                                ↑
                                                                  ahora cubre TODO
```

Aplicado tambien a `memory:sync`:

```
sync-memory → regenerate-context → index-docs → embed-docs
```

### Validacion

```
$ rm ai/memory/framework-agent.db
$ npm run memory:bootstrap
... 6/6 fases sincronizadas
... regenerate-context: AI_CONTEXT.md actualizado
... index-docs: 14 archivos re-indexados
... embed-docs: 2560/2560 chunks

$ node scripts/ai-framework-agent.mjs status
Chunks:      2560
Embeddings:  2560   ← antes 2550
```

## Aplicado tambien en el generador

El `package.json` que emite `create-project` para proyectos reales tiene
los mismos scripts corregidos. Los proyectos generados desde v12.39
nacen sin estos 2 bugs.

## Comparativa

| Metrica                    | v12.38              | v12.39              |
|---------------------------|---------------------|---------------------|
| `status` post-bootstrap   | `STALE` (falso)     | `fresca` (real)     |
| Tolerancia configurada    | 0ms (estricto)      | 2000ms (con buffer) |
| Embeddings post-bootstrap | 2550/2560 (parcial) | 2560/2560 (completo) |
| Orden de pasos             | embed antes de regen | embed al final     |
| `check:all`                | verde               | verde               |

## Migracion desde v12.38

Sin pasos manuales. Las DBs existentes obtienen los embeddings completos
en la proxima `npm run memory:sync`. El cambio de orden no rompe nada:
los pasos son idempotentes.

Si tu pipeline CI usa el script viejo, los proyectos generados desde
v12.39 ya traen el nuevo.

## Por que no se cerro antes

v12.38 cerro 3 problemas estructurales (gates por feature, freshness path,
bootstrap chain). Los 2 de esta version son post-fix que solo se vieron
al observar el output con cuidado:

- El primer `STALE` falso de v12.38 daba la impresion de que algo seguia
  mal, pero realmente eran ms de diferencia.
- Los embeddings parciales solo se notan si comparas `Chunks` vs
  `Embeddings` en `status`.

## Que NO esta todavia

- **`memoryFreshness` con tolerancia configurable** via env
  `MEMORY_FRESHNESS_TOLERANCE_MS` (hoy hardcoded 2000ms).
- **`status --json`** para integraciones que necesitan parsearlo.
- **`memory:sync --quick`** que salte el embed-docs final cuando solo
  cambia metadata (gates, decisions) y no contenido de chunks.
- **Warning de "drift de embeddings"**: si una BD existente tiene
  `chunks > embeddings`, sugerir `npm run memory:embed`.

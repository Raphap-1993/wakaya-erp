# v12.40.0 — Optimizaciones de operacion (config, parseable, quick, drift)

`v12.40.0` cierra las 4 optimizaciones que dejo pendiente v12.39.
Todas son small, todas tocan la operacion diaria de la memoria viva.
Plus 1 fix de UX descubierto al testear.

## Las 4 optimizaciones

### 1. `MEMORY_FRESHNESS_TOLERANCE_MS` env var

```bash
# Default (2 segundos)
node scripts/ai-framework-agent.mjs status

# Modo estricto (cualquier ms cuenta como STALE)
MEMORY_FRESHNESS_TOLERANCE_MS=0 node scripts/ai-framework-agent.mjs status

# Modo CI con clock skew (1 minuto)
MEMORY_FRESHNESS_TOLERANCE_MS=60000 node scripts/ai-framework-agent.mjs status
```

El valor se evalua una vez al cargar el modulo. `memoryFreshness` devuelve
siempre `toleranceMs` en su resultado para diagnostico.

### 2. `status --json` mode

```bash
$ node scripts/ai-framework-agent.mjs status --json
{
  "dbPath": "...",
  "fts5": true,
  "counts": {
    "documents": 267,
    "chunks": 2561,
    "traceLinks": 33,
    "gateRuns": 5,
    "evidence": 38,
    "decisions": 18,
    "openQuestions": 5,
    "sessionEvents": 25,
    "embeddings": 2561
  },
  "freshness": {
    "stale": false,
    "lastIndexed": "2026-05-17T06:33:20.000Z",
    "newestSource": "2026-05-17T01:33:20.681Z",
    "deltaMs": -17999924,
    "toleranceMs": 2000
  },
  "embeddings": {
    "total": 2561,
    "chunks": 2561,
    "drift": 0
  }
}
```

Listo para parsear desde:
- CI dashboards (panel UI ya consume esto via `/api/snapshot`).
- Otros agentes IA (Codex/Cursor/Gemini pueden hacer `npm run memory:status -- --json` y leer el JSON).
- Pipelines (jq, scripts bash).

### 3. `npm run memory:sync:quick`

Cadena:
```
sync-memory && regenerate-context && index-docs
```

Salta el `embed-docs` final. Util cuando:
- Solo cambia metadata (gates, decisiones, sesiones nuevas).
- No cambia contenido de chunks (no hay markdown nuevo).
- El agente quiere refrescar la BD rapido y dejar `embed-docs` para despues.

Ahorra 2-10s por sync (dependiendo del numero de chunks). El comando
`memory:sync` (normal) sigue corriendo `embed-docs` al final.

Si el agente abusa de `:quick` y nunca corre `embed-docs`, el warning
de drift (item 4) lo detecta.

### 4. Warning de drift de embeddings

Cuando `chunks > embeddings` (hay chunks sin vectorizar), `status`
reporta:

```
$ node scripts/ai-framework-agent.mjs status
Embeddings:      2551 (DRIFT: 10 chunks sin embedding)
Freshness:       fresca (ultimo indexado ...)
Embeddings drift: 10 chunks sin embedding.
  Ejecuta: npm run memory:embed   (o corre 'memory:sync' que ya lo hace al final)
```

En modo `--json`:
```json
"embeddings": { "total": 2551, "chunks": 2561, "drift": 10 }
```

El comando no rompe (exit 0); es una **sugerencia accionable**. Captura
el escenario donde un agente IA corre `memory:sync:quick` repetidamente
sin `embed-docs` y los chunks nuevos quedan sin vectorizar.

## Bonus: fix de UX en delta

En Windows con timezone local distinto a UTC, `fs.statSync().mtimeMs` y
SQLite `CURRENT_TIMESTAMP` pueden diferir en horas (interpretacion local
vs UTC). Antes:

```
Freshness: fresca (...) · delta -17999924ms (tolerancia 2000ms)
                                ^^^^^^^^^^^^ inutil
```

Ahora solo se muestra cuando es `> 1000ms` (markdown realmente mas nuevo
que index) y se formatea en segundos:

```
Freshness: fresca (...) · delta +5s (tolerancia 2s)
```

Si el delta es negativo (BD adelantada por timezone o re-index
posterior), no se muestra — no aporta informacion.

## Casos de uso reales

### Pipeline CI con clock skew

```yaml
- run: MEMORY_FRESHNESS_TOLERANCE_MS=60000 npm run memory:sync
- run: node scripts/ai-framework-agent.mjs status --json | jq '.freshness.stale' | grep false
```

### Dashboard externo consumiendo el estado

```bash
# Cada 5 minutos
curl -s http://internal-dashboard/api/state \
  -d "$(node scripts/ai-framework-agent.mjs status --json)"
```

### Sync rapido durante edicion intensiva

```bash
# Editor en watch mode dispara cada save
watchexec -e md "npm run memory:sync:quick"
# Y al cerrar la sesion:
npm run memory:embed
```

### Detectar drift en pre-commit

```bash
# Hook pre-commit
DRIFT=$(node scripts/ai-framework-agent.mjs status --json | jq -r '.embeddings.drift')
if [ "$DRIFT" -gt 0 ]; then
  echo "Embeddings drift: $DRIFT. Run npm run memory:embed first."
  exit 1
fi
```

## Validacion (5 tests)

```
TEST 1  status texto sin delta confuso             -> Freshness: fresca clean
TEST 2  DRIFT auto-detectado tras borrar 3 embeds  -> "Embeddings: 2551 (DRIFT: 10 chunks)"
TEST 3  status --json valido                       -> JSON parseable con embeddings.drift
TEST 4  MEMORY_FRESHNESS_TOLERANCE_MS=0 override   -> toleranceMs: 0 en json
TEST 5  memory:sync:quick salta embed-docs         -> sin linea "Embeddings locales generados"
```

## Migracion desde v12.39

Sin pasos manuales. Los nuevos scripts npm aparecen en `package.json`
del template y de los proyectos generados. La env var es opcional
(default 2000ms). El warning de drift solo aparece si hay drift real.

## Que NO esta todavia

- **`status` con threshold de drift para fail** (hoy siempre exit 0
  para no romper pipelines; podria aceptar `--fail-on-drift`).
- **`memory:bootstrap:quick`** equivalente para reset rapido.
- **`status --watch`** que refresque cada N segundos en terminal.
- **Auto-fix de drift**: al detectar drift, opcionalmente correr
  `embed-docs` automatico con flag `--fix`.

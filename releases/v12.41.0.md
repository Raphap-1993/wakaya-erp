# v12.41.0 — `display_status` semantico por `target_type`

`v12.41.0` cierra la observacion menor de la revision externa de v12.40:
el vocabulario `link_status='implemented'` era ambiguo cuando se aplicaba
a artefactos documentales (`spdd`, `api`, `bd`, `prototipo`). v12.41
introduce una distincion explicita sin romper compat.

## El problema

Antes:

```sql
SELECT target_type, link_status, COUNT(*)
FROM ai_trace_links GROUP BY target_type, link_status;
```

```
target_type   link_status     count
spdd          implemented     5     ← "implementado" suena raro para un .md
api           implemented     6     ← idem para un endpoint
bd            implemented     6     ← idem para una tabla
prototipo     implemented     5     ← idem para un .html
```

Un agente IA podia leer "RF-02 tiene prototipo implementado" y confundirse:
¿implementado en codigo? ¿implementado como diseno? El vocabulario heredado
de codigo (planned/implemented/validated) no encajaba.

## El fix

Nueva columna `display_status` con vocabulario apropiado al target:

| target_type | link_status | display_status |
|---|---|---|
| codigo / test | planned | planned |
| codigo / test | implemented | implemented |
| codigo / test | validated | validated |
| **spdd / api / bd / prototipo / ...** | planned | **pending** |
| **spdd / api / bd / prototipo / ...** | implemented | **documented** |
| **spdd / api / bd / prototipo / ...** | validated | **approved** |
| hu / rf / rnf / estado (cross-refs) | * | documented |
| * | inferred | inferred |
| * | drift | drift |

`link_status` se preserva intacto para compatibilidad. `display_status` es
**aditivo**.

## Despues del fix

```
target_type   link_status     display_status   count
HU            implemented     documented       6
api           implemented     documented       6
bd            implemented     documented       6
estado        implemented     documented       5
prototipo     implemented     documented       5
spdd          implemented     documented       5
```

El agente IA ahora puede decir: "RF-02 tiene **spdd documented** + **prototipo
documented** + **codigo planned**" — vocabulario apropiado por target.

## 3 presets nuevos

Enfocados en artefactos no-code (excluyen `codigo`/`test`/`estado` para no
duplicar con `rf-implemented`/`rf-validated`):

- **`artifacts-pending`**: declarado en matriz pero sin doc real (target no-code con `display_status='pending'`).
- **`artifacts-documented`**: existe en repo (target no-code con `display_status='documented'`).
- **`artifacts-approved`**: Estado de fila aprobado (target no-code con `display_status='approved'`).

```bash
$ npm run memory:query -- --preset artifacts-pending
0 resultados                                ← template canonico esta limpio

$ npm run memory:query -- --preset artifacts-documented
28 resultados · artifact:28
  [artifact] RF-02 -[HU]-> HU-02           display_status=documented · link_status=implemented
  [artifact] RF-02 -[spdd]-> spdd-frontend.md
  [artifact] RF-02 -[prototipo]-> prototype-html5/index.html
  [artifact] RF-02 -[api]-> GET /api/expedientes
  [artifact] RF-02 -[bd]-> expediente
  ...

$ npm run memory:query -- --preset artifacts-approved
0 resultados                                ← ningun Estado dice "Aprobado" formal
```

## `status --json` enriquecido

```json
{
  ...
  "traceLinks": {
    "total": 33,
    "byDisplayStatus": {
      "documented": 33
    }
  }
}
```

Dashboards externos pueden ver de un vistazo cuantos artefactos estan
pending vs documented vs approved sin tener que cruzar con `target_type`.

## Por que importa para agentes IA

El reviewer lo dijo asi:
> "Para evitar ambiguedad futura, yo consideraria una mejora semantica:
> Para target_type = codigo/test: planned | implemented | validated
> Para target_type = spdd/api/bd/prototipo: documented | approved | pending"

Esto resuelve exactamente eso. Un agente IA que lee la BD ya no encuentra
"implementado" donde no encaja conceptualmente. Las preguntas que hacia
con el vocabulario viejo siguen funcionando (link_status compat 100%),
y las nuevas tienen vocabulario preciso.

## Migracion desde v12.40

Sin pasos manuales:
- `ensureColumn("ai_trace_links", "display_status", "TEXT")` agrega la columna
  a DBs existentes (idempotente).
- La proxima `npm run memory:sync` la rellena automaticamente para todos los
  links existentes.

Los presets viejos (`rf-implemented`, `rf-without-code`, `links-drift`,
`rf-planned`) siguen funcionando exactamente igual — usan `link_status`, no
`display_status`. Solo los 3 presets nuevos usan el vocabulario semantico.

## Que NO esta todavia

- **Validar coherencia entre `link_status` y `display_status`** en `check:project`
  (hoy se calcula al sync; podria validarse que no se desincroniza si alguien
  modifica directamente la BD).
- **Vocabulario configurable por proyecto** (algunos equipos prefieren
  `draft` / `final` / `signed-off` en vez de `pending` / `documented` /
  `approved`).
- **Panel UI con filtro por `display_status`** en la tab de Trazabilidad.
- **`memory-query` con multiples filtros combinados** (ej. `--target-type
  prototipo --display-status approved`).

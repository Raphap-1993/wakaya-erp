> "Algunos `evidence_ref` que vienen de `TRACEABILITY_MATRIX.md` todavia

## El problema

Antes en la BD habia 33 trace links pero con `evidence_ref` inconsistente:

```
| evidence_ref                                                        | source            |
|---------------------------------------------------------------------|-------------------|
| docs/fase-3-arquitectura/03.08-auth-authz.md                        | RNF-03            |
| specs/001-bandeja-trabajo-expedientes/prototype-validation.md       | RF-02             |
| specs/001-bandeja-trabajo-expedientes/api-contract.md               | RNF-01            |
| spec-funcional.md                                                   | RF-03 (global)    |
| spec-funcional.md                                                   | RF-04 (global)    |
| api-contract.md                                                     | RNF-01 (global)   |
```

Los 3 ultimos venian de `TRACEABILITY_MATRIX.md` global y `syncTraceLinks`
no sabia a que feature qualificarlos.

Para un agente IA, `spec-funcional.md` solo es ambiguo: ¿cual de los 4?
Tenia que adivinar mirando el `source_ref` (RF-03 -> probable 002).

## La solucion

### Nuevo helper `qualifyEvidenceRef(evidence, scopeFromFile, featureFromColumn, root)`

```javascript
function qualifyEvidenceRef(evidence, scopeFromFile, featureFromColumn, root) {
  const raw = String(evidence || "").trim();
  if (!raw) return raw;
  if (raw.includes("/")) return raw; // ya qualificado

  const candidates = [];
  if (scopeFromFile) candidates.push(`${scopeFromFile}/${raw}`);
  if (featureFromColumn) {
    const slug = String(featureFromColumn).replace(/^specs\//, "");
    candidates.push(`specs/${slug}/${raw}`);
  }

  // Validacion best-effort: el primero que exista en disco gana.
  if (root) {
    for (const cand of candidates) {
      const abs = path.resolve(root, cand);
      try { if (fs.existsSync(abs)) return cand; } catch {}
    }
  }

  return candidates[0] || raw;
}
```

Tres fuentes de qualificacion en orden de preferencia:
1. **Scope del archivo fuente** (per-feature spec).
2. **Columna `feature` de la matriz global** (caso TRACEABILITY_MATRIX.md).
3. **Verificacion best-effort en filesystem** para evitar paths inventados.

### Integracion en `syncTraceLinks`

Misma idea que v12.38 hizo para gates: leer la columna `feature` de la

## Resultado validado

Ahora todos los 33 trace links del template canonico tienen
`evidence_ref` con path completo y verificable:

```
docs/fase-3-arquitectura/03.08-auth-authz.md
specs/001-bandeja-trabajo-expedientes/api-contract.md

specs/003-historial-auditoria-expediente/spec-funcional.md
```

Cero referencias sueltas. Cero paths que el agente tenga que adivinar.

## Validacion best-effort previene paths inventados

Si la matriz declara `evidencia: foo.md` y NO existe
`specs/<feature>/foo.md` en disco, el helper devuelve el path original
(`foo.md`) en vez del fantasma. Asi el agente IA no recibe un path
qualificado que en realidad no apunta a ningun archivo.

Ejemplo:
- Matriz: `| 002-... | RF-03 | ... | spec-funcional.md |`
- Disco: `specs/002-cambio-estado-expediente/spec-funcional.md` EXISTE
- Resultado: `specs/002-cambio-estado-expediente/spec-funcional.md` ✓

Vs:
- Matriz: `| 002-... | RF-03 | ... | not-exists.md |`
- Disco: NO existe
- Resultado: `not-exists.md` (original, no fantasma)

## Impacto en agentes IA

Cualquier comando que devuelva `evidence_ref` ahora muestra rutas
abribles directamente:

- `npm run memory:query -- --preset docs-for --arg RF-03`
- `npm run memory:next-task`
- Panel UI tab Trazabilidad
- Panel UI sub-tab Historial (filtros por `evidence_ref`)
- `status --json` (dashboards externos)

El agente IA puede hacer `cat <evidence_ref>` sin tener que primero
buscar en que feature vive el documento.

## Backwards compat

- `link_status`, `display_status`, `source_file`, `validated_at`,
  `origin`: sin cambios.
- Presets viejos que filtran por `source_file LIKE specs/<feature>/%`:
  siguen funcionando igual.
- Solo cambia el **contenido** de `evidence_ref` cuando se puede mejorar.

## Migracion desde v12.41

Sin pasos manuales. La proxima `npm run memory:sync` re-pobla
`evidence_ref` con paths qualificados.

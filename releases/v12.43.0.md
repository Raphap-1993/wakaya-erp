# v12.43.0 — Cierre de 4 pendientes: evidence, by-evidence, coverage-by-feature, auto-deteccion

`v12.43.0` cierra los 4 pendientes futuros que dejo v12.42. Cada uno
cubre un hueco distinto del ecosistema de trazabilidad.

## 1. Validador `check-evidence-exists.mjs`

Paralelo a `check-trace-drift` pero enfocado en `evidence_ref`. Para
cada evidencia distinta en `ai_trace_links`, verifica que el archivo
exista en disco.

```bash
$ npm run check:evidence-exists
OK. 33 evidence_ref verificadas; todas apuntan a archivos existentes.

# Si falta alguna:
$ npm run check:evidence-exists
EVIDENCIAS INEXISTENTES: 2 evidence_ref apuntan a archivos que no existen.
  - "specs/002/foo.md" (RF-03 -> spdd; declarado en TRACEABILITY_MATRIX.md)
  - "specs/003/runbook.md" (RF-04 -> doc; declarado en specs/003/traceability.md)

Fix: o crear el archivo, o corregir la matriz para usar el path real, o
dejar la celda Evidencia con '-' si aun no esta lista.
```

Ignora silenciosamente:
- URLs (`http://...`, `https://...`)
- Anchors (`#section`)
- Paths absolutos Windows (`C:\...`) o Unix (`/...`)
- Placeholders (`-`, `tbd`, `pendiente`)

Soporta `--strict` (o env `CHECK_STRICT=1`) que ademas marca rutas
cortas sin slash que no se pudieron qualificar.

Integrado en `check:project` y en `EXEC_ACTIONS` del panel UI.

## 2. Preset `by-evidence`

Encuentra todos los RF/RNF/HU que comparten una evidencia (path exacto
o sub-string via LIKE).

```bash
$ npm run memory:query -- --preset by-evidence --arg "spec-funcional.md"
5 resultados · preset: by-evidence
  [trace] RF-03 -[bd]-> transicion_estado            display_status=documented
  [trace] RF-04 -[bd]-> transicion_estado            display_status=documented
  [trace] RF-04 -[api]-> GET /api/expedientes/{id}/historial
  [trace] RF-04 -[prototipo]-> prototype-html5/index.html
  [trace] RF-04 -[spdd]-> spdd-frontend.md
```

Util para responder:
- "¿Que requerimientos cubre `spec-funcional.md`?"
- "¿Que toca este archivo?" antes de modificarlo.
- "¿Si elimino esta evidencia, que RF quedan huerfanos?"

Excerpt incluye `display_status` para que la respuesta sea
inmediatamente accionable.

## 4. Auto-deteccion de columnas-path nuevas

Si un equipo agrega a `traceability.md` una columna nueva (ej. `Runbook`,
`Threat model`, `Diagrama c4`) que no esta en `TRACE_COLUMN_MAP`, antes
se ignoraba. Ahora `syncTraceLinks` detecta:

- Header no mapeado AND
- Valor que parece path: termina en `.md`/`.html`/`.yaml`/`.ts`/`.json`/etc.
  O contiene `/`.

Entonces lo registra como `target_type='doc'` con `relation` derivada
del header normalizado:

```
| ... | Runbook                                  |
| ... | ops/features/001/runbook.md              |
                ↓
ai_trace_links: source='RF-02' target_type='doc' relation='runbook'
                target_ref='ops/features/001/runbook.md'
                display_status='documented' (existe en disco)
```

El nuevo tipo `doc` se trata como `NON_CODE_TARGETS` (display_status:
pending/documented/approved) y `targetRefExists` lo busca en specs/,
docs/, ops/ y qa/.

**Beneficio**: equipos pueden enriquecer la matriz con columnas
relevantes a su dominio (compliance, observabilidad, threat-model,
runbooks) sin tocar el codigo del agente. El validador
`check-evidence-exists` los verifica automaticamente.

## Que NO esta todavia

- **`check-evidence-exists` con sugerencia de fix** (ej. "¿quisiste
  decir specs/002/spec-funcional.md?").
- **Panel UI con drill-down** desde "Por feature" al link individual
  con su historia git.
- **`by-evidence` con feature opcional** para filtrar (ej. `--arg
  "spec-funcional.md|002"`).
- **Validador `check-orphan-evidence`**: detectar archivos en specs/
  que NUNCA aparecen como evidencia (posiblemente abandonados).

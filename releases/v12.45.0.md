# v12.45.0 — release cohesivo de 26 mejoras en 6 ejes

Esta version cierra un backlog acumulado de 26 observaciones en un solo release coherente. No es una sucesion de parches aislados: los 6 grupos cierran 6 huecos distintos del ecosistema de trazabilidad, panel del agente y validacion estricta.

## TL;DR
- 3 validadores nuevos (`check-gates-mentioned`, `check-status-coherence`, `check-orphan-evidence`).
- 4 validadores existentes hechos estrictos (BD valida columna FK, API parsea JSON Schema, Test usa coverage por RF, Runbook exige SLO numerico).
- 5 mejoras en el panel embebido (alertas combinadas, snooze por kind, webhook con retry, dashboard exportable a MD, comparacion lado a lado en Tendencias).
- 3 modos nuevos del comando `status` (`--fail-on-drift`, `--watch`, `--fix`).
- 4 mejoras de query y trace (vocabulario configurable, filtro UI por `display_status`, query libre combinatorio, decisiones recientes en `AI_CONTEXT`).
- 2 mejoras de evidencias (`did-you-mean` en errores, drill-down a git history).
- Plantilla `prototype-validation.md` con regla canonica de aprobacion equivalente a los validadores estrictos.

## Por que importa
Hasta v12.44 el ecosistema tenia varios "huecos" pequenos pero acumulativos:

1. **Validadores tolerantes**: `check-bd-documented` aceptaba FKs apuntando a columnas inexistentes; `check-api-documented` no parseaba schemas; `check-runbook-documented` aceptaba `SLO: TBD` literal. La validacion existia pero no mordia.
2. **Memoria pasiva**: las decisiones (ADR / decisiones-ux) vivian en `adr/` pero el agente nunca las veia al cargar `AI_CONTEXT.md`. Si una feature no mencionaba ningun gate, nadie se daba cuenta.
3. **Panel reactivo**: dos alertas del mismo `action_id` (failure + slow) se mostraban como dos cards separadas. Snooze era binario (todo o nada). El webhook no reintentaba.
4. **Status sin gancho a CI**: `status --json` existia pero no devolvia exit code 1 cuando habia drift, asi que ningun pipeline podia engancharlo.
5. **Vocabulario rigido**: `pending/documented/approved` estaba hardcoded; proyectos con vocabulario propio (`draft/final/signed-off`) tenian que monkey-patchear el codigo.
6. **Evidencias dificiles de auditar**: cuando `check-evidence-exists` decia "no existe", no daba pista de que archivo quisiste decir. Y un archivo en `specs/` jamas referenciado pasaba inadvertido.

Este release cierra los seis a la vez.

## Cambios criticos sin breaks

- **Schema**: `ai_gate_runs.origin` y `ai_action_snoozes.kind` se agregan via `ensureColumn` (ALTER TABLE idempotente). DBs existentes funcionan tal cual; las nuevas columnas quedan en NULL hasta el siguiente `memory:sync`.
- **API alerts**: `/api/action-runs/alerts` agrupa por default. Para volver al formato plano pasa `?combine=0`.
- **API snoozes POST**: acepta `kind` opcional. Sin `kind` (default), preserva el comportamiento v12.30: silencia todo.
- **Comando `status`**: sin flags, comportamiento identico a v12.44. Los flags `--fail-on-drift`, `--watch`, `--fix` son opt-in.
- **`check:project`** ahora corre 10 validadores (antes 7); los 3 nuevos pueden romper pipelines existentes si las matrices estan incompletas. Soluciones: `--warn-only` en `check-orphan-evidence`, o agregar los gates / corregir display_status que el validador reporta.

## Comandos sugeridos para validar localmente

```sh
# Bootstrap rapido (sin embed-docs)
npm run memory:bootstrap:quick

# Status estricto para CI
node scripts/ai-framework-agent.mjs status --fail-on-drift --json

# Watch en otra terminal mientras editas markdowns
node scripts/ai-framework-agent.mjs status --watch 3

# Query combinatorio
node scripts/ai-framework-agent.mjs memory-query --target-type api --display-status documented

# Dashboard en Markdown
curl -s "http://127.0.0.1:4319/api/action-runs/dashboard?days=30&format=md" > dashboard.md

# Snooze especifico solo por slow runs
curl -X POST http://127.0.0.1:4319/api/action-runs/snoozes \
  -H 'Content-Type: application/json' -H 'Origin: http://127.0.0.1:4319' \
  -d '{"action_id":"check-trace-drift","kind":"duration-threshold","duration":"7d","reason":"sabemos que esta accion crece con el repo"}'
```

## Que NO cambia
- El contrato del agente (`AGENTS.md`).
- La capa SDD/SPDD.
- Los stacks de referencia.
- El bootstrap por defecto (`memory:bootstrap` sigue incluyendo `embed-docs`).
- El comportamiento de `check:template` (los 4 validadores siguen verdes en el template canonico).
- El comportamiento de `memory-query --preset <k>` (sin filtros nuevos).

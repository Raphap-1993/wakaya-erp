# v12.48.0 — refuerzo del piso de calidad para agentes IA

Release agregado que consolida v12.46 (bloqueantes), v12.47 (scaffold + ejemplos copiables) y v12.48 (calidad de prototipos + trazabilidad inter-feature) en una sola entrega. Origen: auditoria de 3 proyectos reales (opencode, codex, gemini) que instanciaron el template y fallaron sistematicamente en aplicarlo.

## El problema que se cerro

Auditamos 3 instanciaciones reales hechas por agentes IA distintos. **Los 3 fallaron en patrones consistentes**:

| Patron de error | opencode | codex | gemini |
|---|:-:|:-:|:-:|
| Matriz traceability con 7 cols (no 10) | ❌ 6/11 | ❌ 8/8 | ⚠ 1/5 ok |
| Sin seccion `## Gates` per-feature | ❌ | ❌ | ⚠ |
| `AI_CONTEXT.md` sin zona `decisiones-recientes` | ❌ | ❌ | ❌ |
| `package.json` sin scripts v12.45 | ❌ | ❌ | ❌ |
| `api-contract.md` sin schemas OpenAPI | ❌ | ❌ | ❌ |
| `SESSION_LOG` con `$(date)` literal | — | ❌ | ❌ |
| Terminos residuales (`bandeja`, `expediente`) | ⚠ | ⚠ | ❌ |
| Prototipos esqueleticos o Tailwind CDN | ⚠ | ⚠ | ❌ |
| Slugs `NNN-` duplicados | — | — | ❌ |
| **Validadores `check:project` que pasarian** | **1-2 / 10** | **3-4 / 10** | **3 / 7** |

5 de los 9 patrones eran bugs del propio template (generador). 4 eran problemas de guia documental insuficiente. v12.48 cierra los 9.

## TL;DR

- **11 mejoras documentales** aplicadas: 4 bloqueantes (v12.46), 3 de scaffold (v12.47), 4 de calidad (v12.48).
- **2 archivos nuevos**: `scripts/scaffold-feature.mjs` (genera 11 archivos canonicos por feature), `docs/transversal/90.35-trace-annotations-por-stack.md` (`@trace` por 7 stacks).
- **1 carpeta nueva**: `plantillas/_full-feature-example/` con README de referencia copiable.
- **3 bugs del generador `init-project` corregidos**: `$(date)` ahora se expande, `package.json` incluye v12.45 scripts, `AI_CONTEXT` template tiene las 7 zonas auto.
- **4 reglas nuevas** en `check-template-instantiation` strict.
- `check:all` verde end-to-end con 13/13 validadores.

## v12.46 — bloqueantes (cierran los 4 patrones mas frecuentes)

### P1.1 — Pre-flight checklist obligatorio en `AGENTS.md`

Nueva seccion "CRITICO — Pre-flight checklist OBLIGATORIO antes de cerrar (v12.46+)" con 7 comandos en orden que el agente DEBE correr antes de declarar el proyecto listo:

```bash
npm run memory:sync                                                    # 1
npm run check:template                                                 # 2 — 4/4 OK
npm run check:project                                                  # 3 — 10/10 OK
node ci/scripts/check-template-instantiation.mjs --mode instantiated   # 4
node scripts/ai-framework-agent.mjs status --fail-on-drift --json      # 5
# 6: bucle for que verifica 9 archivos canonicos por feature
# 7: slugs NNN- unicos
```

Si alguno falla, el agente NO puede decir que terminó.

### P1.3 — `check-template-instantiation.mjs` strict (4 reglas nuevas)

Cuando se invoca con `--mode instantiated`, ademas de los chequeos anteriores valida:

1. `checkRequiredScriptsInPackageJson` — los 4 scripts v12.45 deben estar en `package.json`.
2. `checkAiContextAutoZones` — las 7 zonas `<!-- auto:start name=... -->` deben existir en `AI_CONTEXT.md`.
3. `checkFeatureCompleteness` — cada feature `NNN-*` debe tener los 9 archivos canonicos + matriz 10 cols + seccion `## Gates`. Detecta slugs `NNN-` duplicados.
4. `checkResidualTerms` — strings prohibidos (`bandeja`, `expediente`, `$(date)`, `<feature>`, `<slug>`, `<org>`) en archivos generados.

**Verificacion E2E**: contra los 3 casos auditados detecta 174 hallazgos (opencode), 21 (codex), 70 (gemini). Cuando se aplica al template canonico mismo: 0 hallazgos.

## v12.47 — scaffold + ejemplos copiables (cierra el "no se que estructura usar")

### P1.2 — `scripts/scaffold-feature.mjs` (script nuevo)

Genera los 9 archivos canonicos por feature con estructura ya alineada al canon v12.45. Cada archivo viene con placeholders `<...>` que el agente solo rellena:

```bash
node scripts/scaffold-feature.mjs \
  --slug 003-control-parental \
  --titulo "Control parental" \
  --rfs RF-03,RF-04 \
  --rnfs RNF-02 \
  --hus HU-03 \
  --entidad perfil_parental \
  --endpoint "GET /api/parental,POST /api/parental"
```

Resultado: 11 archivos. Spec-tecnica con bloque `Tabla \`perfil_parental\`` + PK + Indices listo. Traceability con matriz 10 cols + seccion `## Gates` listo. Api-contract con OpenAPI snippet listo. Etc. Tambien crea `prototype-html5/index.html` con placeholder que enlaza al hub.

Expuesto via `npm run scaffold:feature`.

### P2.2 — `plantillas/_full-feature-example/README.md`

Referencia visual del formato canonico esperado. Lista los 9 archivos canonicos con su validador asociado, header canonico de la matriz (10 cols), variantes erroneas reales (las 2 vistas en codex y opencode), header canonico de Gates, bloque canonico de tabla BD.

### P2.4 — Loop iterativo de features documentado

## v12.48 — calidad (cierra prototipos pobres y trazabilidad inter-feature)

### P3.1 — `plantillas/fase-4-sdd/api-contract.md` con OpenAPI embebido

Cada endpoint ahora lleva un bloque YAML OpenAPI completo (no solo el header). Ejemplos: `GET /api/expedientes` con `parameters`, `responses`, `schema`. `POST /api/expedientes` con `requestBody`, `responses 201/400/401/403`. Bloque `components/schemas` compartido. Tambien documenta la regla v12.45 A2 strict (`type:` obligatorio, `properties:` si object, `items:` si array).

### P3.3 — `docs/transversal/90.35-trace-annotations-por-stack.md` (doc nuevo)

Convencion de `@trace` para 7 stacks con ejemplos copy-paste:

| Stack | Forma canonica |
|---|---|
| TypeScript / JavaScript | `// @trace RF-02` o `/** @trace RF-02 */` |
| Java / Kotlin | `/** @trace RF-02 */` |
| Python | `# @trace RF-02` |
| Go | `// @trace RF-02` |
| Rust | `/// @trace RF-02` |
| C# / .NET | `/// @trace RF-02` o `// @trace RF-02` |
| SQL | `-- @trace RF-02` |

Mas tabla "donde poner @trace" por capa (controller / service / repository / test / component / migracion / openapi) y errores comunes (`RF-2` vs `RF-02`, etc.).

### P3.4 — `sync-memory` detecta inconsistencia inter-feature

Nueva funcion `detectFeatureMatrixInconsistencies` se corre al final de cada `sync-memory`. Si hay **mezcla** (algunas features con header canonico 10 cols + otras sin), emite warning:

```
[warning] inconsistencia inter-feature en matrices de traceability:
  - 5 feature(s) usan el header canonico (10 cols), pero 6 no:
  -   · specs/001-foo/traceability.md (detectadas 7 columnas, esperadas 10)
  -   · specs/002-bar/traceability.md (detectadas 7 columnas, esperadas 10)
  - ...
```

Captura el patron real de opencode (features 001-004 con matriz mala + 005-008 con matriz buena).

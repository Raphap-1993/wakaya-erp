# v12.5.0 - Enforcement de adopcion profesional y validadores IA

[README principal](../README.md) | [Indice docs](../docs/README.md) | [Volver a releases](README.md)

## Resumen
`v12.5.0` cierra la segunda pasada de profesionalizacion: la plantilla ya no solo documenta el estandar, tambien valida que la adopcion real no se rompa por rutas inexistentes, tokens residuales, skills incompletas, prompts sin criterios minimos o workflows con condiciones fragiles.

El objetivo de esta version es convertir la base `v12.4.0` en un punto de arranque mas seguro para un proyecto real: stack instanciable, IA gobernada por evidencia, checks ejecutables y menor riesgo de falsos positivos.

## Destacado
- **Instanciacion semantica**: nuevos campos `apiResourceName`, `apiResourcePlural` y `apiResourcePath` permiten reemplazar nombres visibles, permisos RBAC, rutas API y contratos de prueba sin dejar el caso canonico en codigo ejecutable.
- **Golden path verificado**: `new-service.mjs` valida docs e instanciacion antes del commit inicial; la pasada se probo con `node-next` y `quarkus-angular` en `.tmp/` sin git ni smoke pesado.
- **Validadores nuevos**: `check-markdown-paths`, `check-ai-artifacts` y `check-github-actions` cubren rutas en backticks, anatomia IA obligatoria y uso riesgoso de `secrets.*` en `if`.
- **IA con enforcement uniforme**: skills y prompts base quedan alineados con anti-rationalizations, red flags, verification evidence, no-usar-cuando y verificacion minima.
- **SOLID explicito**: nuevo documento transversal `90.30` para aplicar SOLID y diseno modular en arquitectura, specs, construccion y review.
- **Adopcion real mas clara**: guia inicial, checklist de adopcion, Day-0 plan y mapa IA aclaran que el stack runnable vive en `stacks/*/template/` y que `new-service.mjs` es el camino recomendado para crear servicios.
- **Angular profesional**: el stack Angular queda documentado y validado como Nx workspace con `apps/`, `libs/`, `nx.json` y `apps/web/project.json`.
- **Limpieza operativa**: se elimina `contracts/api/openapi.yaml.tmp`, se agregan rutas operativas faltantes y `.gitignore` cubre temporales, dependencias y artefactos generados.

## Adaptacion por fase
- **Fase 0**: usar el plan Day-0 sin rutas rotas, registrar stakeholders reales y elegir stack antes de hablar de codigo ejecutable.
- **Fases 1-3**: convertir dominio, API, datos, arquitectura, IAM/OIDC y Keycloak en decisiones trazables por ADR y no en nombres heredados del caso canonico.
- **Fases 4-6**: crear la primera feature desde specs, contratos, codigo y pruebas; cerrar con commands IA, gates y evidencia de build/test.
- **Fases 7-8**: conectar preview, secretos, runbooks, DR, SLO, dashboards y on-call a entornos reales antes del release.

# v12.50.0 — cierre del patron codex (clonacion de prototipos) + 3 consideraciones pendientes

Release que cierra la causa raiz especifica de codex (clonacion mecanica de prototipos entre features) detectada en el segundo audit profundo, mas las 3 consideraciones pendientes de v12.49 (scaffold:project, generate:openapi, validacion de hipotesis).

## TL;DR

- **Causa raíz codex resuelta**: nuevo validador `check:prototype-diversity` que detecta cuando un agente clona el mismo template entre features. Verificado E2E contra los 8 prototipos de codex (todos ~395 líneas) → bloqueante.
- **`scaffold:prototype` mejorado**: ahora calcula hue HSL por slug (variabilidad organica) e inyecta contexto del `spec-funcional.md` en el HTML para que el agente sepa qué adaptar.
- **`scaffold:project`** (nuevo): un solo comando orquesta init + bootstrap + 6 validadores.
- **`generate:openapi`** (nuevo): ensambla `contracts/api/openapi.yaml` consolidado desde todos los `api-contract.md`.
- **Hipótesis v12.49 validada**: codex case pasa **10 de 11 validadores v12.50** (vs 1-4 inicial). Mejora 3-10×.

## El audit profundo de codex

Al re-revisar v12.49, comparé los 3 casos reales con metrics:

| Métrica | Codex | OpenCode | Gemini | Golden |
|---|---:|---:|---:|---:|
| Features generadas | 8 | 9 | 3 | 1 (ref) |
| Líneas mín-máx | **392-392** | 567-712 | 640-1308 | 1309 |
| Líneas promedio | 392 | 644 | 863 | 1309 |
| **Variabilidad inter-feature** | **0% (CERO)** | 25% | 104% | — |
| Mock data diferenciado | NO (idéntico) | SI | SI | SI |

> "Todos los prototipos de Codex tienen exactamente 392 líneas. Los hashes MD5 difieren ligeramente pero el tamaño es idéntico. Únicos cambios: 3 líneas de título/descripción, badge, y color HSL +19° aritmético entre features (145°→164°→183°...)."

**Patrón confirmado**: codex usó un loop programático que clonó el primer prototipo a las otras 7 features sin contextualizar al dominio de cada una. Variabilidad real por feature: ~5 líneas de 392 (1.3%).

## Los 4 cambios principales de v12.50

### C4 — `ci/scripts/check-prototype-diversity.mjs` (nuevo)

Validador con 3 heurísticas para detectar clones:

```javascript
// 1. clone-lines: si >= 5 features en mismo bucket de líneas (±5) = blocker
// 2. clone-structure: hash de HTML/CSS/JS limpiando colores/IDs/textos = blocker si coincide
// 3. clone-mock: hash de textos visibles deduplicados = blocker si coincide
```

**Verificación E2E**:

```bash
# Codex (8 features clone):
$ node ci/scripts/check-prototype-diversity.mjs --root /path/to/codex --strict
✗ [clone-lines] 8 features con prototipos de ~395 lineas (±5). Evidencia fuerte de clonacion sistematica.
exit 1

# OpenCode (9 features diversos):
$ node ci/scripts/check-prototype-diversity.mjs --root /path/to/opencode
OK. 9 prototipos verificados. Diversidad correcta — cada feature tiene contenido propio.
exit 0
```

Wire en `check:project` (ahora 11 validadores).

### `scaffold-prototype.mjs` mejorado (v12.50)

Dos features nuevas que cierran la causa raíz:

1. **`deriveHueFromSlug(slug)`**: calcula un hue HSL 0-360 reproducible del slug. Resultado: cada feature tiene paleta distinta automáticamente, sin que el agente lo recuerde.

```javascript
function deriveHueFromSlug(slug) {
  let h = 0;
  for (let i = 0; i < slug.length; i += 1) {
    h = (h * 31 + slug.charCodeAt(i)) >>> 0;
  }
  return h % 360;
}

// 998-test-a -> hue=163, 997-test-b -> hue=125 (auto-distintos)
```

2. **`readFeatureContext(featureDir)`**: lee `spec-funcional.md` y extrae RFs/HUs/RNFs + actores; inyecta como comentario HTML en el prototipo:

```html
<!-- scaffold-prototype: contexto del spec-funcional.md para adaptar mock data.

     Requerimientos a representar visualmente en este prototipo:
       - RF-02: el usuario puede listar expedientes filtrados por...
       - RNF-01: la consulta frecuente responde p95 <= 800ms con...

     Actores/roles a representar:
       | Operador | gestiona expedientes propios |
       | Supervisor | revisa su area + subordinados |

     NO copies el mock data del golden literal — adapta nombres, IDs,
     registros al dominio de esta feature.
-->
```

Esto deja el contexto **dentro del HTML**, no en otro archivo que el agente pueda ignorar.

### C2 — `scripts/scaffold-project.mjs` (nuevo)

Orquesta el bootstrap completo en 6 pasos:

```bash
$ npm run scaffold:project -- --dry-run
[1/6] ⊘ init-project (si aplica) (skip: --skip-init)
[2/6] → memory:bootstrap (BD + indices + embeddings)
[3/6] → check:template (4 validadores)
[4/6] → check:project (10 validadores)
[5/6] → check:instantiation strict
[6/6] → check:prototype-diversity (detecta clones)
```

Cada paso reporta tiempo + status. Si uno falla, se detiene y reporta el step específico. Cierra el patrón real visto: agentes corrían pasos sueltos en orden incorrecto o saltaban validadores.
